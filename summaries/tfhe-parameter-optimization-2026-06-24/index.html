Parameter Optimization and Larger Precision for (T)FHE

PDF (text-layer, figures rasterized from vector/embedded diagrams) — IACR ePrint 2022/704, retrieved via Wayback Machine · full view

At a glance

Authors: Loris Bergerat, Anas Boudi, Quentin Bourgerie, Ilaria Chillotti, Damien Ligier, Jean-Baptiste Orfila, Samuel Tap — all at Zama (Paris). Published in the Journal of Cryptology (2023); this summary is from the IACR ePrint 2022/704 full version.

One sentence: The paper attacks the practical bottleneck of FHE — choosing cryptographic parameters that are simultaneously secure, correct, and fast — by formalizing it as a constrained optimization problem with per-operator noise and cost models, and pairs this with new large-precision techniques (a padding-free WoP-PBS and a hybrid radix/CRT integer representation) so that TFHE can compute on 16- and 32-bit integers efficiently.

Two intertwined contributions:

  • An optimization framework that takes a crypto-free arithmetic graph plus a target security level and failure probability, and emits an FHE graph together with its optimal parameter set — the first such tool for TFHE-like (bootstrapping-heavy, multi-precision) schemes.
  • Larger-precision machinery: a generalized radix/CRT/hybrid multi-ciphertext encoding of large integers, two new homomorphic modular-reduction algorithms for arbitrary modulus, and a new WoP-PBS (programmable bootstrap Without bit of Padding) that evaluates a generic LUT over a multi-ciphertext integer.

Headline findings:

  • The classical TFHE PBS only scales to about 5 bits of message precision before becoming impractical (each extra bit roughly doubles the cost above 4–5 bits), and no feasible parameters exist beyond ~11 bits.
  • For larger precision it is far better to split a message across several small ciphertexts than to inflate parameters for one. The new WoP-PBS scales roughly linearly (not exponentially) in message size for generic LUTs.
  • Concrete benchmarks (TFHE-rs, single Xeon, 128-bit security): a generic 16-bit LUT in ~0.76–0.82 s; 16-bit homomorphic multiply in ~0.4 s (CRT) to ~5.7 s (radix); 32-bit multiply from ~1.8 s (hybrid) up.
  • Security is enforced via the lattice-estimator as a “security oracle”; correctness via Gaussian-tail failure-probability bounds; the whole search is reduced to optimizing a single “atomic pattern” solvable in well under a second.
  • All parameters are produced by the framework and all benchmarks run on TFHE-rs, Zama’s successor to the Concrete library — the techniques directly underpin Zama’s integer/Concrete stack.

The problem: secure AND correct AND fast, all at once

In theory FHE computes anything over ciphertexts; in practice the obstacle is parameter selection. Every TFHE-like computation must fix many knobs — the authors split them into macro-parameters (LWE dimension n, GLWE dimension k, polynomial size N, ciphertext modulus q, noise levels) and micro-parameters (per-operator decomposition base β and level ℓ for a key-switch or bootstrap). These must jointly satisfy three guarantees:

  • Security — the (R)LWE instance must be hard. The product k·N (or n), the secret-key distribution, the noise, and q determine the level; the LWE/lattice-estimator [APS15] judges it. Crucially, security is “easy” in the sense that one can always add encryption noise to reach the target — so the search reduces to picking dimensions and then taking the minimal secure noise.
  • Correctness — noise grows through each operator; if it crosses a threshold, decryption returns the wrong message. One must track noise via per-operator noise formulae and keep it below a bound at every edge of the circuit.
  • Cost — among all correct, secure choices, minimize a cost model (here, single-thread running time approximated by operation counts).

The central question of Section 3: “For a given graph of FHE operators, how to find parameters so the evaluation is the fastest while preserving both correctness and security?” The authors note this was an open problem before their contribution, and that existing FHE compilers either target batched, bootstrap-free schemes (BGV/BFV/CKKS, where the lever is multiplicative depth) or use hard-coded parameters for Boolean TFHE circuits.

The noise and cost model, and the security oracle

Each FHE operator is equipped with three things (Definitions 4–5): a plain operator (the cleartext function it must reproduce), a noise model (a formula giving the output noise variance from the input variances, the cryptographic parameters, and the plaintext values), and a cost model (a surrogate for the metric to minimize). Noise is always reasoned about as a variance of a centered Gaussian; “more noise” means “larger variance”.

The security oracle (Definition 6) is built from the lattice-estimator: given k·N, a security level λ and modulus q, it returns the minimal noise variance σ²min needed for security. This decouples security from the optimization: once dimensions are chosen, the encryption noise is pinned to the oracle’s value, so the optimizer never wastes effort searching noise.

The cost model used throughout counts the algorithmic work of each operator — additions, multiplications, integer-type casts, and the asymptotic FFT cost — as a proxy for single-thread time. For example a polynomial dot-product (the workhorse inside the blind rotation) costs (2+1)αN log N to enter the FFT domain, 2αN float multiplies, (α−1)N float adds, and 2N log N to return. A subtle but important detail: because TFHE's PBS uses an FFT in f64, the authors add an empirical FFT-noise correction to the bootstrap variance — a term n·2ω·ℓ·β²·N²·(k+1) with ω ≈ 22−2.6, capturing mantissa-bit loss (53-bit mantissa vs 64-bit q) plus an experimental fit. Tight, faithful noise formulae are what make the confidence intervals (and hence the chosen parameters) trustworthy.

Failure probability: from Gaussian tails to a noise bound

Correctness is made quantitative through the standard score z*(pfail) = √2·erf−1(1−pfail): a centered Gaussian of std-dev σ lands outside (−z*σ, z*σ) with probability at most pfail. Decoding fails iff some coefficient of the error exceeds half the scaling factor Δ/2, so the authors derive an explicit noise bound (Definitions 8–9): a ciphertext with message modulus p and π padding bits decodes correctly with probability ≥ 1−α as long as σ ≤ tα = Δ/(2·z*(pfail)), where for an N-coefficient ciphertext pfail = 1−(1−α)1/N (a union bound over the N coefficients; for LWE, N=1).

This turns correctness into a clean set membership: the noise feasible set 𝒮 is the set of parameter choices for which every edge of the graph satisfies σi ≤ tα(pi). The optimization problem (Eq. 2) is then simply arg minx Cost(𝒢, x) subject to x ∈ 𝒮(𝒢) — a constrained minimization that classical solvers (branch-and-bound) can attack. The remaining difficulty is that 𝒮 depends on the whole graph, which can be huge.

Shrinking the search: atomic patterns, domination, and Pareto fronts

The key simplification is to chop the graph at the operators whose output noise is independent of their input noise — in TFHE that is the PBS (category i), as opposed to noise-adding operators like the key-switch or dot product (category ii). Between two consecutive PBSs lies an atomic pattern (AP): a sub-graph whose noise can be analyzed in isolation. The canonical AP type 𝒜(CJP21) is a dot product (DP) → key-switch (KS) → PBS, and it is fully characterized by just two numbers: the dot-product 2-norm ν (the ℓ2 norm of the integer weights, which scales the input noise) and the noise bound t.

This admits a domination relation (Theorem 1): an AP with larger ν and tighter t has a smaller feasible set, so it dominates the others — satisfying its constraint satisfies theirs. Therefore one only needs to keep the non-dominated APs, i.e. the Pareto front of the graph, and 𝒮(𝒢) = 𝒮(𝒢pareto). Because cost is (approximately) independent of the particular AP instance, minimizing the cost of the whole graph reduces to minimizing the cost of a single worst-case atomic pattern. Two bracketing graphs — 𝒢worst (max ν, min t) and 𝒢best — give an upper and lower bound on the optimum, and coincide (optimal) when the graph is uniform. A further shortcut: in 𝒜(CJP21) noise increases monotonically until the modulus-switch inside the final PBS, so the bound need only be checked at that one point. This machinery also lets one fairly compare different bootstrapping algorithms: a noisier bootstrap simply shows up as a higher input noise for the next AP.

Why large precision needs many ciphertexts: TFHE PBS vs Tree-PBS

Using the framework to actually price operators yields the paper’s central practical message. The plain TFHE PBS (𝒜(CJP21)) is the best tool for 1–5 bit messages, but its cost is essentially exponential: above 4–5 bits each extra bit of precision roughly doubles the cost (the polynomial size N must double), and beyond ~11 bits no correct parameters exist at all. The alternative for larger integers is to split the message across blocks and use the Tree-PBS of Guimarães–Borin–Aranha (𝒜(GBA21)), whose first layer uses a multi-value PBS [CIM19]. Tree-PBS reaches higher precision (up to 16 bits with 2 blocks, 21 bits with 3 blocks) but its cost explodes with the number of blocks — a generic 21-bit LUT via Tree-PBS costs more than 220× a Boolean PBS. The takeaway: above ~9 bits even Tree-PBS is too expensive, motivating a better generic-LUT algorithm.

The single-ciphertext curve stops near 11 bits; splitting into blocks reaches further but grows fast.

Encoding large integers: carry buffers, radix, CRT, and the hybrid

To compute on large integers, a message is decomposed into several small LWE ciphertexts. The authors first refine the single-ciphertext plaintext layout: the plaintext space is split into a message subspace (an integer mod β), a carry subspace (room to absorb carries from leveled additions/multiplications), and a padding bit for bootstrapping. A metadata called the degree of fullness — the worst-case message value divided by p−1 — tracks how full the carry buffer is; while it stays below 1, additions and scalar multiplications proceed without a bootstrap, and a PBS is only triggered to flush carries and reduce noise. Notably they also handle a PBS with p not a power of two, by carefully sizing the redundant “mega-cases” of the LUT polynomial and adjusting the failure analysis.

Two classical multi-ciphertext schemes are then generalized:

  • Radix: a number mod Ω = ∏βi stored as digits in mixed bases — like ordinary base-10 notation. Cheap arithmetic by “schoolbook” carry propagation, but historically limited to Ω that factor into small bases (no arbitrary, e.g. prime, modulus).
  • CRT: residues x mod ωi for co-prime ωi, each in its own ciphertext — independent, parallel arithmetic, but limited by the scarcity of small (<8-bit) co-primes. A native CRT variant drops the padding bit and carry buffer, making additions/scalar-mults essentially free (LWE additions) at the price of needing a WoP-PBS for any LUT.

The new hybrid representation (Figure 6) puts a CRT layer on top and a radix layer beneath each residue, recovering both the CRT’s parallelism and the radix’s ability to represent any modulus. It strictly generalizes pure-CRT (all κi=1) and pure-radix (κ=1). The paper also contributes two new algorithms for homomorphic modular reduction for any modulus in the radix approach, closing a previously open gap.

The new WoP-PBS: a padding-free generic LUT over many ciphertexts

The headline algorithmic contribution (Section 4.2, Algorithm 3) is a WoP-PBS — a programmable bootstrap that needs no bit of padding, accepts one or several input LWE ciphertexts, can round/truncate each input to a chosen precision, and can evaluate several LUTs on the same inputs for roughly the price of one. It is built from two TFHE primitives: circuit bootstrapping and (mixed/vertical/horizontal) packing [CGGI20]. The three stages are:

  1. Bit extraction — using generalized PBS [CLOT21] evaluating a scaled (negacyclic) sign function plus homomorphic subtraction, every bit of the encrypted large message is peeled off into its own LWE ciphertext (with a carefully derived correction term εi = ⌊q·2k−2/βi⌋ to avoid sign errors near 0 and q/2, including for non-power-of-two bases).
  2. Circuit bootstrapping — each extracted bit-LWE is converted to a GGSW ciphertext. This is the dominant cost, so the algorithm scales linearly in the number of message bits — until, for very large inputs (about 28 bits with their parameters), the packing stage overtakes it.
  3. Mixed packing — the GGSW bits drive a CMux tree followed by a blind rotation and sample extraction, selecting the LUT output L[m].

Several optimizations are noted: chaining the bit-extraction PBSs from LSB to MSB; replacing the packing key-switches with an initial LWE-to-GLWE KS plus an external product (a la [CCR19]) to shrink keys; using generalized-PBS so the modulus switch directly reads the next bit; and using PBSmanyLUT to do all PBSs of a circuit bootstrap at once. For special LUTs (ReLU, sign in radix mode) only the MSBs matter, so packing collapses and the cost becomes linear in the number of blocks. The paper gives explicit closed-form variance formulae for the circuit-bootstrap and full WoP-PBS noise, derived from the [CLOT21] noise models.

Figure 7 of the paper: cleartext view of the new WoP-PBS on a toy 16-bit message split into blocks m₀..m₃. Each block is bit-extracted and circuit-bootstrapped into GGSW bits, which then drive a CMux tree (vertical packing) plus a final “select in LUT” (blind rotation) to produce L[m].

Comparing the operators: where each bootstrap wins

With all three operators priced by the same framework (and held to the same pfail ≈ 2−35 and 2-norm), the comparison becomes apples-to-apples. The new WoP-PBS (𝒜(this work), in 1/2/4-block variants) scales far better than Tree-PBS for generic LUTs on large precision, and unlike the classical PBS it needs no padding bit. The same framework also compares the new WoP-PBS against the competing padding-free bootstrap of [LMP21] (Figure 9), where the new construction wins at higher precisions. The practical division of labour that emerges: classical PBS for 1–5 bits, and the WoP-PBS (over multi-ciphertext integers) for everything larger, with Tree-PBS rarely the best choice.

Concrete parameters and benchmarks (TFHE-rs / Concrete)

All benchmarks run on TFHE-rs — the Rust successor to Zama’s Concrete library [CJL+20] — on a single Xeon Platinum 8375C, with all parameters produced by the optimization framework, 128-bit security, and (by default) pfail ≤ 2−13.9 (standard score 4). The parameter tables report, per atomic pattern, the LWE dimension n and noise, the GLWE (k, log₂N) and noise, and the decomposition base/level for the key-switch, PBS, packing KS and circuit bootstrap. For example 𝒜(CJP21) set #1 (p=2², ν=3) uses n=615, k=4, N=29; set #6 (p=2⁸, ν=17) uses n=983, k=1, N=214. WoP-PBS sets #8–#12 specify how many bits to extract per block and the number κ of input ciphertexts; sets #11/#12 encode the CRT bases (e.g. Ω=7·8·9·11·13≈216, and 3·11·13·19·23·29·31·32≈232). The framework can also generate compatible pairs of parameter sets so a circuit can switch freely between classical-PBS arithmetic and WoP-PBS LUTs.

Representative timings:

  • 16-bit radix (Ω=216): addition ~6.7–12.8 µs, multiplication ~5.7–29 s, carry cleaning ~0.66–0.93 s, generic LUT via WoP-PBS ~0.82–1.80 s.
  • 16-bit CRT (Ω=7·8·9·11·13): multiply ~0.40 s sequential (~80 ms on 5 threads); a polynomial-function LUT is as cheap as a per-residue carry cleaning. Generic non-CRT-friendly LUT is slower (~23 s) when parameters were tuned for 𝒜(CJP21) compatibility.
  • 16-bit native CRT: multiply ~7.4 s, LUT ~3.8 s sequential (~0.76 s on 5 threads), additions free (~4 µs).
  • 32-bit: radix multiply ~9.6–43.8 s depending on block count; hybrid (CRT over Ω=25·35·54·74) multiply ~1.8 s on 4 threads; native CRT multiply ~5.3 s on 8 threads. Generic LUT on 32-bit integers is deemed impractical. A caution (Remark 12): with 32 single-bit blocks the number of APs in a multiply is quadratic in block count, inflating the overall failure probability — another reason fewer, larger blocks (hybrid) win at 32 bits.

The parameter search itself is fast: most curves are generated in < 1 s; only Tree-PBS (no shortcuts implemented) takes ~26 min.

The full framework: whole-graph failure, PBS insertion, key positions

Section 5 lifts the per-AP analysis to a full-fledged framework (Eq. 4): the input is a crypto-free plain DAG (additions, multiplications, LUTs, with each edge labelled by precision and public/private), and the output is simultaneously the best FHE DAG and its parameter set, chosen over a space of translation rules (e.g. a multiply → two PBSs [CJL+20] or a leveled multiply [CLOT21]). Several features sharpen the result:

  • Whole-graph failure probability: using the domination relation, individual-AP failure probabilities are combined (and iteratively rebalanced via a dominant-AP probability pfail(Adom)=1−(1−pfail(𝒢))1/|I′|) so the user can specify a single target failure probability for the entire use-case rather than per operator (Section 5.2). The relation κ1 between two APs’ standard scores is derived in closed form.
  • Optimal PBS insertion (Theorem 2): when a dot product has a large 2-norm it forces large, slow parameters; the framework can automatically split the DP and insert extra PBSs, trading more bootstraps for smaller, cheaper parameters — sometimes a net win.
  • Key-switch position (Theorem 3): it is analytically always at least as good to place the KS immediately before the PBS (the [CJP21] ordering) as elsewhere (e.g. the [CGGI20] gate-bootstrapping ordering) — the framework formalizes and exploits this.
  • Multiple public keys (Theorems 4–5): allowing several bootstrap/key-switch keys lets high-noise-budget parts of a graph use smaller, faster keys; a monotonicity result prunes the assignment search.
  • Failure-probability spectrum: the cost of tightening pfail from 2−14 down to 2−50 is quantified (Figures 11–12). For the single-ciphertext PBS, tightening can jump the cost when N must quadruple rather than double; for the WoP-PBS the curves merely shift up.
  • Consensus-friendly TFHE: by avoiding the FFT (which is non-deterministic across implementations), an atomic pattern can be made bit-reproducible — useful for blockchain settings where all nodes must agree on the exact ciphertext.

Figure 11 of the paper: cost (log) vs precision for the single-ciphertext PBS (𝒜(CJP21)) at four target failure probabilities pfail ∈ {2−14, 2−20, 2−35, 2−50} and fixed log₂(ν)=4. Each extra bit roughly doubles the cost, and tighter failure targets shift the curve upward (with jumps where N must grow 4×).

Relation to TFHE and takeaways

This work sits squarely on top of the TFHE line [CGGI20] and its programmable bootstrap, and on the generalized PBS / first WoP-PBS of [CLOT21]. Its contribution is twofold and complementary: a methodology (the optimization framework) that finally makes “choose secure + correct + fast parameters” an automatable problem for bootstrapping-heavy schemes, and a set of large-precision constructions (hybrid radix/CRT integers, arbitrary-modulus modular reduction, and the padding-free multi-ciphertext WoP-PBS) that make 16- and 32-bit homomorphic integer arithmetic and generic LUTs practical.

Practically, the results are realized in TFHE-rs, the successor to Zama’s Concrete library, and the framework is the engine behind Concrete’s automatic parameter selection — so the paper is effectively the design document for how Zama’s compiler turns a user’s plain program into a secure, correct, fast FHE circuit. Recurring lessons: for large precision, prefer several small ciphertexts over one big one; the classical TFHE PBS is the right tool only up to ~5 bits; and a faithful, FFT-corrected noise model plus the lattice-estimator are what let the optimizer trust its own answers. Stated future work includes optimizing the integer representation itself (moduli, block counts), optimizing the graph topology, and extending the framework to non-TFHE schemes.

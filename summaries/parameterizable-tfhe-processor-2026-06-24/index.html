Towards a Functionally Complete and Parameterizable TFHE Processor

PDF (text-layer), arXiv:2510.23483 · full view

Overview and Motivation

Fully homomorphic encryption (FHE) lets arbitrary functions be evaluated directly on encrypted data, which makes it attractive for securing outsourced and multiparty computation. TFHE is a fast, torus-based FHE scheme built on the learning with errors (LWE) assumption. It supports linear operations and, through programmable bootstrapping (PBS), the evaluation of arbitrary non-linear lookup tables. TFHE currently offers the fastest bootstrapping of any FHE scheme, yet homomorphic circuit evaluation remains several orders of magnitude slower than the equivalent plaintext computation, which is the main barrier to adoption.

Prior FPGA accelerators mostly offload only the single most expensive step (typically the blind rotation inside PBS) while leaving the rest of the computation on conventional hardware. This forces large ciphertexts and keys to be shuttled across the host-to-FPGA boundary, so overall throughput and scalability become dominated by external memory bandwidth, with prior designs requiring several gigabytes per second. The authors (Reyes Haeusler, Ott, Jayasena, and Peter) propose a fully FPGA-based TFHE accelerator that performs PBS entirely on the FPGA and keeps the keys on chip, eliminating that bandwidth bottleneck. Their improved bootstrapping module delivers 240 percent to 480 percent more bootstrappings per second than the current state of the art. Code and artifacts are open at the project repository.

What "Functionally Complete" and "Parameterizable" Mean Here

Functionally complete means the design targets the full set of operations TFHE needs to evaluate any homomorphic circuit, not just a single accelerated kernel. TFHE messages are stored as LWE ciphertexts; addition is native, scalar multiplication is supported, and arbitrary non-linear functions are realized via programmable bootstrapping with a lookup table. To complete the operation set the authors add a key-switch unit (returning bootstrapped ciphertexts to the original LWE key) that doubles as a fused multiply-add (MulAdd) module. The paper notes that ciphertext-ciphertext multiplication is not a dedicated operation; instead, as in the tfhe-rs library, homomorphic multiplication is done by concatenating messages and performing a PBS with an appropriately constructed lookup table.

Parameterizable means the architecture is designed so that TFHE parameters can be varied without redesign. The key parameters are the LWE mask dimension n, the RLWE polynomial degree N and mask dimension k, the decomposition base and depth, and a throughput knob T (the number of coefficients processed per clock cycle). The authors report that k affects only buffer sizes, while N mainly affects on-chip RAM (BRAM and URAM) blocks that are already partitioned into T sub-buffers. As long as these fit within the FPGA RAM primitives, scaling the parameters proportionally scales performance with negligible per-operation penalty, which lets the same design serve both standard and large parameter sets.

Core Arithmetic: Modular Reduction and the NTT

The dominant cost in TFHE is negacyclic polynomial multiplication; a single PBS requires n times (k+1) times (k+1) times l such multiplications. The design accelerates this with a Number Theoretic Transform (NTT), which gives exact finite-field arithmetic and avoids the rounding errors and elevated decryption-failure risk of fixed-point FFT approaches such as FPT.

The authors adopt the 64-bit NTT-friendly Solinas prime q = 2^64 minus 2^32 plus 1. Its structure allows full modular reduction of a 128-bit product using only a split into four 32-bit segments, three additions, one shift, and a final conditional addition, avoiding costly Barrett or Montgomery reductions. Additions and subtractions use a cheaper simple reduction (one conditional add or subtract of q). Scalar multiplication uses Karatsuba at recursion depth two to cut multiplications from four to three per level.

The NTT is built from Cooley-Tukey butterflies for the forward transform and Gentleman-Sande butterflies for the inverse, so the inverse NTT produces normal-ordered outputs without any bit-reversal permutation network. Butterflies are grouped into per-stage butterfly blocks sized for throughput T, transitioning to a fully parallel final stage. Careful scheduling and inter-stage memory reuse shrink buffering: each stage needs an output buffer of only three-quarters times N over 2^i, and roughly one-quarter of values bypass buffering entirely by streaming directly into the next stage.

Bootstrapping Pipeline: External Product, Blind Rotation, and PBS

The external product (TFHE multiplication) is built in five parts: decomposition, NTT, element-wise multiplication of transformed polynomials, accumulation, and inverse NTT. The decomposition module is pipelined; l NTT modules transform the resulting polynomials in parallel and store them in a ping-pong Multi Polynomial Buffer. The design instantiates (k+1) times l element-wise multipliers so all polynomial-RLWE products run in parallel, with the bootstrapping key precomputed in NTT space (and pre-multiplied by the inverse-NTT rescaling factor) to save resources. Products are summed in a block adder tree before the final inverse NTT.

The blind rotation iterates over the bootstrapping key elements and the LWE mask coefficients, computing acc = (acc external-product BSK_i) times (X^{a_i} minus 1) plus acc per iteration. The external product result is buffered in a ping-pong buffer; the multiplication by X^{a_i} is performed as a negacyclic polynomial rotation in the time domain (chosen over an NTT-domain rotation, which would need large run-time lookup tables), with wrapped coefficients negated. The programmable bootstrapping module wraps an initialization step, the blind rotation, and a sample extraction (which reuses the same polynomial rotation logic). It is pipelined to process multiple ciphertexts at once, taking the input ciphertext address, the bootstrapping-key address, the sample-extract index, and the output address. Crucially, the bootstrapping-key address is not fixed and the sample-extract index is variable, supporting different keys and extraction points.

Key Switching, MulAdd, and Programmability Toward a Processor

A single dual-purpose module provides both key switching and fused multiply-add. Key switching decomposes the incoming mask coefficients, multiplies them by the corresponding key-switch-key entries (kept negated so the whole result is one large sum without sign switching), and accumulates in an addition tree. Because key switching is faster than bootstrapping, its throughput is matched to the PBS module (T_KS at least T_BS over n), and the spare cycles are repurposed: flags deactivate the accumulation to return additions and multiplications directly, giving MulAdd; subtraction uses the modular form of a negative scaling factor.

The step that turns the accelerator into a processor is programmability. Each module is given data addresses to fetch its operands from memory rather than receiving all data from outside, which is what lets a custom instruction set be interpreted and executed on the FPGA instead of the device exposing a single fixed function. The authors describe a minimal instruction set of PBS, AddMul, and KS that needs only two instruction bits plus three 64-bit data addresses and log N bits for the sample-extract index. In the current prototype the PBS module is fully implemented and the dual-purpose key-switch/MulAdd module is designed and shown feasible, but the key-switch module itself, the instruction module, and a compiler are left as future work.

On-Chip Design and Bandwidth

Keeping the whole computation and the keys on chip collapses the external bandwidth requirement. For the largest tested parameter sets at the highest achieved PBS rate, the design needs only about 3 megabits per second of external bandwidth, versus the several gigabytes per second required by prior offloading designs. The dominant internal bandwidth cost is loading the large RGSW bootstrapping-key elements; this is amortized by batched bootstrapping (as in FPT), which keeps each key element in use longer. For the highest-performing configuration (T = 32 at 325 MHz, with k = 1, l = 2, n = 500) a minimum batch size of nine yields about 93 GB/s internal bandwidth for the PBS module plus 73 GB/s for the key-switch module, totaling roughly 166 GB/s, which the on-chip high-bandwidth memory can supply.

Experimental Setup

The design is implemented and evaluated on an AMD Virtex UltraScale+ HBM VCU128 FPGA, which has two HBM stacks providing up to 460 GB/s aggregate bandwidth, manufactured on a 20 nm process. The host communicates with the FPGA over a PCIe Gen2 x8 link (16 GT/s) using the XDMA IP core and memory-mapped I/O; a modified tfhe-rs library on the host drives the XDMA driver. On the FPGA, a custom controller coordinates the XDMA core and an on-chip crossbar that gives the TFHE processor concurrent access to both HBM stacks. The system-level integration is shown below.

Results: NTT and Bootstrapping Throughput

To compare fairly across platforms and parameter choices, the authors baseline on the blind rotation (the heaviest, most widely implemented operation), scale prior 32-bit designs by a factor of two to account for 64-bit arithmetic, and normalize decryption failure rates (DFR) to a common 2^-64 by squaring DFR per repeated computation.

NTT (1024-point). Running at 600 MHz with T = 2, the 64-bit negacyclic NTT achieves about 1172 NTTs per millisecond at 1.17 microseconds latency, the best frequency and throughput among the compared designs. It is only narrowly beaten on latency by YYKKP, which uses over 7 times more DSPs, 5 times more flip-flops, and 7 times more LUTs at half the bit width. Against HLW (same 64-bit modulus) it delivers roughly 9 times the throughput and 6 times lower latency, and it outperforms KL, Proteus, and CM by around 4 times.

PBS on the standard parameter set (n = 500, N = 1024, k = 1, l = 2, log beta = 10, DFR 2^-64). The T = 32 configuration runs at 325 MHz and reaches 10156 PBS per second at 64-bit (20312 normalized 32-bit equivalent), over 3.4 times the throughput of the next fastest FPGA design (FPT) and more than 5.8 times other FPGA implementations, with comparable total resource use. At the best performance-per-resource point (T = 2) it matches ALT, KL, and YKP on PBS rate while using far fewer resources, with about 18 percent higher throughput. Reported PBS/s scale across configurations from 1123 (T = 2) to 10156 (T = 32) at 64-bit, with frequency falling from 575 MHz to 325 MHz as T grows because of longer interconnect and routing congestion (a standalone T = 8 NTT nearly hits 600 MHz, indicating inter-block routing rather than the NTT limits the clock). Energy efficiency is second-best (140 to 171 PBS per watt), behind only ALT; latency trails ALT and FPT, partly because ALT uses smaller composite-NTT multiplications and key unrolling that this design does not yet adopt.

Results: Scalability and Comparison to CPU and GPU

On the demanding large parameter set from Ohba et al. (n = 800, N = 16384, k = 1, l = 5, log beta = 6, DFR 2^-64), the T = 8 configuration runs at 400 MHz and reaches 122 PBS per second, achieving over 30 times higher throughput and 10 times lower latency than the Ohba FPGA baseline, demonstrating the design scales to large parameters.

Against general-purpose hardware (a 16-core AMD 7950X3D and an Nvidia L40 GPU running tfhe-rs), the FPGA delivers a 2.89 times (CPU) and 1.16 times (GPU) speedup on the standard parameter set and 4.35 times (CPU) and 1.12 times (GPU) on the large set, with improving efficiency at scale relative to the CPU and similar scalability to the L40. This is notable because the FPGA is built on a 20 nm node while the CPU and GPU use 5 to 6 nm FinFET processes; despite that disadvantage the T = 32 design reaches over 5 times (28 times) lower latency and 2.8 times (1.16 times) higher throughput than the 16-core CPU (GPU) at similar or lower power.

Conclusion and Future Work

The paper presents a fully FPGA-integrated TFHE accelerator that runs programmable bootstrapping entirely on chip, keeps keys on the device, and thereby removes the off-chip bandwidth bottleneck that limited prior work, reducing external bandwidth from gigabytes per second to about 3 megabits per second. The compact, parameterizable design outperforms the state of the art by 240 percent to 480 percent in bootstrappings per second, scales across standard and large parameter sets, and beats CPU and GPU baselines despite an older process node. The prototype implements PBS and supporting programmable infrastructure, and adds a designed-and-validated dual-purpose key-switch and MulAdd module. Remaining work is to complete the key-switch and instruction modules, define a dedicated instruction set, and build a compiler that translates homomorphic circuits into hardware instructions, with further gains expected from key unrolling and composite NTTs. The authors frame the result as a foundation for future ASIC-based TFHE processors.

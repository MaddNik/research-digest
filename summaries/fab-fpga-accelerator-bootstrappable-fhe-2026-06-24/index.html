FAB: An FPGA-based Accelerator for Bootstrappable Fully Homomorphic Encryption

PDF (text-layer), arXiv:2207.11872 · full view

Motivation: why FHE bootstrapping needs hardware acceleration

Fully Homomorphic Encryption (FHE) lets a third-party cloud compute directly on encrypted data, and is regarded as the gold standard for privacy-preserving machine learning. The obstacle is cost. The authors note that training a logistic regression (LR) model for 30 iterations on plaintext (11,982 samples, 196 features) takes about 1.05 sec on a CPU, whereas the same training on encrypted data takes about 124 minutes on the same CPU, a slowdown of roughly 7086x; the encrypted dataset (198 MB) is 152x larger than the plaintext (1.3 MB).

To support general-purpose unbounded computation, every FHE scheme needs a periodic, expensive operation called bootstrapping, which is both compute-heavy and memory-bandwidth-heavy. Prior accelerators do not solve this: CPUs cannot exploit FHE's parallelism; GPUs have large floating-point units that go unused because FHE is almost entirely integer arithmetic, and neither CPU nor GPU supplies enough main-memory bandwidth. The custom ASIC proposals F1 (Samardzic et al.) and BTS (Kim et al.) use a massive number of processing elements (8192 to 18432) and large on-chip memory (64 to 512 MB), yet still cannot hold the full FHE working set on chip, so they remain bottlenecked by main-memory bandwidth and leave compute units idle.

FAB is the first FPGA accelerator to support full, fully-packed CKKS bootstrapping at a secure, practical parameter set. Its central thesis is that the right design point is not more compute, but a balanced design that is not memory bound: a minimal number of functional units, a low clock frequency, high data rates to and from main memory, careful use of the limited on-chip memory, and careful operation scheduling. FAB uses only commercially available FPGAs (e.g. AWS F1 cloud), making near-ASIC performance broadly accessible.

Background: the CKKS scheme, RNS, NTT, and key switching

FAB targets the CKKS scheme, which operates over vectors of complex numbers. A plaintext is an element of C^n. CKKS is parameterized by the plaintext slot count n, a ciphertext coefficient modulus Q, and a polynomial modulus x^N + 1 with N a power of two. Ciphertexts live in R_Q^2 where R_Q = Z_Q[x]/(x^N + 1); an encryption of m is written as a pair (a_m, b_m). The supported homomorphic operations are component-wise Add, component-wise Mult, Rotate (rotate the vector by k slots), and Conjugate (complex conjugate). For example, the k=1 rotation of (x_0, x_1, ..., x_{n-2}, x_{n-1}) is (x_{n-1}, x_0, ..., x_{n-3}, x_{n-2}).

RNS representation and levels. Q is written as a product of word-sized primes q_1, ..., q_l (the RNS basis B). Each q_i is a limb; a modulus Q with l factors has l-1 levels. Each multiplication consumes one limb (one level); Add, Rotate, and Conjugate do not change the level. A value x in Z_Q is stored as a length-l vector [x]_B = (x_1, ..., x_l) with x_i = x mod q_i, so add and multiply are done limb-wise on machine words: x_i + y_i = x + y (mod q_i) and x_i * y_i = x * y (mod q_i). RNS recombination (converting to a new modulus p in an extended basis) is given by:

[x]_p = sum_{i=1..l} [ x_i * Qtilde_i ]_{q_i} * Q*_i   (mod p)

where  Q*_i = Q / q_i   and   Qtilde_i = (Q*_i)^{-1} (mod q_i)

In memory a ciphertext is an l x n matrix: rows are limbs, columns are coefficients. Row-major (locality within a limb) is called limb-wise access; column-major (locality across limbs for one slot) is slot-wise access.

NTT. To multiply elements of R_Q efficiently, CKKS uses the Number Theoretic Transform, the finite-field analog of the FFT modulo q. Polynomials are kept by default in evaluation representation (N evaluations at fixed roots of unity), giving O(N) multiplication; NTT itself is O(N log N) time and O(N) space. Operations on the coefficient representation require an inverse NTT (iNTT). Separately, CKKS uses a complex FFT on the client during encryption/decryption; during bootstrapping that FFT must be evaluated homomorphically on the encrypted data.

Switching keys and KeySwitch. Mult, Rotate, and Conjugate each produce an intermediate ciphertext decryptable under a different secret key, so they all invoke the KeySwitch subroutine to restore the original key; KeySwitch is the major bottleneck of low-level operations. FAB uses the switching-key structure of Han and Ki, parameterized by length dnum, a 2 x dnum matrix of polynomials:

ksk = [ a_1  a_2  ...  a_dnum ]
      [ b_1  b_2  ...  b_dnum ]

During key switching the ciphertext modulus is raised from Q to P*Q, where P is a fixed product of extension limbs (there are alpha+1 extension limbs); Q is split into at most dnum equal-size digits and P must exceed the largest product of limbs in one digit. P*Q is the maximum modulus for which security must hold. FAB reuses the standard subroutines Decomp, ModUp, ModDown, KeySwitchInnerProd (called KSKIP), and Automorph.

Bootstrapping and the bootstrapping performance metric

To compute indefinitely on a CKKS ciphertext, the modulus Q must periodically be raised while preserving ciphertext structure; this is bootstrapping, the main bottleneck of FHE and the central target of FAB. At a high level bootstrapping has three steps: a linear transform (a homomorphic FFT), a polynomial evaluation, and a second linear transform (the inverse FFT). All three are built from the same Add, Mult, Rotate, Conjugate operations. FAB uses the same polynomial as Bossuat et al. to support non-sparse CKKS secret keys; that polynomial evaluation has multiplicative depth 9. The two linear transforms have a depth-performance trade-off parameterized by fftIter, the chosen multiplicative depth of the FFT algorithm.

The efficiency metric is the amortized multiplication time per slot, T_Mult,a/slot:

T_Mult,a/slot := ( T_Boot + sum_{i=1..l} T_Mult(i) ) / ( l * n )

Here n is the number of slots, T_Boot is the bootstrapping time, and T_Mult(i) is the multiply time at level i. The number of usable levels l after bootstrapping equals the maximum supported levels in the starting bootstrapping modulus minus the bootstrapping depth, which for FAB's algorithm is:

L_Boot := 2 * fftIter + 9

Choosing a practical, secure parameter set

FAB prototypes on the Xilinx Alveo U280. N must be a power of two for NTT efficiency; the largest power of two whose limbs still fit in on-chip memory is N = 2^16. With that N, the maximum supportable ciphertext modulus is log(PQ) = 1728, which gives 128-bit security. A single ciphertext is at most 28.3 MB (max 32 raised limbs), which fits in the 43 MB of on-chip memory, so data movement to and from main memory is minimized. The final parameter set is:

logq = 54    N = 2^16    L = 23    dnum = 3    fftIter = 4    lambda = 128

Limb bit-width (logq = 54). 54 bits enables effective use of the DSP slices' 18-bit multipliers and 27-bit preadders via multi-word arithmetic: 54-bit operands split into three 18-bit operands for multiplication and two 27-bit operands for addition. 54 (a multiple of 18) also packs neatly into on-chip memory, where URAM stores 72-bit words and BRAM stores 18-bit words, so combining blocks fully uses their data width.

dnum = 3. Figure 1 shows that increasing dnum adds compute levels after bootstrapping but also enlarges the KeySwitch keys, raising compute and on-chip-memory cost. dnum = 3 best matches the available on-chip memory to the resulting key size (with a key-compression technique from de Castro et al. halving key size). fftIter = 4. Figure 2 shows the effect of fftIter on bootstrapping time: larger fftIter increases multiplicative depth (fewer compute levels left) but lowers the FFT sub-matrix radix, needing fewer rotations per multiply; fftIter = 4 balances rotations, NTTs and post-bootstrap levels, fixing L_Boot = 2*4 + 9 = 17.

Overall system architecture

The system has four components: a host CPU (x86) that offloads the RTL design and data to the FPGA; the RTL design packaged as kernel code; global memory on the FPGA made of two HBM2 stacks (4 GB each); and a 100G Ethernet (CMAC) subsystem that lets FPGAs exchange data directly without the host. The host attaches to the Alveo U280 over PCIe and uses that link to fill a global-memory buffer with the dataset. The host communicates the buffer base address and all kernel arguments (prime moduli, polynomial degree N, precomputed scalars for the register file) over an AXI4-Lite interface using atomic register reads/writes, then starts the kernel via a Xilinx Runtime (XRT) API call (replaceable by an OpenCL call with trivial changes). Once the kernel runs, no host-to-global-memory transfer occurs; all 32 AXI ports from HBM feed the kernel, and results return to the host on completion.

The kernel instantiates a functional unit of 256 modular-arithmetic and automorph units, a 2 MB register file (RF) holding system parameters, precomputed scalars, and up to four intermediate polynomials, and 32 memory-mapped 256-bit AXI4 master interfaces for bidirectional global-memory transfers. Read (Rd) and Write (Wr) FIFOs stream data between global and on-chip memory; Transmit (Tx) and Receive (Rx) FIFOs stream to/from the CMAC subsystem. URAM and BRAM are organized into banks for on-chip memory; all URAM banks are single-port (URAM has no dual-port mode) while BRAM banks are dual-port. The CMAC core runs an internal 322 MHz clock and a 256/512-bit kernel interface; FAB implements a 512-bit interface, giving about 153 Gbps at 300 MHz (faster than the Ethernet IP), versus about 76 Gbps for a 256-bit interface. With the 512-bit interface, transmitting a single ciphertext limb (a 0.4 MB polynomial) takes about 11,399 clock cycles, and the entire ciphertext about 546,980 cycles.

FAB microarchitecture: the functional unit and modular arithmetic

Every FHE operation reduces to integer modular addition and modular multiplication, so each of the 256 functional units contains a modular multiplier, modular adder, modular subtractor, and an automorph unit. Multi-word arithmetic reduces 54-bit operations to 27-bit operations for addition and 18-bit operations for multiplication, matching the DSP blocks.

Modular add/subtract. FAB follows Hankerson et al. Algorithms 2.7 (add) and 2.8 (subtract). Both algorithms' correction step (line 2) would otherwise need a full 54-bit operation; FAB rewrites the correction step into multiple 27-bit operations. With multi-word arithmetic and full DSP pipeline registers, both run in 7 clock cycles.

Modular multiply. Done as an integer multiply followed by a modular reduction, pipelined. The integer multiply follows the operand-scanning (schoolbook) multi-word algorithm, Hankerson et al. Algorithm 2.9. With 54-bit operands split into three 18-bit operands, a naive version needs 21 cycles per multiply; since roughly 50% of FHE operations are integer multiplies, FAB loop-unrolls and parallelizes the algorithm to cut this to 12 cycles while keeping all DSP pipeline registers.

Modular reduction. FAB proposes a hardware-friendly fast reduction by modifying Will and Ko's shift-and-add technique (which avoids the expensive multiplications of Barrett reduction). Will and Ko's method shifts one bit at a time and takes 2*logq cycles (108 cycles for logq = 54). FAB's Algorithm 1 performs multiple bit shifts at once (shifts = 6), reducing this to 12 cycles for logq = 54. It precomputes a 63-element array madd of logq-bit elements; reduction is needed against 32 prime moduli, so 32 such madd arrays are precomputed offline (about 7 KB total, no runtime overhead). The algorithm is generic in the number of shifts:

Algorithm 1  Modular Reduction in F_q

1:  Modulus q, integer a, shifts = 6           > a is (2 logq - 1) bits
2:  Precompute: madd[i-1] = sum_{j=0..5} i[j] * 2^{logq + j}  (mod q)
        for i = 1 to 2^shifts - 1              > madd has logq-bit elements
3:  Set (A[1], A[0]) <- a,  count = 0,  as1 = 0
4:  while count < logq do
5:      (carry, as1) = A[1] >> shifts          > carry is shifts bit
6:      A[1] = as1 + madd[carry - 1]
7:      count = count + shifts
8:  end while
9:  c = A[1] + A[0]
10: if c > q then
11:     c = c - q
12: end if
13: return c

Automorph. The automorph unit performs the permutation behind Rotate: it reads a polynomial from on-chip memory and writes it to the register file in permuted order for rotation index k. The slot at index i maps to a new slot via:

new_index_k(i) = ( (5^k - 1) / 2 ) + 5 * i   (mod N)

Because bootstrapping uses only about 60 distinct rotation indices, the powers of 5 are precomputed. The division by two is a bit-shift, and since N is a power of two, reduction mod N is just an AND with N-1.

On-chip memory and register file

The U280 has 962 URAM blocks (288 Kb each, single-port) and 4032 BRAM blocks (18 Kb each, single- or dual-port). FAB combines them into banks giving 43 MB total capacity at about 30 TB/s internal SRAM bandwidth.

URAM (Figure 4a). Each URAM block is 72 bits wide, depth 4096. Three URAMs combine to 216 bits, holding four 54-bit coefficients per address (216 / 4 = 54). Laying out 64 of these 216-bit-wide URAMs into one bank stores 256 coefficients, so one read/write touches 256 coefficients per cycle, matching the 256 functional units. A bank is thus 64 x 3 = 192 URAMs and holds 16 polynomials (about 7.08 MB). FAB uses five URAM banks: c0 bank-1 and bank-2 hold 32 limbs (24 original + 8 extension) of the c0 ring element; c1 bank-1 and bank-2 hold 32 limbs of the c1 ring element; the fifth miscellaneous bank holds 16 polynomials and stores twiddle factors, KeySwitch keys, and plaintext vectors read from main memory.

BRAM (Figure 4b). Three 18-bit BRAMs combine to a 54-bit-wide word, so 256 BRAM blocks store 256 coefficients. Each BRAM is depth 1024; stacking two gives depth 2048, holding 8 polynomials per bank. FAB has three BRAM banks: two banks of 1536 BRAMs each (8 polynomials, ideal for extension limbs) and a third miscellaneous bank of 768 BRAMs (4 polynomials) for temporary main-memory data.

Register file. Multiple RFs totaling 2 MB are spread across the functional, address-generation, and control units, each with multiple same-cycle read/write ports. About one quarter holds host-written precomputed values and parameters; the rest stores up to four intermediate polynomials from Rotate or Mult.

FIFOs. 32 synchronous Wr and Rd FIFOs (one per HBM-side AXI port) stream between main and on-chip memory, each 256 bits wide (the AXI port width). Wr FIFO depth is 128 (matching an HBM burst length of 128); Rd FIFO depth is 512 (up to four outstanding reads). Rd FIFO runs in the 450 MHz memory clock domain, Wr FIFO in the 300 MHz kernel domain. Tx and Rx FIFOs to the CMAC subsystem are 512-bit synchronous FIFOs.

NTT/iNTT datapath

FAB uses a unified Cooley-Tukey algorithm for both NTT and iNTT, so the same data-mapping logic serves both. The 256 modular add, subtract, and multiply units operate in parallel as radix-2 butterfly units, processing 512 coefficients of a polynomial per pass. This lets FAB run the log N stages in roughly log N * (N / 512) cycles instead of log N * (N / 2) cycles. The NTT address-generation unit uniquely maps data within each stage via a data-mapping sub-unit, and a twiddle-factor mapping sub-unit reads the needed twiddle factors for a stage from the URAM miscellaneous bank. Both sub-units use data and stage counters to generate addresses on the fly with cheap shift and AND operations. The latency of bit-reversal is not counted separately because bit-reversal is folded into the automorph/multiplication step performed just before NTT/iNTT.

KeySwitch datapath: modified dataflow and smart scheduling

KeySwitch comprises four sub-operations: Decomp, ModUp, KSKIP, and ModDown. The challenge is that KeySwitch must operate on the extension limbs (factors of P) and compute an inner product with KeySwitch keys that are nearly three times the ciphertext size. Decomp splits the limbs of a ring element into dnum digits; each digit feeds ModUp, which outputs L + 1 + alpha limbs, so after all ModUp operations there are dnum * (L + 1 + alpha) limbs. With dnum = 3 and alpha = 8, that is 3 * 32 = 96 limbs. The total live data is about 112 MB (84 MB keys + 28 MB ciphertext), which must be managed within 43 MB of on-chip memory without writing any resultant limb back to main memory.

Modified datapath (Figure 5). The original datapath (Figure 5a) runs Decomp, then the full ModUp, then the full KSKIP, then ModDown in sequence; because the extended polynomials do not fit on chip, it reads input limbs in evaluation form for Decomp, writes them out in coefficient form after ModUp, then reads them back for KSKIP, incurring heavy DRAM traffic. FAB instead splits KSKIP into two steps and greedily makes progress on the inner product as soon as operands are on chip. The key observation is that the alpha limbs of each ModUp operation do not change, so KSKIP can begin on them right after Decomp. Decomp takes the L limbs and splits them into beta <= dnum blocks of alpha limbs each; these alpha limbs follow two paths: (1) they begin KSKIP, with the intermediate sum written to URAM, and (2) they feed the basis-conversion step to generate the extension limbs, which then complete KSKIP. This both reduces the number of NTT computations (the most expensive KeySwitch subroutine) and reduces memory traffic. The split does not change the underlying KeySwitch algorithm or its noise; only the order of steps changes.

Smart operation scheduling. The RNS extension equation (Equation 1) naively needs 2*l multiplications per generated limb p_j. FAB observes that one factor, x_i * Qtilde_i (mod q_i), is independent of the output modulus, so the l inner products of form x_i * Qtilde_i are computed once and reused for every extension limb p_j, halving the modular multiplications. Scheduling also generates an entire block of extension limbs at once for a given block of alpha original limbs; as soon as a block's extension limbs are ready, an NTT followed by KSKIP runs on them before the next block is generated, repeating dnum times to produce all dnum * (L + 1 + alpha) limbs. Only the key block matching the just-extended alpha block is fetched (not all keys at once). All extension limbs are kept in the c0 BRAM bank, whose dual-port BRAMs allow simultaneous read and write for limb-wise inner products in BasisConvert; this lets the whole KSKIP run using a 4-limb BRAM miscellaneous bank while hiding the roughly 300-cycle key-read latency behind computation. ModDown (step 5) is scheduled like ModUp and repeated for the c1 polynomial using the URAM c1 bank. The net effect is high data reuse, exploitation of limb-wise parallelism, uniform address generation (no switching between limb-wise and slot-wise access), and no writing/reading of resultant ciphertext limbs to main memory. The optimization is specific to the U280 but portable to smaller FPGAs as long as at least one key limb and one ciphertext polynomial fit on chip (otherwise BasisConvert would need to go fully slot-wise to stay compute-bound).

Evaluation setup and resource utilization

FAB is written in Verilog 2001, synthesized with Xilinx Vivado 2020.2 at 300 MHz, packaged as kernel code with Vitis 2020.2 into an .xclbin, and run on an Alveo U280 (Xilinx 16nm UltraScale, 8 GB HBM2, up to 460 GB/s). Resource utilization: 899,232 LUTs (68.96%, of which functional units are about 37%), 2,073K flip-flops (79.54%), 5,120 DSPs (56.70%, entirely for modular arithmetic), 3,840 BRAMs (95.24%), and 960 URAMs (99.80%).

Resource comparison against the ASIC proposals highlights FAB's frugality: F1 (N=2^14, logq=32) uses 18432 modular multipliers, an 8 MB register file, and 64 MB on-chip memory; BTS (N=2^17, logq=50) uses 8192 modular multipliers, a 22 MB register file, and 512 MB on-chip memory; FAB (N=2^16, logq=54) uses just 256 modular multipliers, a 2 MB register file, and 43 MB on-chip memory. Versus BTS, FAB uses 32x fewer modular multipliers, an 11x smaller register file, and 12x smaller on-chip memory.

Results: basic operations, bootstrapping, and LR training

Basic CKKS operations vs GPU. Against the optimized GPU implementation of Jung et al. (closest parameter set N=2^16, logQ=1693, 100-bit security), FAB execution times (ms) and speedups are: Add 0.04 vs 0.16 (3.85x); Mult 1.71 vs 2.96 (1.73x); Rescale 0.19 vs 0.49 (2.62x); Rotate 1.57 vs 2.55 (1.62x). Average speedup about 2.4x in absolute time.

Throughput vs HEAX (FPGA). At the matching HEAX parameters (N=2^14, logQ=438), FAB throughput (operations/sec): NTT 167K vs 42K (3.97x); Mult 5.7K vs 2.6K (2.12x). Average about 3x higher throughput than HEAX.

Bootstrapping (amortized per-slot multiply time T_Mult,a/slot). FAB runs at 0.3 GHz with 2^15 slots at 0.477 us. Comparisons: Lattigo (CPU, 3.5 GHz, 2^15 slots) 101.78 us, FAB is 213x faster in time and 2485x in cycles; GPU-1 (1.2 GHz, 97-bit, 2^15 slots) 0.740 us, FAB 1.55x (time) / 6.35x (cycles); GPU-2 (1.2 GHz, 173-bit, 2^16 slots) 0.716 us, FAB 1.50x (time) / 6.14x (cycles); F1 (1 GHz, 1 slot) 254.46 us, FAB 533x (time) / 1775x (cycles); BTS-2 (1.2 GHz, 2^16 slots) 0.0455 us, where FAB is 0.09x (time) / 0.38x (cycles), i.e. FAB is about 9x slower in absolute time and 4x slower in cycles than BTS's best case. The authors estimate that if FAB were scaled to 8192 modular multipliers and 512 MB on-chip memory it would be at least 3x faster than BTS, owing to its algorithmic and microarchitectural optimizations.

Logistic regression training. The target application trains a binary LR classifier on the MNIST 3-vs-8 subset (11,982 samples, 196 features), the same HELR task used by all baselines: 30 iterations, 1024 encrypted images per mini-batch, evaluation depth 150, requiring a bootstrap after every iteration. Training uses sparsely-packed ciphertexts (256 slots) for fair comparison with GPU-2 and BTS-2. FAB-1 is a single-FPGA design (about 6.65 GB of ciphertexts and KeySwitch keys offloaded to HBM2). FAB-2 scales to eight FPGAs in pairs (primary/secondary, one master that can broadcast a ciphertext), using direct FPGA-to-FPGA network communication; each FPGA processes 128 ciphertexts, inter-FPGA communication happens only twice per LR iteration with about 12 ms overhead per iteration. Average training time per iteration (sec): Lattigo 37.05; GPU-2 0.775; F1 1.024; BTS-2 0.028; FAB-1 0.103; FAB-2 0.081. FAB-2 is thus 456x faster than CPU and 9.5x faster than GPU-2 in absolute time (for the baselines' N=2^17, logQ=2395 parameters), 12x faster than F1, and competitive with BTS-2. FAB-2 does not reach an 8x speedup over FAB-1 because parallelism is limited by the single-FPGA bootstrapping runtime (Amdahl's law); only the non-bootstrapping operations within an iteration are parallelized.

Versus leveled FHE. The leveled alternative (send exhausted ciphertexts back to the client to decrypt and re-encrypt) has two drawbacks. First, it leaks intermediate values to the client; defending against this needs a lambda-bit mask (minimum justifiable lambda = 40), and because the CKKS plaintext space is not finite this requires extra plaintext precision and substantially larger parameters, increasing LR time. Second, even ignoring leakage, FAB still wins: one LR iteration consumes 5 compute levels and a bootstrap, and FAB does it in 0.103 sec including bootstrapping, whereas just the client-side re-encryption (SEAL subroutines, 2.8 GHz CPU) already costs 0.162 sec, before adding cloud computation and client-cloud communication.

Related work and conclusion

Related work. CPU libraries (SEAL, HElib, PALISADE, Lattigo, HEAAN) remain impractical for FHE; several GPU works accelerate only NTT, and Jung et al. give the state-of-the-art full GPU CKKS (including bootstrapping). On ASICs, F1 (Samardzic et al.) supports multiple schemes (BGV, CKKS, GSW) but only small parameter sets, making it incompatible with multi-slot bootstrapping and real-time ML; BTS (Kim et al.) targets CKKS bootstrapping with many processing elements and large on-chip memory and register file, but does not modify the algorithm to optimize for memory bandwidth, leaving performance unrealized relative to its resources. On FPGAs, HEAX accelerates only CKKS encrypted multiplication (other operations go to the host) and supports only small parameter sets (depth up to 10), insufficient for bootstrapping or LR training. FAB is the first to implement full-scale CKKS bootstrapping on an FPGA at secure parameters supporting an end-to-end application.

Applicability to other schemes. FAB's basic Add, Mult, and Rotate are common across schemes and reusable for BGV and B/FV. The KeySwitch-datapath and smart-scheduling ideas are also relevant to TFHE and FHEW (which evaluate Boolean gates and carry several-GB key footprints), though the exact KeySwitch steps differ per scheme and need fresh analysis for a balanced design.

Conclusion. FAB combines algorithmic and architectural optimizations to overcome the memory-bandwidth bottleneck and perform the first fully-packed CKKS bootstrapping on an FPGA at a practical, secure parameter set. Through smart operation scheduling and on-chip memory management it delivers 456x and 6.5x better LR-training performance than CPU and GPU respectively, reaches performance comparable to costly ASIC designs while using only commercially available hardware, and is immediately deployable in public cloud environments (e.g. AWS F1).

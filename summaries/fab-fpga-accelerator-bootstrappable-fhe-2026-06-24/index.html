FAB: An FPGA-based Accelerator for Bootstrappable Fully Homomorphic Encryption

arXiv preprint (arXiv:2207.11872, cs.CR, 2022) · July 2022 · full view ↗

Motivation: why FHE bootstrapping needs hardware acceleration

Fully Homomorphic Encryption (FHE) lets a third-party cloud compute directly on encrypted data, and is regarded as the gold standard for privacy-preserving machine learning training and inference in an untrusted cloud. The obstacle is cost. The authors note that training a logistic regression (LR) model for $30$ iterations on plaintext ($11{,}982$ samples, $196$ features) takes about $1.05$ sec on a CPU, whereas the same training on encrypted data takes about $124$ minutes on the same CPU, a slowdown of roughly $7086\times$; the encrypted dataset ($198$ MB) is $152\times$ larger than the plaintext ($1.3$ MB).

To support general-purpose unbounded computation, every FHE scheme needs a periodic, expensive operation called bootstrapping, which is both compute-heavy and memory-bandwidth-heavy. Prior accelerators do not solve this. CPUs cannot adequately exploit FHE's inherent parallelism. GPUs have massive floating-point units that go unused because FHE is almost entirely integer arithmetic, and neither CPU nor GPU supplies enough main-memory bandwidth for the data-intensive FHE working set. The custom ASIC proposals F1 (Samardzic et al.) and BTS (Kim et al.) use a massive number of processing elements ($8192$ to $18432$) and large on-chip memory ($64$ to $512$ MB), yet still cannot hold the full FHE working set on chip, so they remain bottlenecked by main-memory bandwidth and leave compute units idle.

FAB is the first FPGA accelerator to support full, fully-packed CKKS bootstrapping at a secure, practical parameter set. Its central thesis is that the right design point is not more compute, but a balanced design that is not memory bound: a minimal number of functional units, a low clock frequency, high data rates to and from main memory, careful use of the limited on-chip memory, and careful operation scheduling. FAB uses only commercially available FPGAs (e.g. the AWS F1 cloud), making near-ASIC performance broadly accessible. For bootstrapping a fully-packed ciphertext at $300$ MHz, FAB outperforms state-of-the-art CPU and GPU implementations by $213\times$ and $1.5\times$ respectively, and on the end-to-end LR-training application (scaled to eight FPGAs) by $456\times$ (CPU) and $6.5\times$ (GPU), while being competitive with the costly ASIC design BTS.

Background: the CKKS scheme, RNS, NTT, and key switching

FAB targets the CKKS scheme, which operates over vectors of complex numbers. A plaintext is an element of $\mathbb{C}^n$, and the plaintext operations are component-wise addition and component-wise multiplication. CKKS is parameterized by the plaintext slot count $n$ (with $n \le N/2$ required), a ciphertext coefficient modulus $Q \in \mathbb{Z}$, and a polynomial modulus $x^N + 1$ with $N$ a power of two. Ciphertexts live in $R_Q^2$ where $R_Q := \mathbb{Z}_Q[x]/(x^N + 1)$; an encryption of a vector $m \in \mathbb{C}^n$ is written as a pair $\llbracket m \rrbracket = (a_m, b_m)$ with $a_m, b_m \in R_Q$. The supported homomorphic operations are component-wise $\mathrm{Add}(\llbracket m_1 \rrbracket, \llbracket m_2 \rrbracket) \to \llbracket m_1 + m_2 \rrbracket$, component-wise $\mathrm{Mult}(\llbracket m_1 \rrbracket, \llbracket m_2 \rrbracket) \to \llbracket m_1 \odot m_2 \rrbracket$, $\mathrm{Rotate}(\llbracket m \rrbracket, k) \to \llbracket \varphi_k(m) \rrbracket$ (rotate the vector by $k$ slots), and $\mathrm{Conjugate}(\llbracket m \rrbracket) \to \llbracket \bar{m} \rrbracket$ (complex conjugate). For example, the $k = 1$ rotation maps $x = (x_0, x_1, \dots, x_{n-2}, x_{n-1})$ to $\varphi_1(x) = (x_{n-1}, x_0, \dots, x_{n-3}, x_{n-2})$.

RNS representation and levels. $Q$ is written as a product of word-sized primes over an RNS basis $B := \{q_1, \dots, q_\ell\}$, so $Q := \prod_{i=1}^{\ell} q_i$. Each $q_i$ is a limb; a modulus with $\ell$ factors supports $\ell - 1$ levels and a circuit of multiplicative depth $\ell$ before bootstrapping is required. Each multiplication consumes one limb (one level); Add, Rotate, and Conjugate do not change the level. A value $x \in \mathbb{Z}_Q$ is stored as a length-$\ell$ vector $[x]_B = (x_1, \dots, x_\ell)$ with $x_i \equiv x \pmod{q_i}$, so add and multiply are done limb-wise on machine words: $x_i + y_i \equiv x + y \pmod{q_i}$ and $x_i \cdot y_i \equiv x \cdot y \pmod{q_i}$, with no native multi-precision support needed. RNS recombination, which takes an RNS representation $[x]_B$ and outputs $x \bmod p$ for a new modulus $p$ in an extended basis, is given by Equation 1:

$$[x]_p = \sum_{i=1}^{\ell} \big[\, x_i \cdot \tilde{Q}_i \,\big]_{q_i} \cdot Q_i^{*} \pmod{p}, \qquad Q_i^{*} = \frac{Q}{q_i}, \quad \tilde{Q}_i = (Q_i^{*})^{-1} \pmod{q_i}$$

In memory a ciphertext is an $\ell \times n$ matrix: rows are limbs, columns are coefficients modulo $Q$. Row-major order (locality within one limb) is called limb-wise access; column-major order (locality across limbs for one slot) is slot-wise access.

NTT. To multiply elements of $R_Q$ efficiently, CKKS uses the Number Theoretic Transform, the finite-field analog of the FFT modulo $q$. Polynomials are kept by default in evaluation representation ($N$ evaluations at fixed roots of unity), giving $O(N)$ multiplication instead of $O(N^2)$; the NTT itself runs in $O(N \log N)$ time and $O(N)$ space for a degree-$(N-1)$ polynomial. Operations on the coefficient representation require an inverse NTT (iNTT) to move back from the evaluation representation. Separately, CKKS uses a complex FFT on the client during encryption and decryption; during bootstrapping that FFT must be evaluated homomorphically on the encrypted data.

Switching keys and KeySwitch. Mult, Rotate, and Conjugate each produce an intermediate ciphertext decryptable under a different secret key, so they all invoke the KeySwitch subroutine to restore the original key; KeySwitch is the major bottleneck of low-level operations. KeySwitch takes a switching key $\mathrm{ksk}_{s \to s'}$ and a ciphertext $\llbracket m \rrbracket_s$ decryptable under $s$, and produces $\llbracket m \rrbracket_{s'}$ encrypting the same message but decryptable under $s'$. FAB uses the switching-key structure of Han and Ki, parameterized by length $\mathrm{dnum}$, a $2 \times \mathrm{dnum}$ matrix of polynomials shown in Equation 2:

$$\mathrm{ksk} = \begin{pmatrix} a_1 & a_2 & \cdots & a_{\mathrm{dnum}} \\ b_1 & b_2 & \cdots & b_{\mathrm{dnum}} \end{pmatrix}$$

During key switching the ciphertext modulus is raised from $Q$ to $P \cdot Q$, where $P$ is a fixed product of extension limbs (there are $\alpha + 1$ extension limbs, $\alpha = \lceil (L+1)/\mathrm{dnum} \rceil$); $Q$ is split into at most $\mathrm{dnum}$ equal-size digits and $P$ must exceed the largest product of the limbs in one digit. $P \cdot Q$ is the maximum modulus for which security must hold. FAB reuses the standard subroutines Decomp, ModUp, ModDown, KeySwitchInnerProd (called KSKIP), and Automorph.

Bootstrapping and the bootstrapping performance metric

To compute indefinitely on a CKKS ciphertext, the modulus $Q$ must periodically be raised while preserving ciphertext structure; this is bootstrapping, the main bottleneck of FHE and the central target of FAB. At a high level bootstrapping has three steps: a linear transform (a homomorphic FFT), a polynomial evaluation, and a second linear transform (the inverse FFT). All three are built from the same Add, Mult, Rotate, Conjugate operations. FAB uses the same polynomial as Bossuat et al. to support non-sparse CKKS secret keys; that polynomial evaluation has multiplicative depth $9$. The two linear transforms have a depth-performance trade-off parameterized by $\mathrm{fftIter}$, the chosen multiplicative depth of the FFT algorithm.

The efficiency metric, first introduced by Jung et al., is the amortized multiplication time per slot, $T_{\mathrm{Mult},a/\mathrm{slot}}$, given in Equation 3:

$$T_{\mathrm{Mult},a/\mathrm{slot}} := \frac{T_{\mathrm{Boot}} + \sum_{i=1}^{\ell} T_{\mathrm{Mult}}(i)}{\ell \cdot n}$$

Here $n$ is the number of slots in the ciphertext, $T_{\mathrm{Boot}}$ is the bootstrapping time, and $T_{\mathrm{Mult}}(i)$ is the time to multiply at level $i$. The number of usable levels $\ell$ in the resulting ciphertext equals the maximum supported levels in the starting bootstrapping modulus minus the depth of bootstrapping, which for FAB's algorithm is given in Equation 4:

$$L_{\mathrm{Boot}} := 2 \cdot \mathrm{fftIter} + 9$$

Choosing a practical, secure parameter set

FAB prototypes on the Xilinx Alveo U280. $N$ must be a power of two for NTT efficiency; the largest power of two whose limbs still fit in on-chip memory is $N = 2^{16}$. With that $N$, the maximum supportable ciphertext modulus is $\log(PQ) = 1728$, which gives a $128$-bit security level. A single ciphertext is at most $28.3$ MB (based on the maximum number of raised limbs, $32$), which fits in the $43$ MB of on-chip memory, so data movement to and from main memory is minimized. The final parameter set is $\log q = 54$, $N = 2^{16}$, $L = 23$, $\mathrm{dnum} = 3$, $\mathrm{fftIter} = 4$, $\lambda = 128$.

Limb bit-width ($\log q = 54$). A $54$-bit width enables effective use of the DSP slices' $18$-bit multipliers and $27$-bit preadders via multi-word arithmetic: the $18 \times 27$-bit DSP multipliers let $54$-bit operands split into three $18$-bit operands for multiplication, and into two $27$-bit operands for addition. $54$ (a multiple of $18$) also packs neatly into on-chip memory, where URAM stores $72$-bit words and BRAM stores $18$-bit words, so combining blocks fully uses their data width.

$\mathrm{dnum} = 3$. Figure 1 shows that increasing $\mathrm{dnum}$ adds compute levels after bootstrapping but also enlarges the KeySwitch keys, raising compute and on-chip-memory cost. $\mathrm{dnum} = 3$ best matches the available on-chip memory to the resulting key size (the authors also use a key-compression technique that halves key size).

$\mathrm{fftIter} = 4$. Figure 2 shows the effect of $\mathrm{fftIter}$ on bootstrapping time (measured in the amortized per-slot multiplication metric of Equation 3): larger $\mathrm{fftIter}$ increases multiplicative depth (fewer compute levels left) but lowers the FFT sub-matrix radix, needing fewer rotations per multiply. $\mathrm{fftIter} = 4$ balances rotations, NTTs and post-bootstrap levels, fixing $L_{\mathrm{Boot}} = 2 \cdot 4 + 9 = 17$ via Equation 4.

Overall system architecture

As shown in Figure 3, the system has four components: a host CPU (x86) that offloads the RTL design and data to the FPGA; the RTL design packaged as kernel code; global memory on the FPGA made of two HBM2 stacks ($4$ GB each); and a 100G Ethernet (CMAC) subsystem that lets FPGAs exchange data directly without involving the host. The host attaches to the Alveo U280 over PCIe and uses that link to fill a global-memory buffer with the dataset. The host communicates the buffer base address and all kernel arguments (prime moduli, polynomial degree $N$, and precomputed scalars for the register file) over an AXI4-Lite interface using atomic register reads and writes, then starts the kernel via a Xilinx Runtime (XRT) API call (replaceable by an OpenCL call with trivial host-code changes). Once the kernel runs, no host-to-global-memory transfer occurs; all $32$ AXI ports from HBM feed the kernel, and results return to the host on completion.

The kernel instantiates a functional unit of $256$ modular-arithmetic and automorph units, a $2$ MB register file (RF) holding system parameters, precomputed scalars, and up to four intermediate polynomials, and $32$ memory-mapped $256$-bit AXI4 master interfaces for bidirectional global-memory transfers. Read (Rd) and Write (Wr) FIFOs stream data between global and on-chip memory; Transmit (Tx) and Receive (Rx) FIFOs stream to and from the CMAC subsystem. URAM and BRAM are organized into banks for on-chip memory; all URAM banks are single-port (URAM has no dual-port mode) while BRAM banks are dual-port. The CMAC core runs an internal $322$ MHz clock and a $256$/$512$-bit kernel interface; FAB implements a $512$-bit interface, giving about $153$ Gbps at $300$ MHz (faster than the Ethernet IP), versus about $76$ Gbps for a $256$-bit interface (which would dictate the transfer time). With the $512$-bit interface, transmitting a single ciphertext limb (a $0.4$ MB polynomial) takes about $11{,}399$ clock cycles, and the entire ciphertext about $546{,}980$ cycles.

FAB microarchitecture: the functional unit and modular arithmetic

Every FHE operation reduces to integer modular addition and modular multiplication, so each of the $256$ functional units contains a modular multiplier, modular adder, modular subtractor, and an automorph unit. Multi-word arithmetic reduces $54$-bit operations to $27$-bit operations for addition and $18$-bit operations for multiplication, matching the DSP blocks, and FAB issues multiple scalar operations per cycle through fine-grained pipelining.

Modular add and subtract. FAB follows Hankerson et al. Algorithms 2.7 (add) and 2.8 (subtract). Both algorithms' correction step (line 2) would otherwise need a full $54$-bit operation; FAB rewrites that correction step into multiple $27$-bit operations. With multi-word arithmetic and full DSP pipeline registers, both run in $7$ clock cycles.

Modular multiply. Done as an integer multiply followed by a modular reduction, pipelined across the two operations. The integer multiply follows the operand-scanning (schoolbook) multi-word algorithm, Hankerson et al. Algorithm 2.9. With $54$-bit operands split into three $18$-bit operands, a naive version needs $21$ cycles per multiply; since roughly $50\%$ of FHE operations are integer multiplies, FAB loop-unrolls and parallelizes the algorithm to cut this to $12$ cycles while keeping all DSP pipeline registers.

Modular reduction. FAB proposes a hardware-friendly fast reduction by modifying Will and Ko's shift-and-add technique, which avoids the expensive multiplications of Barrett reduction and uses only shift and add operations. Will and Ko's method shifts one bit at a time and takes $2 \log q$ cycles ($108$ cycles for $\log q = 54$). FAB's Algorithm 1 performs multiple bit shifts at once ($\mathrm{shifts} = 6$), reducing this to $12$ cycles for $\log q = 54$. It precomputes a $63$-element array $\mathrm{madd}$ of $\log q$-bit elements; reduction is needed against $32$ prime moduli, so $32$ such $\mathrm{madd}$ arrays are precomputed offline (about $7$ KB total, no runtime overhead). The algorithm is generic in the number of shifts; the full listing is given in the Algorithms appendix.

Automorph. The automorph unit performs the permutation behind Rotate: it reads a polynomial from on-chip memory and writes it to the register file in permuted order for rotation index $k$. The slot at index $i$ maps to a new slot via the automorphism in Equation 5:

$$\mathrm{new\_index}_k(i) = \frac{5^k - 1}{2} + 5 \cdot i \pmod{N}$$

Because bootstrapping uses only about $60$ distinct rotation indices, the powers of $5$ are precomputed for each $k$. The division by two is a bit-shift, and since $N$ is a power of two, reduction $\bmod\, N$ is just an AND with $N - 1$. Bit-reversal for NTT/iNTT is folded into this automorph step (or the multiply) just before the transform, so its latency is not counted separately.

On-chip memory and register file

The U280 has $962$ URAM blocks ($288$ Kb each, single-port) and $4032$ BRAM blocks ($18$ Kb each, single- or dual-port), all single-cycle access. FAB combines them into banks giving $43$ MB total capacity at about $30$ TB/s internal SRAM bandwidth. Figure 4 shows the layout.

URAM (Figure 4a). Each URAM block is $72$ bits wide, depth $4096$. Three URAMs combine to $216$ bits, holding four $54$-bit coefficients per address ($216 / 4 = 54$). Laying out $64$ of these $216$-bit-wide URAMs into one bank stores $256$ coefficients, so one read or write touches $256$ coefficients per cycle, matching the $256$ functional units. A bank is thus $64 \times 3 = 192$ URAMs and holds $16$ polynomials (about $7.08$ MB). FAB uses five URAM banks: c0 bank-1 and bank-2 hold $32$ limbs ($24$ original plus $8$ extension) of the c0 ring element; c1 bank-1 and bank-2 hold $32$ limbs of the c1 ring element; the fifth miscellaneous bank holds $16$ polynomials and stores twiddle factors, KeySwitch keys, and plaintext vectors read from main memory.

BRAM (Figure 4b). Three $18$-bit BRAMs combine to a $54$-bit-wide word, so $256$ BRAM blocks store $256$ coefficients. Each BRAM is depth $1024$; stacking two gives depth $2048$, holding $8$ polynomials per bank. FAB has three BRAM banks: two banks of $1536$ BRAMs each ($8$ polynomials, ideal for extension limbs) and a third miscellaneous bank of $768$ BRAMs ($4$ polynomials) for temporary main-memory data.

Register file. Multiple RFs totaling $2$ MB are spread across the functional, address-generation, and control units, each with multiple same-cycle read and write ports. About one quarter holds host-written precomputed values and parameters; the rest stores up to four intermediate polynomials from Rotate or Mult.

FIFOs. $32$ synchronous Wr and Rd FIFOs (one per HBM-side AXI port) stream between main and on-chip memory, each $256$ bits wide (the AXI port width), built from distributed RAM. Wr FIFO depth is $128$ (matching an HBM burst length of $128$); Rd FIFO depth is $512$ (up to four outstanding reads). The Rd FIFO runs in the $450$ MHz memory clock domain, the Wr FIFO in the $300$ MHz kernel domain. Tx and Rx FIFOs to the CMAC subsystem are $512$-bit synchronous FIFOs.

NTT/iNTT datapath

FAB uses a unified Cooley-Tukey algorithm for both NTT and iNTT, so the same data-mapping logic serves both. The $256$ modular add, subtract, and multiply units operate in parallel as radix-2 butterfly units, processing $512$ coefficients of a polynomial per pass. This runs the $\log N$ stages in roughly $\log N \cdot \frac{N}{512}$ cycles instead of $\log N \cdot \frac{N}{2}$ cycles. The NTT address-generation unit uniquely maps data within each stage via a data-mapping sub-unit, and a twiddle-factor mapping sub-unit reads the needed twiddle factors for a stage from the URAM miscellaneous bank. Both sub-units use data and stage counters to generate addresses on the fly with cheap shift and AND operations. The latency of bit-reversal is not counted separately because bit-reversal is folded into the automorph or multiplication step performed just before NTT/iNTT.

KeySwitch datapath: modified dataflow and smart scheduling

KeySwitch comprises four sub-operations: Decomp, ModUp, KSKIP, and ModDown. The challenge is that KeySwitch must operate on the extension limbs (factors of $P$) and compute an inner product with KeySwitch keys that are nearly three times the ciphertext size. Decomp splits the limbs of a ring element into $\mathrm{dnum}$ digits; each digit feeds ModUp, which outputs $L + 1 + \alpha$ limbs, so after all ModUp operations there are $\mathrm{dnum} \cdot (L + 1 + \alpha)$ limbs. With $\mathrm{dnum} = 3$ and $\alpha = 8$, that is $3 \cdot 32 = 96$ limbs. The total live data is about $112$ MB ($84$ MB keys plus $28$ MB ciphertext), which must be managed within $43$ MB of on-chip memory without writing any resultant limb back to main memory.

Modified datapath (Figure 5). The original datapath (Figure 5a) runs Decomp, then the full ModUp, then the full KSKIP, then ModDown in sequence; because the extended polynomials do not fit on chip, it reads input limbs in evaluation form for Decomp, writes them out in coefficient form after ModUp, then reads them back for KSKIP, incurring heavy DRAM traffic. FAB instead splits KSKIP into two steps and greedily makes progress on the inner product as soon as operands are on chip. The KSKIP step is an inner product between the input polynomial and the switching key in the raised basis. The key observation is that the $\alpha$ limbs of each ModUp operation do not change, so KSKIP can begin on them right after Decomp. Decomp takes the $L$ limbs and splits them into $\beta \le \mathrm{dnum}$ blocks of $\alpha$ limbs each; these $\alpha$ limbs follow two paths: (1) they begin KSKIP, with the intermediate sum written to URAM, and (2) they feed the basis-conversion step to generate the extension limbs, which then complete KSKIP. This both reduces the number of NTT computations (the most expensive KeySwitch subroutine) and alleviates the memory-bandwidth bottleneck by reducing memory traffic.

Smart operation scheduling. The RNS extension equation (Equation 1) naively needs $2\ell$ multiplications per generated limb $p_j$. FAB observes that one factor, $x_i \cdot \tilde{Q}_i \pmod{q_i}$, is independent of the output modulus, so the $\ell$ inner products of form $x_i \cdot \tilde{Q}_i$ are computed once and reused for every extension limb $p_j$, halving the modular multiplications. Scheduling also generates an entire block of extension limbs at once for a given block of $\alpha$ original limbs; as soon as a block's extension limbs are ready, an NTT followed by KSKIP runs on them before the next block is generated, repeating $\mathrm{dnum}$ times to produce all $\mathrm{dnum} \cdot (L + 1 + \alpha)$ limbs. Only the key block matching the just-extended $\alpha$ block is fetched (not all keys at once). All extension limbs are kept in the c0 BRAM bank, whose dual-port BRAMs allow simultaneous read and write for limb-wise inner products in BasisConvert; this lets the whole KSKIP run using a $4$-limb BRAM miscellaneous bank while hiding the roughly $300$-cycle key-read latency behind computation. ModDown (step 5) is scheduled like ModUp and repeated for the c1 polynomial using the URAM c1 bank. The net effect is high data reuse, exploitation of limb-wise parallelism, uniform address generation (no switching between limb-wise and slot-wise access), and no writing or reading of resultant ciphertext limbs to main memory. The split does not change the underlying KeySwitch algorithm or its noise; only the order of steps changes. The optimization is specific to the U280 but portable to smaller FPGAs as long as at least one key limb and one ciphertext polynomial fit on chip (otherwise BasisConvert would need to go fully slot-wise to stay compute-bound).

Evaluation setup and resource utilization

FAB is written in Verilog 2001, synthesized with Xilinx Vivado 2020.2 at $300$ MHz, packaged as kernel code with Vitis 2020.2 into an .xclbin, and run on an Alveo U280 (Xilinx 16nm UltraScale, $8$ GB HBM2, up to $460$ GB/s). Resource utilization (Table 3): $899{,}232$ LUTs ($68.96\%$ of $1{,}304$K, of which functional units are the largest share at about $37\%$), about $2{,}073$K flip-flops ($79.54\%$ of $2{,}607$K, mostly the distributed register file, control logic, and functional units), $5{,}120$ DSPs ($56.70\%$ of $9{,}024$, entirely for modular arithmetic), $3{,}840$ BRAMs ($95.24\%$ of $4{,}032$), and $960$ URAMs ($99.80\%$ of $962$).

Resource comparison against the ASIC proposals (Table 4) highlights FAB's frugality: F1 ($N = 2^{14}$, $\log q = 32$) uses $18432$ modular multipliers, an $8$ MB register file, and $64$ MB on-chip memory; BTS ($N = 2^{17}$, $\log q = 50$) uses $8192$ modular multipliers, a $22$ MB register file, and $512$ MB on-chip memory; FAB ($N = 2^{16}$, $\log q = 54$) uses just $256$ modular multipliers, a $2$ MB register file, and $43$ MB on-chip memory. Versus BTS, FAB uses $32\times$ fewer modular multipliers, an $11\times$ smaller register file, and a $12\times$ smaller on-chip memory.

Results: basic operations, bootstrapping, and LR training

Basic CKKS operations vs GPU (Table 5). Against the optimized GPU implementation of Jung et al. (closest parameter set $N = 2^{16}$, $\log Q = 1693$, $100$-bit security), FAB execution times (ms) and speedups are: Add $0.04$ vs $0.16$ ($3.85\times$); Mult $1.71$ vs $2.96$ ($1.73\times$); Rescale $0.19$ vs $0.49$ ($2.62\times$); Rotate $1.57$ vs $2.55$ ($1.62\times$). Average speedup is about $2.4\times$ in absolute time.

Throughput vs HEAX, FPGA (Table 6). At the matching HEAX parameters ($N = 2^{14}$, $\log Q = 438$), FAB throughput (operations per second): NTT $167$K vs $42$K ($3.97\times$); Mult $5.7$K vs $2.6$K ($2.12\times$). Average is about $3\times$ higher throughput than HEAX, due to low-latency modular arithmetic, fine-grained pipelining, the optimized NTT datapath, and the modified KeySwitch datapath.

Bootstrapping, amortized per-slot multiply time $T_{\mathrm{Mult},a/\mathrm{slot}}$ (Equation 3, Table 7). FAB runs at $0.3$ GHz with $2^{15}$ slots at $0.477\ \mu s$. Comparisons: Lattigo (CPU, $3.5$ GHz, $2^{15}$ slots) $101.78\ \mu s$, FAB $213\times$ faster in time and $2485\times$ in cycles; GPU-1 ($1.2$ GHz, $97$-bit, $2^{15}$ slots) $0.740\ \mu s$, FAB $1.55\times$ (time) / $6.35\times$ (cycles); GPU-2 ($1.2$ GHz, $173$-bit, $2^{16}$ slots) $0.716\ \mu s$, FAB $1.50\times$ (time) / $6.14\times$ (cycles); F1 ($1$ GHz, $1$ slot) $254.46\ \mu s$, FAB $533\times$ (time) / $1775\times$ (cycles); BTS-2 ($1.2$ GHz, $2^{16}$ slots) $0.0455\ \mu s$, where FAB is $0.09\times$ (time) / $0.38\times$ (cycles), i.e. FAB is about $9\times$ slower in absolute time and $4\times$ slower in cycles than BTS's best case, attributed to BTS's far larger on-chip memory and multiplier count. The authors estimate that if FAB were scaled to $8192$ modular multipliers and $512$ MB on-chip memory it would be at least $3\times$ faster than BTS, owing to its algorithmic and microarchitectural optimizations.

Logistic regression training (Table 8). The target application trains a binary LR classifier on the MNIST 3-vs-8 subset ($11{,}982$ samples, $196$ features), the same HELR task used by all baselines: $30$ iterations, $1024$ encrypted images per mini-batch, evaluation depth $150$, requiring a bootstrap after every iteration. Training uses sparsely-packed ciphertexts ($256$ slots) for fair comparison with GPU-2 and BTS-2. FAB-1 is a single-FPGA design (about $6.65$ GB of ciphertexts and KeySwitch keys offloaded to HBM2). FAB-2 scales to eight FPGAs in pairs (each with a primary and secondary FPGA, plus one master that can broadcast a ciphertext), using direct FPGA-to-FPGA network communication; each FPGA processes $128$ ciphertexts, inter-FPGA communication happens only twice per LR iteration with about $12$ ms overhead per iteration. Average training time per iteration (sec): Lattigo $37.05$; GPU-2 $0.775$; F1 $1.024$; BTS-2 $0.028$; FAB-1 $0.103$; FAB-2 $0.081$. FAB-2 is thus $456\times$ faster than CPU and $9.5\times$ faster than GPU-2 in absolute time (for the baselines' $N = 2^{17}$, $\log Q = 2395$ parameters), $12\times$ faster than F1, and competitive with BTS-2. FAB-2 does not reach an $8\times$ speedup over FAB-1 because parallelism is limited by the single-FPGA bootstrapping runtime (Amdahl's law); only the non-bootstrapping operations within an iteration are parallelized.

Versus leveled FHE. The leveled alternative (send exhausted ciphertexts back to the client to decrypt and re-encrypt) has two drawbacks. First, it leaks intermediate values to the client; defending against this needs a $\lambda$-bit mask (minimum justifiable $\lambda = 40$), and because the CKKS plaintext space is not finite this requires extra plaintext precision and substantially larger parameters, increasing LR time. (By contrast BGV/BFV have finite plaintext spaces, so a uniform mask hides the message without extra precision.) Second, even ignoring leakage, FAB still wins: one LR iteration consumes $5$ compute levels and a bootstrap, and FAB does it in $0.103$ sec including bootstrapping, whereas just the client-side re-encryption (SEAL subroutines, $2.8$ GHz CPU) already costs $0.162$ sec, before adding cloud computation and client-cloud communication.

Related work and conclusion

Related work. CPU libraries (SEAL, HElib, PALISADE, Lattigo, HEAAN) remain impractical for full FHE; several GPU works accelerate only NTT, and Jung et al. give the state-of-the-art full GPU CKKS (including bootstrapping). On ASICs, F1 (Samardzic et al.) supports multiple schemes (BGV, CKKS, GSW) but only small parameter sets, making it incompatible with multi-slot bootstrapping and real-time ML; BTS (Kim et al.) targets CKKS bootstrapping with many processing elements, large on-chip memory and a large register file, but does not modify the algorithm to optimize for memory bandwidth, leaving performance unrealized relative to its resources. On FPGAs, HEAX accelerates only CKKS encrypted multiplication (other operations go to the host) and supports only small parameter sets (depth up to $10$), insufficient for bootstrapping or LR training. FAB is the first to implement full-scale CKKS bootstrapping on an FPGA at secure parameters supporting an end-to-end application.

Applicability to other schemes. FAB's basic Add, Mult, and Rotate are common across schemes and reusable for BGV and B/FV. The KeySwitch-datapath and smart-scheduling ideas are also relevant to TFHE and FHEW (which evaluate Boolean gates and carry several-GB key footprints), though the exact KeySwitch steps differ per scheme and need fresh analysis to derive a balanced design.

Conclusion. FAB combines algorithmic and architectural optimizations to overcome the memory-bandwidth bottleneck and perform the first fully-packed CKKS bootstrapping on an FPGA at a practical, secure parameter set. Through smart operation scheduling and on-chip memory management it delivers $456\times$ and $6.5\times$ better LR-training performance than CPU and GPU respectively, reaches performance comparable to costly ASIC designs while using only commercially available hardware, and is immediately deployable in public cloud environments (e.g. AWS F1).

Algorithms

The paper presents one formal numbered algorithm (Algorithm 1, modular reduction) and refers to several others by external reference (Hankerson et al. Algorithms 2.7 / 2.8 / 2.9 for modular add, subtract, and operand-scanning multiply) and to the KeySwitch sub-operation flow described in prose and Figure 5. The self-contained listings below reproduce Algorithm 1 verbatim and reconstruct the externally-referenced and prose-only procedures at a glance.

Algorithm 1: Modular reduction in $\mathbb{F}_q$ (FAB's multi-shift variant).

Algorithm 1  Modular Reduction in F_q
Input:   modulus q, integer a (a is 2*log q - 1 bits wide), shifts = 6
Output:  c = a mod q

1:  Precompute (offline): for i = 1 to 2^shifts - 1
        madd[i-1] = sum_{j=0..5} i[j] * 2^(log q + j)  (mod q)
        > madd has 2^shifts - 1 = 63 elements, each (log q)-bit wide
        > one such madd array per prime modulus (32 arrays, ~7 KB total)
2:  Set (A[1], A[0]) <- a          > split a into high half A[1], low half A[0]
3:  count <- 0;  as1 <- 0
4:  while count < log q do
5:      (carry, as1) <- A[1] >> shifts     > carry is the top 'shifts' bits
6:      A[1] <- as1 + madd[carry - 1]
7:      count <- count + shifts
8:  end while
9:  c <- A[1] + A[0]
10: if c > q then
11:     c <- c - q
12: end if
13: return c

Algorithm A: Modular addition / subtraction (multi-word, per Hankerson et al. 2.7 / 2.8). FAB splits the $54$-bit operands into two $27$-bit words to use the DSP preadders, and rewrites the correction step (line 2) as multiple $27$-bit operations; latency $7$ cycles.

ModAdd(x, y, q):                       ModSub(x, y, q):
Input:  x, y in [0, q), modulus q      Input:  x, y in [0, q), modulus q
Output: (x + y) mod q                  Output: (x - y) mod q

1: s <- x + y    (multi-word, 27-bit)  1: d <- x - y    (multi-word, 27-bit)
2: if s >= q then                      2: if d < 0 then          > borrow
3:     s <- s - q  (split into 27-bit)  3:     d <- d + q  (split into 27-bit)
4: end if                             4: end if
5: return s                           5: return d

Algorithm B: Integer multiply, operand-scanning / schoolbook (per Hankerson et al. 2.9), feeding modular reduction. $54$-bit operands are split into three $18$-bit words $a_0,a_1,a_2$ and $b_0,b_1,b_2$; FAB loop-unrolls and parallelizes the partial products to reach $12$ cycles (vs $21$ naive). The product is then reduced with Algorithm 1 to complete a modular multiply.

ModMult(x, y, q):
Input:  x = (a2,a1,a0), y = (b2,b1,b0)   > each ai,bi is 18 bits, x,y < q
Output: (x * y) mod q

1: for i = 0 to 2:                       > operand scanning
2:     carry <- 0
3:     for j = 0 to 2:
4:         t <- P[i+j] + a_i * b_j + carry   > 18x18-bit DSP multiply
5:         P[i+j] <- t mod 2^18
6:         carry  <- t div 2^18
7:     end for
8:     P[i+3] <- P[i+3] + carry
9: end for
        > steps 1-9 unrolled/parallelized -> 12 cycles
10: prod <- concatenate(P[..])           > full ~108-bit product
11: return ModReduce(prod, q)            > Algorithm 1

Algorithm C: Automorph (Rotate permutation).

Automorph(poly, k, N):
Input:  polynomial poly (N coefficients), rotation index k, ring dim N (power of 2)
Output: permuted polynomial written to the register file

1: base <- (5^k - 1) / 2     > 5^k precomputed for each of ~60 indices; /2 is a shift
2: for i = 0 to N - 1:
3:     new_index <- (base + 5 * i) AND (N - 1)    > mod N via AND with N-1
4:     RF[new_index] <- poly[i]
5: end for
6: return RF

Algorithm D: KeySwitch with FAB's modified datapath and smart scheduling. Reconstructed from the prose and Figure 5; the algebra (Decomp, ModUp = BasisConvert via Equation 1, KSKIP inner product, ModDown) is unchanged from prior work, only the ordering and on-chip scheduling differ.

KeySwitch(ct = (a_m, b_m), ksk, parameters dnum, alpha, L):
Input:  ciphertext ring elements, switching key ksk, RNS parameters
Output: key-switched ciphertext (no resultant limbs written back to HBM)

1: precompute t_i = x_i * Qtilde_i (mod q_i) for the L input limbs   > reused per output limb
2: Decomp: split the L limbs of a_m into beta <= dnum blocks of alpha limbs each
3: for each block B of alpha limbs (repeat dnum times):
4:     // path 1: greedy KSKIP on the alpha (unchanged) limbs
5:     begin KSKIP inner product on B with matching ksk block; write partial sum to URAM
6:     // path 2: generate this block's extension limbs
7:     BasisConvert(B): use Equation 1 with reused t_i to produce the extension limbs
            into the dual-port c0 BRAM bank
8:     NTT the freshly generated extension limbs
9:     complete KSKIP on the extension limbs; accumulate into the URAM partial sum
10:    fetch only the ksk block matching B (hide ~300-cycle latency behind compute)
11: end for
12: ModDown: rescale result back to modulus Q (scheduled like ModUp);
        repeat for the c1 polynomial using the URAM c1 bank
13: return key-switched ciphertext

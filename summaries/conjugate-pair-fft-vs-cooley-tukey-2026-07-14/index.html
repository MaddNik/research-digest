The Conjugate-Pair FFT, and how it compares to Cooley-Tukey

Research Tracker - background note

Why the FFT matters here

Almost every fast (T)FHE kernel reduces to multiplying polynomials in a ring such as $\mathbb{Z}_q[X]/(X^N+1)$. Done naively, multiplying two degree $N$ polynomials costs $O(N^2)$ coefficient products. A Fast Fourier Transform (FFT) turns that convolution into $O(N\log N)$ work: transform both operands, multiply pointwise, transform back. The Discrete Fourier Transform (DFT) it accelerates is

$$X[k]=\sum_{n=0}^{N-1} x[n]\,\omega_N^{nk},\qquad \omega_N=e^{-2\pi i/N},$$

shown in Equation 1, where $\omega_N$ is a primitive $N$-th root of unity. Evaluating Equation 1 directly is $O(N^2)$; the FFT family computes exactly the same result in $O(N\log N)$ by recursively reusing shared sub-sums. This note compares the classic radix-2 Cooley-Tukey FFT with the conjugate-pair FFT, and explains why the latter is attractive for FPGA and ASIC accelerators. For where transforms sit inside TFHE itself (NTT, negacyclic convolution, gadget decomposition), see the companion note "Transforms and Polynomial Multiplication for FHE".

Radix-2 Cooley-Tukey: the baseline

The Cooley-Tukey algorithm (1965) splits a length $N$ DFT (with $N$ a power of two) into the DFT of the even-indexed samples and the DFT of the odd-indexed samples, each of length $N/2$. Writing $E[k]$ for the even sub-DFT and $O[k]$ for the odd one, the two output halves recombine through a single "butterfly":

$$X[k]=E[k]+\omega_N^{k}\,O[k],\qquad X[k+N/2]=E[k]-\omega_N^{k}\,O[k],$$

as in Equation 2, for $0\le k \lt N/2$. The factor $\omega_N^{k}$ is the twiddle factor. Recursing until length 1 gives $\log_2 N$ stages of $N/2$ butterflies each, so the cost is on the order of $N\log_2 N$ operations, and in optimized real-arithmetic terms roughly $5N\log_2 N$ real additions and multiplications. This is the textbook FFT and maps to a very regular butterfly datapath, which is why it is so common in hardware.

A decimation-in-time radix-2 pass, in place, looks like this:

radix2_dit(x[0..N-1]):
    bit_reverse_permute(x)
    for s in 1 .. log2(N):              # stage
        m = 2**s
        w_m = exp(-2*pi*i / m)          # stage root
        for j in 0 .. N-1 step m:       # each length-m block
            w = 1
            for r in 0 .. m/2 - 1:      # butterflies
                u = x[j + r]
                t = w * x[j + r + m/2]
                x[j + r]        = u + t
                x[j + r + m/2]  = u - t
                w = w * w_m
    return x

Each inner iteration loads a fresh twiddle $w$; over a full transform the algorithm touches $\tfrac{N}{2}\log_2 N$ twiddle values, and the distinct twiddles per stage double every stage. Twiddle generation and twiddle memory bandwidth, not the adds, are often what limit a hardware FFT.

Split-radix: fewer operations

Split-radix (Yavne 1968; Duhamel and Hollmann 1984) improves on radix-2 by treating the even and odd parts asymmetrically. It splits the length $N$ DFT into one half-length DFT over the even indices, plus two quarter-length DFTs over the indices $n\equiv 1$ and $n\equiv 3 \pmod 4$. With $U[k]$ the length $N/2$ even sub-DFT and $Z_1[k],Z_2[k]$ the two length $N/4$ odd sub-DFTs, the recombination is

$$X[k]=U[k]+\big(\omega_N^{k}\,Z_1[k]+\omega_N^{3k}\,Z_2[k]\big),$$

given in Equation 3. Because the two odd sub-transforms are only a quarter length, split-radix performs about $4N\log_2 N$ real operations, versus roughly $5N\log_2 N$ for radix-2 Cooley-Tukey. For power-of-two sizes it held the record for the lowest exact operation count for decades. The price is a slightly less uniform recursion (an $L$-shaped butterfly mixing one half-length and two quarter-length results), and, awkwardly for hardware, two different twiddle factors per butterfly: $\omega_N^{k}$ and $\omega_N^{3k}$.

The conjugate-pair FFT

The conjugate-pair FFT (Kamar and Elcherif 1989; popularized by Bernstein and by the "tangent FFT" line of work) is a re-indexing of split-radix that removes the awkward $\omega_N^{3k}$ twiddle. It cyclically shifts the last quarter-length subsequence by four positions before transforming it. Call that shifted transform $Z_2'[k]$. The shift converts the troublesome $\omega_N^{3k}$ into $\omega_N^{-k}$, so the recombination becomes

$$X[k]=U[k]+\omega_N^{k}\,Z_1[k]+\omega_N^{-k}\,Z_2'[k],$$

shown in Equation 4. The key observation is that the two remaining twiddle factors are complex conjugates of each other,

$$\omega_N^{-k}=\overline{\omega_N^{k}},$$

as in Equation 5. Instead of two independent twiddles per butterfly, there is now a single twiddle $\omega_N^{k}$ and its conjugate, which comes for free by negating the imaginary part. The arithmetic operation count is identical to ordinary split-radix (about $4N\log_2 N$ real operations), so the conjugate-pair form is not cheaper in adds and multiplies. Its advantage is structural:

  • Only one twiddle table indexed by $k$ is needed; the second factor is that table's conjugate. This roughly halves twiddle ROM size and, more importantly, twiddle load bandwidth.
  • Both quarter-length sub-transforms read the same twiddle address, giving a regular, conflict-free memory access pattern that suits banked on-chip RAM and SIMD lanes.
  • It is the natural starting point for the "tangent FFT" (Van Buskirk; Bernstein 2007), which rescales these conjugate twiddles to push the operation count slightly below $4N\log_2 N$.

A decimation-in-time conjugate-pair recursion, sketched:

conjpair_fft(x[0..N-1]):
    if N == 1: return x
    U  = conjpair_fft( x[0], x[2], x[4], ... )        # even indices,  length N/2
    Z1 = conjpair_fft( x[1], x[5], x[9], ... )        # n = 1 mod 4,   length N/4
    Z2 = conjpair_fft( x[N-1], x[3], x[7], ... )      # n = 3 mod 4, shifted by -4 -> Z2'
    for k in 0 .. N/4 - 1:
        w  = exp(-2*pi*i*k / N)                        # single twiddle
        a  = w * Z1[k]
        b  = conj(w) * Z2[k]                           # conjugate twiddle, no extra table
        t1 = a + b
        t2 = i * (b - a)                               # rotate by -90 degrees
        X[k]         = U[k]        + t1
        X[k + N/2]   = U[k]        - t1
        X[k + N/4]   = U[k + N/4]  + t2
        X[k + 3*N/4] = U[k + N/4]  - t2
    return X

Note that $Z_2$ is fed the input in shifted order (the $X[N-1]$ first), which is exactly the cyclic $-4$ shift that produces $Z_2'$ and turns $\omega_N^{3k}$ into $\omega_N^{-k}$.

Side-by-side comparison

The three algorithms compute the identical DFT of Equation 1; they differ in operation count, twiddle handling, and how friendly they are to a pipelined datapath.

Property              Radix-2 Cooley-Tukey   Split-radix            Conjugate-pair
-------------------   --------------------   --------------------   --------------------
Real operations       ~ 5 N log2 N           ~ 4 N log2 N           ~ 4 N log2 N
Recursion shape       even / odd (2-way)     even + two odd (L)     even + two odd (L)
Twiddles / butterfly  1  (w^k)               2  (w^k and w^3k)      1 + its conjugate
Distinct twiddle ROM  1 table                up to 2 tables         1 table (conj is free)
Memory access         very regular           two twiddle streams    one twiddle stream
Op count record?      no                     long-standing record   ties split-radix
Extends to            -                      -                      tangent FFT (below 4N log2 N)

In short: radix-2 is the simplest and most regular but does the most arithmetic; split-radix does the least arithmetic but wants two twiddle streams; the conjugate-pair FFT keeps split-radix's low operation count while collapsing back to a single twiddle stream, trading a slightly more intricate index schedule for smaller, higher-bandwidth-friendly twiddle storage.

Why this matters for FHE hardware

TFHE and related schemes work in the negacyclic ring $\mathbb{Z}_q[X]/(X^N+1)$. One fast route for that product is a complex FFT with negacyclic pre- and post-weighting: fold the sign-flip of the wrap-around $X^N=-1$ into a weight $\psi$ with $\psi^2=\omega_N$,

$$\hat{x}[n]=x[n]\,\psi^{n},\qquad \psi=e^{-\pi i/N},$$

as in Equation 6, transform with an ordinary length $N$ complex FFT, multiply pointwise, and unweight after the inverse transform. Libraries that take this complex-FFT route (for example the FFT backend used by Concrete and other TFHE stacks) lean on split-radix and conjugate-pair kernels precisely because floating-point FFT throughput dominates gate-bootstrapping time.

On an FPGA or ASIC the bottleneck is rarely the multipliers; it is feeding them. Twiddle ROM capacity, twiddle read ports, and memory-bank conflicts set the achievable clock and utilization. The conjugate-pair FFT is appealing there because it delivers the split-radix operation count from a single twiddle table whose conjugate is obtained by negating one sign bit, and because both quarter-length sub-transforms hit the same twiddle address, which keeps banked RAM access regular. The same "spend an index trick to save memory traffic" instinct shows up in the NTT dataflows and systolic layouts discussed in the companion transforms note; conjugate-pair FFT is the complex-FFT counterpart of that idea.

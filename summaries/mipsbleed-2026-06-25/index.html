MIPSBLEED: Uncovering Microarchitectural Timing Leaks in Pervasive Embedded Processors

arXiv · June 2026 · full view ↗

Introduction

Microarchitectural side-channel attacks have become one of the most consequential classes of hardware security vulnerability, exposing cryptographic secrets through physical phenomena such as cache access timing, execution latency, and resource contention. While the research community has extensively characterized these channels on x86 (Intel, AMD) and, more recently, ARM, the MIPS architecture has received comparatively little scrutiny despite being embedded in an estimated 8.5 billion shipped chips as of 2024, powering routers, industrial controllers, IoT sensors, and network equipment worldwide.

This paper introduces MIPSBLEED, the first comprehensive framework for systematically uncovering microarchitectural timing leaks in SMT-enabled MIPS processors. By instrumenting three distinct channels (L1 data cache, L1 instruction cache, and execution engine), and by quantifying leakage rigorously with the NICV statistical metric, the authors demonstrate that an unprivileged attacker can recover cryptographic secrets from a co-located victim process with no kernel access, no hardware privilege escalation, and no reliance on speculative execution bugs.

Background

The evaluation platform is the MIPS32 1004K, a multi-threaded embedded processor core used in routers and network SoCs. Its L1 caches are 32 KB each, 4-way set-associative with 256 sets and 32-byte lines. The cache stride (distance between same-set, different-way addresses) is $256 \times 32\ \text{B} = 8192\ \text{B} = 8\ \text{KB}$.

SMT on the 1004K shares the L1 data cache, L1 instruction cache, and execution engine (ALU and MDU) between two logical cores. Each logical core maintains its own register file and program counter. This sharing is the root cause of cross-thread timing channels: when the victim thread accesses a shared resource, the attacker thread on the other logical core observes a measurable latency increase.

MIPS provides the RDHWR (Read Hardware Register) instruction for unprivileged access to the cycle counter. This differs significantly from ARM (requires privilege) and x86 (rdtsc available in user space but with serialization caveats). This capability is critical for the attacker: high-resolution timing measurements can be taken from ordinary user-space code without system calls or privilege elevation.

The paper evaluates SECCURE 0.5, a lightweight elliptic curve toolkit distributed in mainstream Linux packages. SECCURE implements scalar multiplication via the classical left-to-right double-and-add algorithm, which has a secret-dependent branch creating timing variation.

Threat Model

The MIPSBLEED threat model assumes an attacker capable of executing unprivileged user-space code on the victim system, with SMT enabled and the attacker and victim co-located on the same physical processor core. The attacker and victim share the L1 data cache, L1 instruction cache, and execution engine. The attacker can invoke rdhwr to read the cycle counter with nanosecond-level precision.

The attacker does NOT require shared memory mappings with the victim, root or kernel privileges, hardware performance counter access via kernel API, knowledge of the victim's virtual address space layout, or any speculative execution exploitation. This threat model is realistic for embedded Linux deployments on routers or industrial controllers where multiple services run on the same CPU with SMT enabled and minimal process isolation.

Methodology

Framework Overview: MIPSBLEED consists of three phases: (1) Probe construction—write assembly loops that exercise one shared microarchitectural resource and measure cycle-accurate time via rdhwr; (2) Leakage quantification—use NICV (Normalized Inter-Class Variance) to determine whether timing measurements statistically correlate with victim behavior; (3) Exploitation—map the timing trace onto the victim's secret-dependent control flow to recover key bits.

NICV Definition: NICV measures the fraction of total timing variance explained by a class label $Z$ (e.g., a secret bit): $\text{NICV} = \frac{\text{Var}_{z \in \mathcal{Z}}[\mathbb{E}[T \mid Z = z]]}{\text{Var}[T]}$. NICV ranges in $[0, 1]$; high NICV indicates strong, exploitable leakage.

L1 Data Cache Probe: The attacker issues loads to four cache lines per set at 8 KB intervals, covering all 4 ways of each of the 256 sets. Multiplication instructions form a dependency chain to prevent out-of-order execution from collapsing timing measurements. When the victim accesses a cache set, it displaces the attacker's probe lines via LRU eviction, creating measurable latency spikes.

L1 Instruction Cache Probe: The attacker places jump targets at 8 KB-aligned code addresses and measures the latency of instruction fetches. If the victim's instruction stream causes fetches from cache set $s_v$, those fetches evict the attacker's probe instructions from that set, creating the same horizontal-band timing signature as the D-cache probe.

Execution Engine Probe: Two variants: (1) ALU variant—execute 25 iterations of dependent add/subtract pairs to saturate the shared ALU; (2) MDU variant—execute dependent multiply instructions to saturate the Multiply/Divide Unit. Victim instruction execution creates contention, increasing measured latency.

Experimental Results

L1 Data Cache Leakage: Running the D-cache probe concurrently with a victim that accesses a specific cache set (set 50 in the paper's examples) reveals a persistent horizontal band of elevated latency at that set index. This pattern is reproducible across many probe passes and confirmed by high NICV values, indicating significant exploitable information leakage through the data cache channel.

L1 Instruction Cache Leakage: The I-cache probe exhibits an analogous pattern. Critically, it can distinguish between two code paths: the double-only path (executed when scalar bit is 0) and the double+add path (executed when scalar bit is 1). These paths touch different I-cache sets, creating a binary timing signal that directly encodes the key bit stream. NICV values confirm high leakage comparable to the D-cache.

Execution Engine Leakage: Both ALU and MDU contention probes show very high NICV values, with the MDU probe exhibiting the strongest leakage signal of all three channels. Shared functional unit contention creates highly visible latency spikes when the victim issues arithmetic instructions.

Summary: All three channels exhibit high, exploitable NICV. The execution engine (particularly MDU) shows the strongest signal. Combined, the three channels provide complementary views into victim execution and memory access patterns.

End-to-End Attack: SECCURE Key Recovery

Target: SECCURE 0.5 implements P-256 scalar multiplication using the classical left-to-right double-and-add algorithm. The vulnerable function is pointmul(), which iterates over 256 scalar bits: it unconditionally calls point_double() for each bit, then conditionally calls point_add() only when the current bit is 1.

Vulnerability: The conditional branch is secret-dependent. point_double() and point_add() are separate code paths that fetch different I-cache lines. An attacker monitoring the I-cache observes: (1) When bit = 0: only point_double instructions are fetched; fewer I-cache sets disturbed; lower probe latency. (2) When bit = 1: both point_double and point_add instructions are fetched; more I-cache sets disturbed; higher probe latency. This is a timing-based variant of Simple Power Analysis (SPA).

Key Generation: For visual demonstration, the authors modified SECCURE to generate a deterministic scalar with alternating 8-bit blocks: 11111111 00000000 11111111... The authors explicitly note this structured key is for visual clarity only; the attack works equally on random keys.

Attack Execution: (1) Attacker pinned to logical core 0; SECCURE key generation on logical core 1. (2) Attacker runs I-cache probe continuously, recording per-set cycle counts. (3) Single trace (one pointmul execution) is sufficient to recover all key bits. (4) Probe trace segmented into 256 time windows matching scalar multiplication loop iterations. (5) Each window's probe latency classified as high (bit=1) or low (bit=0).

Results: The partial trace clearly shows eight consecutive high-latency windows (bits 1-8: 11111111) followed by eight low-latency windows (bits 9-16: 00000000), matching the demo key's design. The binary pattern is directly readable from a single trace without statistical averaging. Recovery of the 256-bit P-256 private scalar enables full ECDH session decryption or signature forgery.

Mitigations and Defenses

Software-Level Countermeasures:

Constant-Time Scalar Multiplication: The fundamental vulnerability is the secret-dependent branch in pointmul(). The standard fix is the Montgomery ladder formulation, which performs the same operations regardless of scalar bit value using a branchless conditional swap (cswap) primitive. A constant-time implementation causes both I-cache and D-cache channels to observe identical patterns regardless of key bits, eliminating the leakage.

Timing Noise Injection: Adding artificial random delays degrades SNR. However, since MIPSBLEED's I-cache attack is single-trace (not averaging-based), noise must be large enough to mask the signal in a single observation, which is difficult without unacceptable performance degradation.

Hardware-Level Countermeasures:

Cache Partitioning: Assigning disjoint cache set ranges to different SMT threads via cache coloring would prevent cross-thread contention. However, most MIPS embedded platforms lack hardware support (e.g., Intel CAT), and software-based cache coloring is complex in embedded Linux.

SMT Disabling: Disabling SMT eliminates the shared-resource attack surface entirely but imposes a performance penalty.

Privilege-Gating the Cycle Counter: Requiring kernel-level privilege to access the cycle counter (as ARM does by default) would eliminate the attacker's ability to make high-resolution timing measurements from user space. This significantly raises the attacker's privilege requirement without disabling SMT or degrading non-attacker performance.

Lightweight Isolation in Embedded SMT: The authors call for future research into lightweight isolation mechanisms tailored to resource-constrained embedded systems, unlike server-class x86 with many hardware security features.

Limitations

The paper acknowledges several important limitations: (1) All experiments are on the MIPS32 1004K; generalization to other MIPS cores is unvalidated. (2) The attack requires attacker-victim co-location on the same physical core; multi-core systems require co-location assurance. (3) The SECCURE demo uses a structured (non-random) key for visual clarity; success rate on random keys is not quantified. (4) Only SECCURE is evaluated as an end-to-end target; other implementations may have different structures or mitigations. (5) No remote timing attack scenarios are explored. (6) Proposed hardware mitigations are not empirically evaluated.

Conclusion

MIPSBLEED establishes MIPS as a critical and underexplored target in microarchitectural security. By systematically characterizing three independent timing channels (L1 data cache, L1 instruction cache, execution engine) on the MIPS32 1004K, the paper demonstrates that SMT-enabled MIPS platforms are vulnerable to cross-thread side-channel attacks without special privileges. The NICV-based leakage quantification confirms high exploitable leakage on all three channels, with the execution engine exhibiting the strongest signal.

The end-to-end SECCURE key recovery attack demonstrates this leakage is not merely theoretical: an attacker can recover a P-256 private key bit-by-bit from a single scalar multiplication trace using only user-space code and the rdhwr cycle counter instruction. The 8.5 billion deployed MIPS chips include long-lifecycle embedded devices unlikely to receive patches. Near-term practical mitigations are software-level (constant-time cryptographic implementations, SMT disabling for security-critical workloads). The paper calls for the research community to design lightweight, low-cost isolation primitives for the embedded MIPS ecosystem.

Algorithms

Algorithm 1: L1 Data Cache Probe
An assembly loop that reads four cache lines per set at 8 KB intervals, covering all 4 ways of each of the 256 cache sets. Cycle counts measured via rdhwr before and after each per-set block. A multiplication dependency chain prevents out-of-order execution. Output: a 256-element timing vector per probe invocation.

Algorithm 2: L1 Instruction Cache Probe
A code traversal loop that jumps through four 8 KB-aligned code blocks per cache set, executing from addresses that map to all 4 ways of each set. Timing measured via rdhwr. Output: a 256-element timing vector per probe invocation.

Algorithm 3: ALU Contention Probe
Executes 25 iterations of dependent add/subtract instruction pairs to saturate the shared ALU. Timing measured via rdhwr. Output: a scalar cycle count per probe invocation.

Algorithm 4: MDU Contention Probe
Executes a sequence of dependent multiply instructions to saturate the shared Multiply/Divide Unit. Timing measured via rdhwr. Output: a scalar cycle count per probe invocation.

Algorithm 5: NICV Computation
Given $n$ timing traces labeled by secret variable $Z \in \mathcal{Z}$, compute $\text{NICV} = \text{Var}_{z}[\mathbb{E}[T \mid Z=z]] / \text{Var}[T]$ at each sample point. Identifies leakage hotspots in the trace.

Algorithm 6: SECCURE Scalar Multiplication (Vulnerable Double-and-Add)
The standard pointmul function in SECCURE 0.5. Iterates over 256 scalar bits; unconditionally calls point_double(); conditionally calls point_add() only when the current bit is 1. The conditional branch creates the I-cache timing leakage.

Algorithm 7: Montgomery Ladder (Constant-Time Countermeasure)
A secret-independent scalar multiplication that performs the same sequence of point operations regardless of key bits, using a conditional swap (cswap) primitive to avoid data-dependent branches.

Algorithm 8: End-to-End SECCURE Key Recovery Attack
(1) Pin attacker to logical core 0, victim to logical core 1. (2) Victim runs SECCURE P-256 key generation. (3) Attacker runs I-cache probe loop, collecting a continuous timing trace. (4) Segment trace into 256 windows aligned to scalar multiplication iterations. (5) Threshold each window's probe latency to recover the corresponding key bit. (6) Assemble the 256 recovered bits into the full scalar.

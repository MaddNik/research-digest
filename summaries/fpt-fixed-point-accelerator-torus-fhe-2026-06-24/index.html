FPT: a Fixed-Point Accelerator for Torus FHE

IACR Transactions on Cryptographic Hardware and Embedded Systems (TCHES) 2024, Issue 1 / CHES 2024; arXiv:2211.13696 preprint (v3, 18 Oct 2023) · November 2022 · full view ↗

Problem and Motivation

Fully Homomorphic Encryption (FHE) lets a cloud server compute directly on encrypted data, so a client can outsource computation (for example machine-learning inference) without revealing inputs or letting the server learn the result. The obstacle to adoption is speed: FHE can still be on the order of 10,000 times slower than plaintext computation, and the first encrypted AND gate once took up to 30 minutes. TFHE (Torus FHE), a third-generation scheme, brings an encrypted gate down to about 13 ms on a CPU, but it relies on a noise-removal step called bootstrapping after essentially every operation.

In TFHE this step is Programmable Bootstrapping (PBS), which both refreshes ciphertext noise and applies an arbitrary lookup function to the data. PBS dominates TFHE programs, taking up to 99.9% of an encrypted circuit's runtime, which makes it the prime target for hardware acceleration. The authors present FPT, an FPGA accelerator for PBS that, unlike prior second-generation FHE accelerators that hit the off-chip memory-bandwidth wall, stays entirely compute-bound.

Background: TFHE and Programmable Bootstrapping

TFHE is built on the Learning With Errors problem over the real torus T = R/Z (the interval [0,1) modulo 1), with torus elements discretized as 32-bit or 64-bit integers. A basic TLWE ciphertext is a tuple c = (a, b = a·s + e + μ), decryptable as b − a·s = μ + e using the secret s. Two variants extend this: TGLWE, where messages and errors are polynomials in T_N[X] = T[X]/(X^N + 1), and TGGSW (a matrix of TGLWE rows). Together they support a homomorphic External Product, written with the box-dot operator, which multiplies a TGLWE by a TGGSW. It first applies a gadget decomposition that splits each polynomial into l polynomials of β bits, then performs a vector-times-matrix polynomial product.

Programmable Bootstrapping homomorphically computes b − a·s and simultaneously evaluates a function f. A test polynomial F encodes N values of f and is blindly rotated by the secret amount b − a·s. Because the secret bits s_i cannot be revealed, each rotation step is performed with a Controlled MUX (CMUX): ACC is updated as (ACC·X^{a_i} − ACC) box-dot BK_i + ACC, where the bootstrapping key BK_1..BK_n are TGGSW encryptions of the secret coefficients. The full PBS (Algorithm 1) initializes ACC = F·X^{-b}, runs n CMUX iterations (the Blind Rotation), and extracts the output ciphertext. FPT implements two parameter sets: Set I (128-bit security, n=586, k=2, N=512, β=8, l=2, used by the CONCRETE Boolean library) and Set II (110-bit security, n=500, k=1, N=1024, β=10, l=2, used for benchmarking comparisons).

FFT-Based Polynomial Multiplication

PBS is dominated by the External Product, which is a vector-matrix product over large degree-N polynomials, so polynomial multiplication is the bottleneck. The FFT reduces schoolbook O(N^2) multiplication to O(N log N) via the convolution theorem. FHE needs multiplication modulo X^N + 1, requiring a negacyclic (negative-wrapped) convolution with period 2N, which naively needs a size-2N FFT. FPT combines three reductions: real-to-complex (r2c) FFTs exploit purely-real inputs for roughly a 2x saving; a twisting pre-processing step (multiplying coefficients by powers of the 2N-th root of unity) turns the negacyclic transform into a regular cyclic FFT of size N; and a tangent-FFT folding technique from NuFHE packs coefficients a[i] and a[i+N/2] into one complex number a[i] + j·a[i+N/2] before twisting. The net effect quarters the transform versus the naive size-2N FFT, so FPT uses cyclic FFTs of size N/2 = 256 (Set I) and N/2 = 512 (Set II).

FPT Core Idea: Streaming Architecture and Fixed-Point Arithmetic

FPT rests on two contributions. First, its microarchitecture is a streaming processor inspired by streaming DSPs rather than the classical CPU style of prior FHE accelerators. Instead of a control unit driving multiple Processing Elements from a register file, FPT instantiates a single wide CMUX PE whose datapath output hard-wires back into its input, exploiting the regular n-iteration structure of PBS. Internally the CMUX is a fixed cascade of throughput-balanced stages, each processing the same number of coefficients per clock cycle (the user-configurable streaming width), with static latency and minimal control logic. This yields 100% utilization of arithmetic units. Throughput-per-area rises with streaming width, motivating a single wide PE rather than many narrow ones.

Second, and giving the accelerator its name, FPT (Fixed-Point TFHE) performs PBS entirely in compact approximate fixed-point arithmetic. CPU and GPU TFHE are restricted to double-precision floating-point FFTs because single precision adds too much noise; some hardware works instead switch to a prime modulus to use exact NTTs, paying for modular reduction units. FPT avoids both FPUs and modular reduction by using fixed-point and analyzing the FFT approximation noise so that the bit widths are just large enough that this noise stays below the noise inherently present in FHE. This produces fixed-point representations up to 50% smaller than prior float or integer FFTs, which in turn enable up to 80% smaller FFT kernels. Crucially, finite-precision arithmetic does not affect security, since the server operates only on public data.

Batch Bootstrapping

Each CMUX iteration needs one bootstrapping-key coefficient BK_i, a large matrix of up to tens of kB that does not fit on-chip and must be streamed from off-chip memory. At high CMUX throughput the required BK bandwidth could exceed 1.0 TB/s, more than even HBM provides. FPT solves this with batch bootstrapping. Because the deeply pipelined CMUX has latency far larger than its throughput, FPT fills the pipeline with a batch of b ciphertexts that are all at the same iteration and encrypted under the same BK, so they all need the same BK_i. A small BRAM ping-pong buffer holds two BK coefficients: the CMUX reads BK_i from one half at 1.0 TB/s while off-chip memory fills BK_{i+1} into the other half at only 1.0/b TB/s. Using a C-slow-retiming-style trick, the batch size b can be increased by adding pipeline registers without any throughput penalty. A batch of just b=16 lets DDR4 instead of HBM supply the bandwidth, and such small batches are easy for TFHE applications such as homomorphic neural networks with hundreds to thousands of parallel ciphertexts. The result decouples on-chip from off-chip bandwidth, leaving PBS compute-bound with caches no larger than two BK coefficients.

Balancing the External Product and Streaming FFTs

The External Product needs (k+1)l forward FFTs but only (k+1) inverse FFTs, so the two must be throughput-balanced. FPT explores two layouts: a dotproduct-unrolled design that instantiates l times more FFT kernels than IFFT kernels, and the more unconventional FFT-unrolled design that gives the forward FFT l times the streaming width of the IFFT. The FFT-unrolled choice is more complex (it needs MAC units, a parallel-in serial-out buffer, and FFT blocks at differing throughputs) but uses fewer, wider FFT kernels, which favor the strong trend that pipelined FFTs gain throughput-per-area as streaming width grows, up to 300% more throughput per DSP or per LUT at the fully parallel extreme.

FPT uses continuous-flow pipelined FFTs (cascaded log(N) stages of butterflies and shuffling circuits) rather than the iterative FFTs/NTTs of prior FHE processors, which suffer memory conflicts at arbitrary streaming widths. The team evaluated several generators (Vitis SSR FFTs, Spiral, FloPoCo-based tools) and settled on the open-source Scala generator SGen, which they extended substantially: operators for forward and inverse twisting (for negacyclic support); radix-2^k structures (radix-2^4 FFTs average 10% smaller than radix-4 or radix-16); a Karatsuba/Gauss complex multiply needing 3 real multiplies and 3 adds instead of 4 and 2, saving DSPs; decoupling twiddle bit-width from data bit-width (matching the asymmetric 27x18 FPGA DSP multipliers and quantizing twiddles about 4 bits smaller); and a scaling schedule that halves data when the MSB would grow, gaining about 2 bits of precision. Together these cut DSP usage of a negacyclic size-256 FFT by over 60% versus stock SGen while keeping LUTs comparable.

Compact Fixed-Point Representation

FFTs operate on irrational complex numbers, so finite precision introduces approximation error; the goal is to keep that error below the inherent FHE noise so that the finite-precision failure probability stays arbitrarily close to the full-precision one. Reducing the bit width of intermediate variables sharply reduces area: a size-256 FFT at bit widths below 24 uses only about 20% of the LUTs and DSPs it needs at bit width 53 (which loosely matches double-precision significand size). Bit-width reduction has two parts. For the most significant bit, FPT avoids overflow by choosing the MSB position from the measured variance and a Gaussian (central-limit) model, targeting an overflow probability of P_of = 2^-64 rather than a costly worst-case guarantee. For the least significant bit, FPT splits the acceptable total noise so the approximation noise is at most half the inherent noise, treats the three main noise sources (forward FFT, inverse FFT, and the bootstrapping key BK) as independent, limits each to one third of the total variance, and sweeps each parameter down from a high width until it hits the noise threshold.

The resulting per-variable formats (Table 2) differ across blocks: for Set I, BK is FixedPoint26(7,19), the forward FFT FixedPoint29(15,14), and the IFFT FixedPoint29(23,6); for Set II, BK is FixedPoint27(8,19), FFT FixedPoint30(18,12), IFFT FixedPoint30(27,3). Versus the prior MATCHA approach, which used a single uniform 38-bit or 64-bit twiddle width and validated by running 10^8 tests, FPT's per-variable analysis is leaner: it gives smaller and differentiated representations (notably a smaller BK, shrinking both on-chip buffers and off-chip bandwidth) and a rigorous, measurable failure bound. Twiddle widths were heuristically set to the intermediate width minus 4; a full parameter search is left as future work. The inner streaming CMUX is built in Chisel using a DspComplex[FixedPoint] datatype, with experiments run directly on the generated Verilog.

FPGA Implementation

FPT is implemented on an AMD Alveo U280 datacenter card (1.3M LUTs, 2.6M FFs, 9024 DSPs, 41 MB on-chip SRAM) at 200 MHz, built with Vivado/Vitis 2022.2 and packaged as a Xilinx Run Time (XRT) RTL kernel. Both parameter sets use the FFT-unrolled architecture with a forward FFT streaming width of 128 complex coefficients per cycle and an IFFT width of 64. A CMUX completes every 12 cycles for Set I and every 16 cycles for Set II, while CMUX latency is 144 and 192 cycles respectively, so a batch of b = 12 ciphertexts keeps the pipeline at peak throughput. The dotproduct-unrolled variant, tested at iso-throughput for Set II, used about 10% more DSP and LUT area and was not pursued further.

The full fixed-point-trimmed BK is about 15 MB and fits entirely in on-chip BRAM and URAM, but a small ping-pong BK cache (only 2/n of the full BK) is still instantiated as a proof of concept so the design stays compute-bound on smaller FPGAs or memory-trimmed ASICs. BK is supplied at only tens of GB/s, split across HBM pseudo-channels 0-7 (14 GB/s each); input/output ciphertexts and the test-polynomial LUT RAM use negligible bandwidth. DSP blocks are the limiting resource: FPT uses about 61% of DSPs (5494) for Set I and 66% (5980) for Set II, with the CMUX consuming all the DSPs and the FFT/IFFT/MAC units the dominant consumers. Because FPGAs are reprogrammable, FPT pre-generates bitstreams tailored to each TFHE parameter set rather than being one-size-fits-all.

Results and Benchmarks

FPT achieves a PBS throughput of 1 bootstrap per 35 microseconds, or 28.4 PBS/ms for Set I (latency 0.66 ms) and 25.0 PBS/ms for Set II (latency 0.74 ms), where the latency covers a full batch of 12. Against the CONCRETE library on an Intel Xeon Silver 4208 CPU at 2.1 GHz (0.03 PBS/ms), FPT is about 937x faster; against the optimized TFHE-rs CPU library (0.11 PBS/ms) about 258x. It is 7.1x faster than the concurrent FPGA design YKP (Ye et al., 3.5 PBS/ms), and 2.5x faster than both the 36.96 mm^2 16 nm ASIC emulation MATCHA (10.0 PBS/ms) and the cuFHE GPU implementation on an NVIDIA RTX 3090 (9.6 PBS/ms). FPT draws about 99 W total on-chip (FPGA plus HBM), giving throughput-per-watt similar to MATCHA (40 W) and well above cuFHE (over 200 W) or YKP (50 W). Latency was not prioritized in this version but remains competitive with MATCHA, and can be reduced by shrinking the CMUX pipeline depth at the cost of more BK bandwidth or full on-chip BK caching.

An end-to-end demo runs encrypted Conway's Game of Life on an 8x8 board, where each time-step needs 64x44 = 2816 programmable bootstraps, integrated as an FPT backend inside TFHE-rs. In software a board update takes 29.1 s (26.7 s in bootstrapping, 2.3 s in key-switching). A direct FPT integration cuts this to 2.5 s, but Amdahl's law dominates: bootstrapping drops to 0.11 s while key-switching stays at 2.3 s, leaving the FPGA idle 96% of the time. Eliminating key-switching algorithmically (raising n to match kN, which makes bootstrapping about 1.75x more expensive) reduces the FPT update to 0.3 s while bloating software to 47.3 s, for a total ~100x to 158x application speedup with the FPGA active about 60% of the time. The authors note encrypted deep neural networks (about 1 million times slower than plaintext) could be brought down toward 10,000x with FPT. A cost study against AWS instances reports roughly a 15x monthly-cost advantage to reach FPT's performance, and at least 10x even when FPT is used only half the time. In summary FPT is, to the authors' knowledge, the first FHE bootstrapping design that is compute-bound rather than memory-bound, with 100% arithmetic-unit utilization, small caches, and up to 80% area-reduced fixed-point FFTs, reaching 28.4 bootstraps per millisecond.

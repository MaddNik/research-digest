Resource Estimation of CGGI and CKKS Scheme Workloads on FracTLcore® Computing Fabric

arXiv (Cornami, Inc.) · October 2025 · full view ↗

Overview

Cornami, Inc. builds custom silicon for accelerating Fully Homomorphic Encryption (FHE) — computation directly on encrypted data. This paper, authored by a Cornami engineering team, describes a resource-estimation methodology built while constructing a general-purpose compiler backend for Cornami's Mx2 chip. The Mx2 is a FracTLcore® compute fabric processor: a systolic array of cores with in-memory compute and a network-on-chip (NoC), supporting reconfigurable arrays, immediate memory access, pipeline parallelism, and fast function reprogramming — capabilities the authors contrast with what GPUs/CPUs traditionally offer.

The paper's core contribution is not a new FHE scheme but a compiler-integrated cost model: given an FHE compute graph in either the CGGI/TFHE-rs (Boolean, gate-bootstrapped) scheme or the CKKS (approximate, packed-arithmetic) scheme, estimate how many Mx2 hardware resources (chips, boards, cores) and how much latency (critical path) the workload will consume — before or without running it on hardware. Notably, one of the co-authors, Craig Gentry, is also the author of the paper's own reference [1] — the 2009 STOC construction that introduced fully homomorphic encryption using ideal lattices.

Software Stack: SCIFR API and TruStream SDK

Cornami exposes its hardware through SCIFR (Secure Computing Interface FRamework): a standardized, hardware-agnostic API for deploying industry-standard FHE frameworks on the FracTLcore fabric. SCIFR abstracts hardware orchestration so fabric resources can be dynamically tuned for performance vs. cost, supports mixed-schema support for hybrid encryption flows, and supports full-circuit execution directly in hardware — eliminating co-processor round-trips. The TruStream® SDK sits alongside SCIFR, letting developers extend FHE applications and integrate new cryptographic primitives. Target markets named include smart contracts, healthcare, manufacturing, and FinTech.

Figure 1 shows the layering: applications sit on the FHE Library, which sits on SCIFR API, which sits on the TruStream® SDK, running on Linux OS, ultimately executing on the FracTLcore® Computing Fabric Hardware (the Mx2 chip).

Compiler Pipeline: HEIR / MLIR Backend

Rather than hand-coding every FHE application against SCIFR/TruStream directly, Cornami built a compiler backend on top of Google's HEIR (Homomorphic Encryption Intermediate Representation), an MLIR-based framework designed to standardize intermediate representations across FHE schemes (BGV, CKKS, TFHE-rs, etc.).

The pipeline (Figure 2) runs: a source compute graph from TensorFlow/PyTorch/JAX → lowered into the HEIR FHE graph → HEIR transforms (polynomial approximation, "secretize", CGGI lowering, etc.) → conversion into Cornami's own exit dialects: SCIFRBool for the CGGI/TFHE-rs (Boolean) scheme and SCIFRCkks for the CKKS scheme → a codegen emitter produces C++ against the Cornami Concrete/AppStream framework, which runs on the Mx2 emulator or real hardware. The resource-estimation pass described in this paper is attached directly at the exit-dialect stage, so estimates are produced as a natural side effect of compilation rather than a separate simulation step.

Resource Model: Boards, Chips, Tiles, FCs

The Mx2 resource hierarchy used throughout the paper, largest to smallest:

  • Boards ("Mx8" boards, each holding 4 Mx2 chips)
  • Chips (Mx2)
  • Tiles
  • FCs — FractalCores, the fundamental Processing Elements (PEs) of the fabric
  • HBM and DDR memory units

Estimation proceeds by a cumulative-sum graph traversal: every FHE scheme operator (for CKKS: Key Switch (KS), Bootstrap (PBS), Ciphertext Multiply, Plaintext Multiply, Square, Rotate) is lowered by the compiler into a known Cornami hardware topology with a known FC cost; the compiler walks the compute graph and sums these elemental costs to produce total resource consumption for the whole application. The authors note their approach differs from prior FPGA resource-estimation work (ref. [8], de Castro et al., "Does fully homomorphic encryption need compute acceleration?") by being tuned specifically to the Mx2 architecture and wired directly into a compiler pipeline rather than being a standalone estimator.

Alongside resource count, the paper estimates a critical-path (CP) metric — the longest time for input to propagate to output, i.e. time-to-first-output-ciphertext. This matters specifically because, per the authors, the Cornami architecture and other dataflow machines are inherently pipelined, and smaller critical paths are Pareto-optimal in that setting. Two algorithms are given for this — an approximate version (fast, ignores actual path lengths between nodes) and an exact version (computes true shortest/critical paths between all source-sink pairs); see the Algorithms section below.

The 8-bit multiply is by far the heaviest case shown: it needs 18 Mx2 chips across 5 boards and a critical path 35× longer than a single LUT canonicalization, illustrating how quickly gate-level Boolean FHE circuits scale in hardware demand as bit-width grows.

To handle circuits too large to fit at once, the authors added an MLIR transformation that inserts Sections into SCIFR Bool, dividing large Boolean circuits into pieces that can be mapped onto the Mx2 processor in a time-division-multiplexed fashion. Both the CGGI→SCIFRBool lowering and this Sections transform are described as extensible, so new operations, types, or transformations can be added to the framework.

CKKS Estimation

For the packed-arithmetic CKKS scheme, the compiler exposes a SCIFRCkks exit dialect (Table 4): AddOp, AddPlainOp, SubOp, SubPlainOp, MulOp, MulPlainOp, RotateOp, ExtractOp, NegateOp, RelinearizeOp, RescaleOp. Each of these operators corresponds to a known hardware topology consisting of 500+ FCs, and a full application's implementation may span several Mx2 chips or Mx8 boards.

The same cumulative-sum-over-graph-traversal approach used for CGGI is reused here, invoked as:

$ scifr-opt --ckks-tigris-estimate ckks_example.mlir

Notably, box_blur and gx_kernel produce exactly identical counts (both columns) between their 4×4/8×8 and 64×64 variants, while roberts_cross keeps the same total op count (18) between sizes but its critical-path count rises slightly (30 → 32) — so size-invariance holds for total work but is not universal for the critical-path metric. The source does not explain this pattern; it is plausibly related to CKKS's packed-ciphertext (SIMD-style) model, where enlarging the data packed into each ciphertext slot need not change the operator count, but this is the summarizer's inference, not a claim made in the paper. The paper notes future performance work: denser PBS/KS operation packing, fast-bootstrapping, SIMD, and an in-progress block-placer to maximize core utilization on a single Mx2 chip.

Results: int8 Multiplication Case Study

The authors trace a concrete end-to-end example — encrypted 8-bit integer multiplication — through their full toolchain:

  1. Write the source program in the Arith MLIR dialect with plaintext inputs.
  2. Use heir-opt to generate the encrypted (CGGI) version with LWE ciphertext inputs.
  3. Use scifr-opt to convert CGGI → SCIFR Bool.
  4. Use scifr-translate to generate C++ against the Cornami TFHE-rs implementation.

The generated int8-multiply Boolean circuit has a critical path 14 gates deep. At an assumed 1 unit-time per gate operation, that gives roughly 14 unit-time per int8 product on a single Mx8 board, running at 50% per-chip occupancy — the current bottleneck is manual placement/organization of PBS+KS operations rather than a fundamental hardware limit; the authors expect an automatic place-and-route solution (in development) to push utilization to 75–90%. Because the Mx2's systolic/pipelined architecture keeps the pipeline full across a batch, throughput = batch size / per-output latency: at 14 unit-time per output and a batch of 1000, this yields 71 encrypted values produced per unit time once the pipeline is filled. The authors also flag their current gate library as an efficiency target: it implements all 2-bit encrypted gates as linear combinations built from a single fundamental AND-gate LUT, and note that adding a unique dedicated LUT per gate type would shrink the area footprint of these fundamental blocks.

Conclusion

The paper's contribution is a resource-estimation methodology — covering both hardware resource counts (FCs/chips/boards/memory) and critical-path latency — for FHE workloads in both the CGGI/TFHE-rs Boolean scheme and the CKKS approximate-arithmetic scheme, built as an integrated analysis pass inside a HEIR/MLIR compiler backend targeting Cornami's Mx2 FracTLcore fabric. This supports a companion effort (ref. [2], presented at NIST's 6th PQC Conference, 2025) building the full TFHE-rs compiler backend for the same hardware. The work positions Cornami's approach as differentiated from prior FPGA-targeted FHE resource-estimation work by being co-designed with the compiler pipeline and tuned specifically to the systolic, in-memory-compute Mx2 architecture rather than being a general or FPGA-generic model.

Algorithms

The paper gives two critical-path (CP) estimation algorithms over the input MLIR operator compute graph G_in. Both rely on a topological sort of the graph plus its identified source and sink nodes.

Algorithm: ApproximateCriticalPathEstimation[ G_in : Input MLIR Operator Graph ]
  Input:  G_in, a DAG of FHE operator nodes
  Output: cp_result, an ordered list of non-source/non-sink nodes on the topological order

  1. result  <- topologicalSort(G_in)
  2. sinks   <- getSinks(G_in)
  3. sources <- getSources(G_in)
  4. cp_result <- []
  5. for node in result[:-1]:
  6.     if node in sources or node in sinks:
  7.         continue
  8.     cp_result.append(node)
  9. return cp_result

This approximate version does not compute an actual path length between specific source/sink pairs — it simply returns all "interior" (non-boundary) nodes in topological order as a proxy for the critical path.

Algorithm: ExactCriticalPathEstimation[ G_in : Input MLIR Operator Graph ]
  Input:  G_in, a DAG of FHE operator nodes
  Output: cp_result, the longest source-to-sink path (the true critical path)

  1. result  <- topologicalSort(G_in)
  2. sinks   <- getSinks(G_in)
  3. sources <- getSources(G_in)
  4. cp_result <- []
  5. plen <- 0
  6. for src in sources:
  7.     for sink in sinks:
  8.         if src == sink:
  9.             continue
  10.        path    <- getShortestPath(src, sink)
  11.        currlen <- len(path)
  12.        if currlen > plen:
  13.            plen      <- currlen
  14.            cp_result <- path
  15. return cp_result

The exact version enumerates every (source, sink) pair and keeps the longest such path — giving the true "time to first output ciphertext" at the cost of being more expensive to compute than the approximate version, since it checks every source against every sink. In the paper's usage, getShortestPath over an unweighted DAG effectively measures hop-count depth, so despite the name this recovers the longest dependency chain length between any source and any sink.

Multi-Key Homomorphic Encryption from TFHE

PDF (text-layer; tables and matrix rasterized from the typeset PDF) — IACR ePrint 2019/116, retrieved via Wayback Machine · full view

At a glance

Authors: Hao Chen (Microsoft Research), Ilaria Chillotti (imec-COSIC, KU Leuven), Yongsoo Song (Microsoft Research). Venue: ASIACRYPT 2019 (IACR ePrint 2019/116).

One sentence: The paper builds the first practical, implemented Multi-Key Homomorphic Encryption (MKHE) scheme by generalizing TFHE's gate bootstrapping so a single bootstrapped binary gate can be evaluated on ciphertexts encrypted under different parties' independently generated keys, with a new "hybrid product" that multiplies one party's bootstrapping key into a multi-key ciphertext.

Headline results (single thread, Intel Core i7-4910MQ @ 2.90 GHz; the same machine bootstraps a single-key TFHE gate in 13 ms):

  • Bootstrapped NAND across k = 2 / 4 / 8 parties: 0.27 s / 1.45 s / 7.16 s.
  • Ciphertext length grows linearly in k; gate cost grows quadratically in k (Õ(k2n2)).
  • Per-party multi-key ciphertext: 2.19 KB; combined NAND-input ciphertext 4.38 KB (k=2) to 17.32 KB (k=8); bootstrapping key 0.62-1.03 MB.
  • At least 110-bit security (LWE Estimator); two hybrid-product algorithms, the second with better complexity and lower noise.

Significance: the first MKHE scheme ever implemented — prior MKHE constructions (Lopez-Alt et al., Clear-McGoldrick, Mukherjee-Wichs, Peikert-Shiehian, Brakerski-Perlman, Chen-Zhang-Wang) were all purely asymptotic and impractical. Open-sourced as MK-TFHE, built on the TFHE library.

Keywords: Multi-key homomorphic encryption, bootstrapping, TFHE, LWE/RLWE, external product, GSW ciphertext extension.

The multi-key problem and why it matters

Ordinary Homomorphic Encryption lets a cloud compute on encrypted data, but every data provider must encrypt under the same public key — so someone must own the matching secret key, which is awkward for joint computation between mutually distrustful parties. Multi-Party Computation (MPC) avoids a single key owner but pays in communication and rounds.

Multi-Key HE (MKHE), introduced by Lopez-Alt et al. (2012), sits between the two: parties encrypt under their own independently generated keys, the server homomorphically computes a function whose result depends on several parties' inputs, and decryption requires the cooperation of (only) the relevant parties' secret keys. It enables round-efficient MPC and spooky encryption. The catch had always been efficiency: every prior MKHE was GSW-based with ciphertext size growing at least quadratically in the number of parties k, and none had ever been implemented.

This paper makes the multi-key setting concrete. A multi-key ciphertext of a bit m is a vector ct = (b, a1, ..., ak) ∈ Tkn+1 such that b + <a1,s1> + ... + <ak,sk> ≈ (1/4)m (mod 1), where each party i contributes its own LWE secret si. The length grows linearly in k (one ai block per party), and any two parties' ciphertexts are merged simply by rearranging blocks and zero-padding the empty slots so they share the concatenated secret s = (s1, ..., sk).

Relation to TFHE and the core challenge

The scheme reuses TFHE's machinery wholesale: torus T = R mod 1, the ring variant T = T[X]/(XN+1), TLWE/TRLWE samples, the gadget vector g = (B-1, ..., B-d), and the external product — multiply a Ring-GSW (RGSW) ciphertext into an RLWE ciphertext to get an RLWE ciphertext, with additive noise growth. In single-key TFHE, bootstrapping a gate is a recursion of CMux gates driven by RGSW encryptions of the secret-key bits (the "bootstrapping key"), blind-rotating a fixed test polynomial.

The single new difficulty in the multi-key case is the second bootstrapping step. The bootstrapping key for party i is a set of RGSW-like encryptions of i's LWE secret bits si,j, generated by party i alone under i's own key. But during bootstrapping it must be multiplied into a multi-key RLWE accumulator that lives under the concatenated secret z = (1, z1, ..., zk) of all parties. The paper calls this single-key-times-multi-key multiplication the hybrid product, and designing it efficiently is the central technical contribution. The other two bootstrapping steps (the initial linear combination / NAND gate, and the final key switch) generalize straightforwardly to the concatenated secret.

To support the hybrid product the authors introduce uni-encryption, an RGSW-like symmetric cryptosystem in the Common Reference String (CRS) model. Every party shares a public CRS vector a ∈ Td and publishes a public key bi ≈ -zi·a. A uni-encryption of a plaintext μi is a pair (di, Fi = [f0|f1]) ∈ Td × Td×2 with di ≈ ri·a + μi·g (encrypting μi under randomness ri) and f0 + zi·f1 ≈ ri·g (encrypting that randomness ri under zi). It is three d-dimensional vectors — only three-quarters the size of an ordinary RGSW ciphertext in T2d×2 — because it is used only to build bootstrapping keys, never to encrypt real data.

Hybrid product, Method 1: improved ciphertext extension

The first hybrid-product algorithm follows the classical "GSW ciphertext extension" idea (Clear-McGoldrick, simplified by Mukherjee-Wichs, EUROCRYPT 2016) but improves it. It works in two phases: (1) extend a uni-encryption (di, Fi) of μi — together with all parties' public keys {bj} — into a full multi-key RGSW ciphertext D̄i ∈ Td(k+1)×(k+1) that encrypts the same message under the concatenated secret z, satisfying D̄i·z ≈ μi·Gk+1z; then (2) apply the multi-key external product c' = G-1k+1(c)·D̄i, the natural generalization of TFHE's external product, which yields a multi-key RLWE ciphertext c' with <c', z> ≈ μi·<c, z>.

The extension fills D̄i using xj, yj computed from gadget-decomposing each public key bj against Fi (correctness rests on xj + zi·yj ≈ ri·bj and zj·di ≈ -ri·bj + μizj·g, which combine to μizj·g). Two improvements over prior work: the extended dimension is cut from 2k down to k+1 by merging the duplicated (1, zi) components into the single concatenated key z = (1, z1, ..., zk); and the symmetric uni-encryption shrinks the keys further. Extension costs O(kd2) polynomial multiplications and the sparse D̄i stores in O(kd) polynomials; the external product is O(kd) multiplications. Crucially, the extended ciphertext D̄i depends only on the keys, so it can be pre-computed once and reused as a shared bootstrapping key for any gate on the same set of parties — at the price of O(k2nd) storage.

Figure (page 11 of the paper): the extended multi-key RGSW ciphertext D̄i ∈ Td(k+1)×(k+1) produced by the Extend step. It is sparse — each block-row carries di on a diagonal-like position plus the xj, yj terms in the 0th and i-th block columns — which is what makes the external product cheap.

Hybrid product, Method 2: acting directly on the ciphertext

The second algorithm is a genuinely new approach and is the one used in the implementation. Instead of first building a multi-key RGSW ciphertext independently of the input, it operates directly on the multi-key RLWE ciphertext c = (c0, ..., ck), never materializing any multi-key GSW object. For each j it gadget-decomposes cj and forms uj = <g-1(cj), di>, vj = <g-1(cj), bj>, then wj,0 = <g-1(vj), f0> and wj,1 = <g-1(vj), f1>. The output absorbs the w-sums into the 0th and i-th components, again giving <c', z> ≈ μi·<c, z>.

Two advantages over Method 1, both important in practice:

  • Lower complexity: only 4(k+1)d = O(kd) polynomial multiplications, giving asymptotic cost O(kd·N log N) versus O(kd2·N log N) for the extension in Method 1.
  • Less noise: output noise variance V2 ≈ (1/2)(kd + k + 1)·N2·VB·β2, smaller than Method 1's V1 ≈ (k+1)d2·N2·VB2·β2 by roughly a factor of O(d·VB) — the paper's analogy is the GSW fact that (C1⊠C2)⊠C3 and C1⊠(C2⊠C3) compute the same product but with very different error.

The trade-off: Method 1's extended ciphertext is reusable (pre-processable), and its Step-2 external product alone is about 3/4 the cost of Method 2; so with heavy reuse Method 1 can win on per-gate time at the cost of quadratic-in-k storage. But Method 2's smaller noise lets one pick smaller parameters, so the authors judge it more practical overall.

The CMux (controlled selector) gate is the direct application: CMux(c0, c1, (di,Fi), {bj}) = c0 + Prod(c1-c0, (di,Fi), {bj}), selecting between two multi-key ciphertexts under an encrypted bit μi. CMux is the workhorse inside the multi-key blind rotation.

Multi-key gate bootstrapping (the NAND pipeline)

MKHE.NAND runs in three phases, mirroring TFHE but over the concatenated keys:

Phase 1 — linear combination. Extend ct1, ct2 to share the concatenated secret s by rearranging blocks and zero-filling, then form ct' = (5/8, 0, ..., 0) - ct'1 - ct'2 (mod 1), which encodes m = m1 NAND m2 with <ct', (1,s)> ≈ (1/2)m + e' and |e'| ≤ 1/4.

Phase 2 — homomorphic accumulator / blind rotation. Quantize to b̃ = round(2N·b'), ãi = round(2N·a'i) and initialize the multi-key RLWE accumulator c = (-(1/8)h(X)·Xb̃, 0) ∈ Tk+1 with the fixed test polynomial h(X) = ∑-N/2<j<N/2 Xj, defined over the concatenated RLWE secret z = (1, z1, ..., zk). Then recursively apply, for every party i and bit j, c ← CMux(c, Xãi,j·c, (di,j, Fi,j), {bℓ}) using the encryptions of the secret bits si,j. This blind-rotates the accumulator by b̃ + ∑i<ãi, si> ≈ N·m (mod 2N), so the constant term of <c, z> becomes ≈ (1/4)m - 1/8; adding 1/8 cancels the offset. This is exactly where the hybrid product is invoked — k·n times — and it dominates the whole pipeline's cost.

Phase 3 — sample extraction and multi-key switching. Extract an LWE ciphertext ct* = (b*, a*1, ..., a*k) from the accumulator (the constant term of ci·zi equals <a*i, z*i>, using the permuted coefficient vector z*i), then run multi-key switching: apply each party's ordinary key-switching key KSi (from z*i to si) independently to return a fresh multi-key LWE ciphertext under s with <ct, (1,s)> ≈ (1/4)m. Per-party key switching generalizes cleanly because the concatenated KS = {KSi} acts as a key-switch from t ∈ ZkN to s ∈ Zkn.

Any binary gate (AND/OR/XOR/...) follows by changing only the Phase-1 linear combination. The scheme inherits TFHE's circular-security assumption (each party publishes a bootstrapping key encrypting its own LWE secret bits, plus a key-switch from its RLWE to its LWE key).

Complexity and comparison to prior MKHE

Net asymptotics: ciphertext length is Õ(kn) (linear in parties), the evaluation key is Õ(k2n2), and a bootstrapped NAND costs Õ(k2n2) — quadratic in the number of parties. This is the first MKHE that is simultaneously (a) bootstrappable, (b) compact (linear-size ciphertexts, like Brakerski-Perlman), and (c) concretely fast. Brakerski-Perlman also has linear ciphertexts but uses a purely abstract branching-program bootstrapping of length poly(k,n) whose concrete cost was never analyzed; the GSW-based schemes (CZW17, PS16, Clear-McGoldrick, Mukherjee-Wichs) have at least quadratic-in-k ciphertexts.

The improved (compact, symmetric) ciphertext-extension of Method 1 can be dropped into the GSW-based schemes [7,10] to speed up their evaluation-key generation. The construction is also compatible with existing extensions — threshold/distributed decryption via noise smudging (PartDec + Merge), circuit bootstrapping, and plaintext packing — in the multi-key setting.

Parameters, security, and experimental results

The proof-of-concept ("a few days of coding") sits on top of the TFHE library and is open source at github.com/ilachill/MK-TFHE. It uses Method 2 for the hybrid product and binary LWE/RLWE secret distributions. Three parameter sets share LWE dimension n = 560 (α = 3.05·10-5) and RLWE dimension N = 1024 (β = 3.72·10-9); they differ only in the RGSW decomposition base/degree (B = 29/28/26, d = 3/4/5) so that larger sets tolerate more parties. All achieve at least 110-bit security per the LWE Estimator. Because bootstrapping-error standard deviation grows only linearly in k, the parameter growth with the maximum party count is very slow, controlled by adjusting the decomposition degree (which does not affect security).

Sets I / II / III support up to 2 / 4 / 8 parties respectively. Per-party local cost is independent of k: key generation 1.1-1.3 s, bootstrapping key 0.62-1.03 MB, key-switching key 70.1 MB, and a single-party ciphertext 2.19 KB. The global bootstrapped-NAND timings (single thread): k=2 in 0.27 s, k=4 in 1.45 s, k=8 in 7.16 s — the quadratic-in-k growth is visible. The merged multi-key ciphertext grows linearly: 4.38 KB (k=2) → 8.77 KB (k=4) → 17.32 KB (k=8). Noise estimation (Appendix A) confirms the bootstrapping noise is the sum of the accumulation and multi-key-switching errors, and includes a modified key-switching variant with smaller error.

Performance table and takeaways

The work delivers the first concrete demonstration that multi-key FHE can be run at all: a bootstrapped binary gate across up to eight parties in seconds on a laptop, with ciphertexts that grow only linearly in the number of parties. Its core technical contribution — the hybrid product, and especially Method 2 acting directly on the multi-key ciphertext — cleanly extends TFHE's external-product-based gate bootstrapping to the multi-key world while improving storage, time, and noise simultaneously.

Stated open directions: porting TFHE's advanced functionalities (circuit bootstrapping, packing) to the multi-key setting; building practical MKHE from amortized schemes such as BFV or CKKS; and the headline open problem of constructing MKHE without the CRS model. The CRS assumption (a shared public a) underlies the uni-encryption security proof here.

Multi-Key Homomorphic Encryption from TFHE

ASIACRYPT 2019 (IACR ePrint 2019/116) · full view ↗

At a glance

Authors: Hao Chen (Microsoft Research), Ilaria Chillotti (imec-COSIC, KU Leuven), Yongsoo Song (Microsoft Research). Venue: ASIACRYPT 2019 (IACR ePrint 2019/116).

One sentence: the paper builds the first practical, implemented Multi-Key Homomorphic Encryption (MKHE) scheme by generalizing TFHE gate bootstrapping so that a single bootstrapped binary gate can be evaluated on ciphertexts encrypted under different parties' independently generated keys, using a new hybrid product that multiplies one party's bootstrapping key into a multi-key ciphertext.

Headline results (single thread, Intel Core i7-4910MQ at 2.90 GHz; the same machine bootstraps a single-key TFHE gate in about 13 ms):

  • Bootstrapped NAND across $k = 2 / 4 / 8$ parties: 0.27 s / 1.45 s / 7.16 s.
  • Ciphertext length grows linearly in $k$; gate cost grows quadratically in $k$, i.e. $\tilde{O}(k^2 n^2)$.
  • Per-party single-key ciphertext: 2.19 KB; merged multi-key ciphertext 4.38 KB ($k=2$) up to 17.32 KB ($k=8$); bootstrapping key 0.62 to 1.03 MB.
  • At least 110-bit security (LWE Estimator); two hybrid-product algorithms, the second with better complexity and lower noise.

Significance: the first MKHE scheme ever implemented. Prior MKHE constructions (Lopez-Alt et al., Clear and McGoldrick, Mukherjee and Wichs, Peikert and Shiehian, Brakerski and Perlman, Chen-Zhang-Wang) were all purely asymptotic and impractical. Open-sourced as MK-TFHE, built on the TFHE library at github.com/ilachill/MK-TFHE.

Keywords: multi-key homomorphic encryption, bootstrapping, TFHE, LWE/RLWE, external product, GSW ciphertext extension, uni-encryption.

The multi-key problem and why it matters

Ordinary Homomorphic Encryption (HE) lets a cloud compute on encrypted data, but every data provider must encrypt under the same public key, so a single secret-key owner must exist, which is awkward for joint computation between mutually distrustful parties. Multi-Party Computation (MPC) avoids a single key owner but pays in communication and round complexity.

Multi-Key HE (MKHE), introduced by Lopez-Alt et al. (2012), sits between the two: parties encrypt under their own independently generated keys, the server homomorphically computes a function whose result depends on several parties' inputs, and decryption requires the cooperation of (only) the relevant parties' secret keys. It enables round-efficient MPC and spooky encryption. The catch had always been efficiency: every prior MKHE was GSW-based with ciphertext size growing at least quadratically in the number of parties $k$, and none had ever been implemented.

A multi-key ciphertext of a bit $m \in \{0,1\}$ is a vector $\mathrm{ct} = (b, a_1, \dots, a_k) \in \mathbb{T}^{kn+1}$ such that

$$b + \langle a_1, s_1 \rangle + \cdots + \langle a_k, s_k \rangle \approx \tfrac{1}{4}m \pmod 1,$$

where each party $i$ contributes its own LWE secret $s_i = (s_{i,j})_{1 \le j \le n} \in \{0,1\}^n$ (Equation 1). The length grows linearly in $k$ (one block $a_i$ per party), and any two parties' ciphertexts are merged simply by rearranging blocks and zero-padding the empty slots so that they share the concatenated secret $s = (s_1, \dots, s_k)$.

Syntax. An MKHE scheme is a tuple of five PPT algorithms: $\mathsf{Setup}(1^\lambda) \to pp$; $\mathsf{KeyGen}(pp) \to (sk, pk)$ (each party has an ID mapped to its keys); $\mathsf{Enc}(m; pk) \to \mathrm{ct}$ (every ciphertext records the IDs of the parties it involves); $\mathsf{Dec}(\mathrm{ct}; \{sk_i\}_{i \in [k]}) \to m$ over the relevant secret keys; and $\mathsf{NAND}(\mathrm{ct}_1, \mathrm{ct}_2, \{pk_i\}_{i \in [k]}) \to \mathrm{ct}'$, where $k$ is the number of parties involved in either input. Correctness requires $\mathsf{Dec}(\mathrm{ct}', \{sk_i\}) = m_1 \,\overline{\wedge}\, m_2$ with overwhelming probability, and security is semantic security of the encryption.

Relation to TFHE and the core challenge

The scheme reuses TFHE machinery wholesale: the torus $\mathbb{T} = \mathbb{R} \bmod 1$, the ring variant $\mathcal{T} = \mathbb{T}[X]/(X^N+1)$ with $R = \mathbb{Z}[X]/(X^N+1)$ for power-of-two $N$, TLWE/TRLWE samples, the gadget vector $g = (B^{-1}, \dots, B^{-d})$, and the external product, which multiplies a Ring-GSW (RGSW) ciphertext into an RLWE ciphertext and returns an RLWE ciphertext with only additive noise growth: $A \boxdot b = G_2^{-1}(b) \cdot A$. In single-key TFHE, bootstrapping a gate is a recursion of CMux gates driven by RGSW encryptions of the secret-key bits (the bootstrapping key), blind-rotating a fixed test polynomial.

The single new difficulty in the multi-key case is the second bootstrapping step. The bootstrapping key for party $i$ is a set of RGSW-like encryptions of $i$'s LWE secret bits $s_{i,j}$, generated by party $i$ alone under $i$'s own key. But during bootstrapping it must be multiplied into a multi-key RLWE accumulator that lives under the concatenated secret $z = (1, z_1, \dots, z_k)$ of all parties. The paper calls this single-key-times-multi-key multiplication the hybrid product, and designing it efficiently is the central technical contribution. The output $c'$ must satisfy

$$\langle c', z \rangle \approx \mu_i \cdot \langle c, z \rangle \pmod 1,$$

where $\mu_i$ is the plaintext carried by party $i$'s bootstrapping key (Equation 2). The other two bootstrapping steps (the initial NAND linear combination and the final key switch) generalize straightforwardly to the concatenated secret.

To support the hybrid product the authors introduce uni-encryption, an RGSW-like symmetric cryptosystem in the Common Reference String (CRS) model. Every party shares a public CRS vector $a \in \mathbb{T}^d$ and publishes a public key $b_i \approx -z_i \cdot a$. A uni-encryption of a plaintext $\mu_i \in R$ is a pair $(d_i, F_i = [f_0 \mid f_1]) \in \mathbb{T}^d \times \mathbb{T}^{d \times 2}$ satisfying

$$d_i \approx r_i \cdot a + \mu_i \cdot g, \qquad f_0 + z_i \cdot f_1 \approx r_i \cdot g \pmod 1.$$

That is, $d_i$ together with the CRS $a$ encrypts $\mu_i$ under randomness $r_i$, and $F_i$ encrypts that same randomness $r_i$ under $z_i$ (Equation 3). It is three $d$-dimensional vectors, only three-quarters the size of an ordinary RGSW ciphertext in $\mathbb{T}^{2d \times 2}$, because it is used only to build bootstrapping keys, never to encrypt real data.

Background: LWE, RLWE, RGSW, and multi-key switching

A TLWE sample is a pair $(b, a) \in \mathbb{T}^{n+1}$ with $a$ uniform over $\mathbb{T}^n$ and $b = \langle a, s \rangle + e$, where $s \leftarrow \chi$ and $e \leftarrow D_\alpha$ (Gaussian of standard deviation $\alpha$). The ring analogue (TRLWE) is a pair of polynomials $(b, a) \in \mathcal{T}^2$ with $b = a \cdot z + e \pmod 1$, secret $z \leftarrow \psi$ (the paper takes $\psi$ uniform over binary-coefficient polynomials), and error of standard deviation $\beta$. Encryption adds the message: an LWE ciphertext of $m$ is $\mathrm{ct} = (b, a)$ with $b = -\langle a, s \rangle + \tfrac{1}{4}m + e \pmod 1$ (scaling factor $1/4$, as in FHEW/TFHE), and decryption computes the phase $\varphi_s(\mathrm{ct}) = b + \langle a, s \rangle$ and rounds to the nearest message.

For base $B \ge 2$ and degree $d$, the gadget matrix is $G_k = I_k \otimes g \in \mathbb{T}^{dk \times k}$, and the base decomposition $G_k^{-1}(u)$ has coefficients in $\mathbb{Z} \cap (-B/2, B/2]$ and minimizes $\lVert v^\top G_k - u^\top \rVert_\infty$, with decomposition error bounded by $\tfrac{1}{2}B^{-1}$. An RGSW sample is $C = Z + \mu \cdot G_2$, where each row of $Z \in \mathcal{T}^{d \times 2}$ is a TRLWE encryption of zero; the external product is $A \boxdot b = G_2^{-1}(b) \cdot A$.

The paper first generalizes ordinary LWE key-switching to the multi-key case. An extended ciphertext $\mathrm{ct} = (b, a_1, \dots, a_k) \in \mathbb{T}^{kn+1}$ encrypts $m$ under the concatenated secret $s = (s_1, \dots, s_k)$ if

$$\langle \mathrm{ct}, (1, s) \rangle = b + \sum_{i=1}^{k} \langle a_i, s_i \rangle \approx \tfrac{1}{2}m \pmod 1.$$

This is the encoding used between linear operations (Equation 4). A key-switching key $\mathrm{KS}_i = \{K_{i,j}\}_{j \in [N]}$ from secret $t_i$ to $s_i$ is built with $K_j = [b_j \mid A_j]$, $b_j = -A_j s_i + e_j + t_{i,j} \cdot g'$, using its own gadget $g' = (B'^{-1}, \dots, B'^{-d'})$. Multi-key switching applies each party's key-switching key independently. The procedure verbatim:

LWE.MKSwitch(ct, {KS_i}_{i in [k]}):
  // Input: ct = (b, a_1, ..., a_k) in T^{kN+1}
  //        KS_i = {K_{i,j}}_{j in [N]}   (key-switch key from t_i to s_i)
  for i in [k]:
      (b'_i, a'_i) = sum_{j=1..N}  g'^{-1}(a_{i,j}) . K_{i,j}   (mod 1)
  b' = b + sum_{i=1..k} b'_i   (mod 1)
  return ct' = (b', a'_1, ..., a'_k) in T^{kn+1}

// Correctness: KS = {KS_i} acts as a single key-switch from
//   t in Z^{kN}  to  s in Z^{kn}, so <ct',(1,s)> approx <ct,(1,t)>.

Correctness follows because $b' + \sum_i \langle a'_i, s_i \rangle \approx b + \sum_i \sum_j \langle g'^{-1}(a_{i,j}), t_{i,j} \cdot g' \rangle \approx \langle \mathrm{ct}, (1, t) \rangle$, so $\mathrm{KS} = \{\mathrm{KS}_i\}$ behaves as a single key-switch from $t \in \mathbb{Z}^{kN}$ to $s \in \mathbb{Z}^{kn}$. Security: each $K_j$ is LWE instances under $s_i$ with $t_{i,j} \cdot g'$ added to the first column, so $\mathrm{KS}_i$ is computationally indistinguishable from uniform under LWE with parameter $(n, \chi, \beta)$.

Uni-encryption and its security

The ring-based building block is a single-key symmetric encryption $\mathsf{UniEnc}$ that can encrypt a ring element and is later multiplied into multi-key ciphertexts. Its setup fixes the RLWE dimension $N$ (power of two), key distribution $\psi$ over $R$, error parameter $\alpha$, base $B \ge 2$ with degree $d$, and a CRS vector $a \leftarrow U(\mathbb{T}^d)$ (this CRS is what places the scheme in the CRS model). Key generation samples $z \leftarrow \psi$, sets $z = (1, z)$, and publishes $b = -z \cdot a + e \pmod 1$ for $e \leftarrow D_\alpha^d$. Uni-encryption of $\mu \in R$ samples $r \leftarrow \psi$ and errors $e_1, e_2 \leftarrow D_\alpha^d$, $f_1 \leftarrow U(\mathbb{T}^d)$, and returns

$$d = r \cdot a + \mu \cdot g + e_1, \qquad F = [f_0 \mid f_1], \quad f_0 = -z \cdot f_1 + r \cdot g + e_2 \pmod 1.$$

So $(d, a)$ encrypts $\mu$ under randomness $r$, while $F$ encrypts $r$ under $z$. A uni-encryption is three $d$-dimensional polynomial vectors, three-quarters the size of an ordinary RGSW ciphertext in $\mathcal{T}^{2d \times 2}$.

Semantic security. The authors show the joint distribution $D_0 = \{(a, b, d, F)\}$ produced by $\mathsf{Setup}$, $\mathsf{KeyGen}$, $\mathsf{UniEnc}(\mu, z)$ is computationally indistinguishable from uniform over $\mathbb{T}^{d \times 5}$ for any $\mu$ under RLWE with parameter $(N, \psi, \alpha)$, via a two-step hybrid: first the RLWE assumption on secret $z$ makes $(b, a)$ and $F = [f_0 \mid f_1]$ uniform (giving distribution $D_1$); then the RLWE assumption on secret $r$ makes $d$ uniform (giving $D_2$), which is independent of $\mu$.

Hybrid product, Method 1: improved ciphertext extension

The first hybrid-product algorithm follows the classical GSW ciphertext-extension idea (Clear and McGoldrick, simplified by Mukherjee and Wichs, EUROCRYPT 2016) but improves it. It works in two phases. First it extends a uni-encryption $(d_i, F_i)$ of $\mu_i$, together with all parties' public keys $\{b_j\}$, into a full multi-key RGSW ciphertext $\overline{D}_i \in \mathbb{T}^{d(k+1) \times (k+1)}$ that encrypts the same message under the concatenated secret $z$, satisfying $\overline{D}_i \cdot z \approx \mu_i \cdot G_{k+1} z$. Then it applies the multi-key external product

$$c' = G_{k+1}^{-1}(c) \cdot \overline{D}_i \pmod 1,$$

the natural generalization of TFHE's external product (Equation 5), yielding a multi-key RLWE ciphertext with $\langle c', z \rangle \approx \mu_i \cdot \langle c, z \rangle$. The extended matrix $\overline{D}_i$ is shown in Figure 1.

The extension uses the conventions $z_0 = 1$ and $b_0 = -a$. For each $0 \le j \le k$ it gadget-decomposes the public key $b_j$ against $F_i$: with $M_j$ the matrix whose $\ell$-th row is $g^{-1}(b_j[\ell])$, it sets $[x_j \mid y_j] = M_j F_i \in \mathbb{T}^{d \times 2}$, i.e. $x_j[\ell] = \langle g^{-1}(b_j[\ell]), f_0 \rangle$ and $y_j[\ell] = \langle g^{-1}(b_j[\ell]), f_1 \rangle$. Correctness rests on the pair of relations

$$x_j + z_i \cdot y_j \approx r_i \cdot b_j, \qquad z_j \cdot d_i \approx -r_i \cdot b_j + \mu_i z_j \cdot g \pmod 1,$$

which add to $\mu_i z_j \cdot g$, exactly the $j$-th row of a valid multi-key RGSW encryption: $\overline{D}_i \cdot z \approx \mu_i \cdot G_{k+1} z$ (Equation 6). Two improvements over prior work: the extended dimension is cut from $2k$ down to $k+1$ by merging the duplicated $(1, z_i)$ components into the single concatenated key $z = (1, z_1, \dots, z_k)$; and the symmetric uni-encryption shrinks the keys further. Extension costs $2(k+1)d^2 = O(kd^2)$ polynomial multiplications and the sparse $\overline{D}_i$ stores in $(2k+3)d = O(kd)$ polynomials; the external product is $(3k+1)d = O(kd)$ multiplications. Crucially, $\overline{D}_i$ depends only on the keys, so it can be pre-computed once and reused as a shared bootstrapping key for any gate on the same set of parties, at the price of $O(k^2 nd)$ storage.

The two steps of Method 1 in pseudocode:

RLWE.Prod -- Method 1 (extend, then external product)

Step 1. Extend((d_i, F_i = [f_0|f_1]), {b_j}_{j in [k]}):
  set z_0 = 1,  b_0 = -a       // conventions
  for j = 0 .. k:
      M_j = matrix whose l-th row is g^{-1}(b_j[l])     // l in [d]
      [x_j | y_j] = M_j . F_i                            // in T^{d x 2}
  return the (sparse) extended ciphertext  Dbar_i  in  T^{d(k+1) x (k+1)}:
     row 0:  (d_i + x_0)  on col 0,  y_0 on col i,  0 elsewhere
     row j:       x_j     on col 0,  d_i on col j,  y_j on col i  (j != i)
     row i:       x_i     on col 0,  (d_i + y_i) on col i
  // each block-row carries d_i on a near-diagonal position plus the
  // x_j, y_j terms in the 0th and i-th block columns (see Figure 1).

Step 2. Multi-key external product:
  c' = G_{k+1}^{-1}(c) . Dbar_i   (mod 1)
  // gives <c', z> approx mu_i . <c, z>

Figure 1 (page 11 of the paper): the extended multi-key RGSW ciphertext $\overline{D}_i \in \mathbb{T}^{d(k+1) \times (k+1)}$ produced by the Extend step. It is sparse, each block-row carrying $d_i$ on a near-diagonal position plus the $x_j, y_j$ terms in the 0th and $i$-th block columns, which is what makes the external product cheap.

Hybrid product, Method 2: acting directly on the ciphertext

The second algorithm is a genuinely new approach and is the one used in the implementation. Instead of first building a multi-key RGSW ciphertext independently of the input, it operates directly on the multi-key RLWE ciphertext $c = (c_0, \dots, c_k)$, never materializing any multi-key GSW object. With conventions $z_0 = 1$, $b_0 = -a$, for each $0 \le j \le k$ it gadget-decomposes $c_j$ and forms $u_j = \langle g^{-1}(c_j), d_i \rangle$ and $v_j = \langle g^{-1}(c_j), b_j \rangle$, then $w_{j,0} = \langle g^{-1}(v_j), f_{i,0} \rangle$ and $w_{j,1} = \langle g^{-1}(v_j), f_{i,1} \rangle$. The output absorbs the $w$-sums into the 0th and $i$-th components:

RLWE.Prod -- Method 2 (act directly on c)

Input: c = (c_0, c_1, ..., c_k) in T^{k+1};  uni-enc (d_i, F_i=[f_0|f_1]);
       public keys {b_j};  conventions z_0 = 1, b_0 = -a.

for j = 0 .. k:
    u_j     = < g^{-1}(c_j), d_i >
    v_j     = < g^{-1}(c_j), b_j >
    w_{j,0} = < g^{-1}(v_j), f_0 >
    w_{j,1} = < g^{-1}(v_j), f_1 >

c'_0 = u_0 + sum_{j=0..k} w_{j,0}     (mod 1)
c'_i = u_i + sum_{j=0..k} w_{j,1}     (mod 1)
c'_j = u_j                            for j in [k]\{i}
return c' = (c'_0, ..., c'_k)
// gives <c', z> approx mu_i . <c, z>

Correctness: $\langle c', z \rangle = \sum_{j=0}^{k} u_j z_j + \sum_{j=0}^{k}(w_{j,0} + w_{j,1} z_i)$. The first sum equals $\mu_i \cdot \langle c, z \rangle - r_i \sum_j v_j$ (plus error), while the second equals $\sum_j \langle g^{-1}(v_j), f_{i,0} + z_i f_{i,1} \rangle \approx r_i \sum_j v_j$, so the $r_i \sum_j v_j$ terms cancel and $\langle c', z \rangle \approx \mu_i \cdot \langle c, z \rangle$.

Two advantages over Method 1, both important in practice:

  • Lower complexity: only $4(k+1)d = O(kd)$ polynomial multiplications, giving asymptotic cost $O(kd \cdot N \log N)$ versus $O(kd^2 \cdot N \log N)$ for the extension in Method 1.
  • Less noise: the output noise variance is

$$V_2 \approx \tfrac{1}{2}(kd + k + 1) \cdot N^2 \cdot V_B \cdot \beta^2,$$

which is smaller than Method 1's variance $V_1 \approx (k+1)d^2 \cdot N^2 \cdot V_B^2 \cdot \beta^2$ by roughly a factor of $O(d \cdot V_B)$, where $V_B \approx B^2/12$ (Equation 7). The paper's analogy is the GSW fact that $(C_1 \boxtimes C_2) \boxtimes C_3$ and $C_1 \boxtimes (C_2 \boxtimes C_3)$ compute the same product but with very different error.

The trade-off: Method 1's extended ciphertext is reusable (pre-processable), and its Step-2 external product alone is about $3/4$ the cost of Method 2; so with heavy reuse Method 1 can win on per-gate time at the cost of quadratic-in-$k$ storage. But Method 2's smaller noise lets one pick smaller parameters, so the authors judge it more practical overall.

The CMux (controlled selector) gate is the direct application: $\mathrm{CMux}(c_0, c_1, (d_i, F_i), \{b_j\}) = c_0 + \mathrm{Prod}(c_1 - c_0, (d_i, F_i), \{b_j\})$, selecting $c_\mu = (1 - \mu_i) \cdot c_0 + \mu_i \cdot c_1$ between two multi-key ciphertexts under an encrypted bit $\mu_i \in \{0,1\}$. CMux is the workhorse inside the multi-key blind rotation.

Multi-key gate bootstrapping (the NAND pipeline)

The full scheme $\mathsf{MKHE}$ wires LWE and RGSW together. $\mathsf{Setup}$ runs $\mathsf{LWE.Setup}$ to fix $(n, \chi, \alpha, B', d')$ and $\mathsf{RLWE.Setup}$ to fix $(N, \psi, \beta, B, d, a)$. Per-party $\mathsf{KeyGen}$ samples $s_i \leftarrow \mathsf{LWE.KeyGen}$ and $(z_i, b_i) \leftarrow \mathsf{RLWE.KeyGen}$, writes the permuted vector $z_i^* = (z_{i,0}, -z_{i,N-1}, \dots, -z_{i,1})$, generates the bootstrapping key $\mathrm{BK}_i = \{(d_{i,j}, F_{i,j}) \leftarrow \mathsf{UniEnc}(s_{i,j}, z_i)\}_{j \in [n]}$ (a uni-encryption of each LWE secret bit), the key-switching key $\mathrm{KS}_i \leftarrow \mathsf{LWE.KSGen}(z_i^*, s_i)$, and publishes $(\mathrm{PK}_i = b_i, \mathrm{BK}_i, \mathrm{KS}_i)$. A useful identity: for $a = \sum a_\ell X^\ell$, the constant term of $a \cdot z$ equals $\langle a, z^* \rangle \bmod 1$.

$\mathsf{MKHE.NAND}$ runs in three phases, mirroring TFHE but over the concatenated keys. Phase 1 (linear combination): extend $\mathrm{ct}_1, \mathrm{ct}_2$ to share the concatenated secret $s$ by rearranging blocks and zero-filling, then form

$$\mathrm{ct}' = (\tfrac{5}{8}, 0, \dots, 0) - \mathrm{ct}'_1 - \mathrm{ct}'_2 \pmod 1,$$

which encodes $m = m_1 \,\overline{\wedge}\, m_2$ with $\langle \mathrm{ct}', (1, s) \rangle = \tfrac{1}{2}m + e' \pmod 1$ where $e' = \pm\tfrac{1}{8} - e_1 - e_2$ is bounded by $\tfrac{1}{4}$ when $|e_i| \le \tfrac{1}{16}$ (Equation 8). Phase 2 (homomorphic accumulator / blind rotation): quantize to $\tilde{b} = \lfloor 2N b' \rceil$, $\tilde{a}_i = \lfloor 2N a'_i \rceil$ and initialize the multi-key RLWE accumulator $c = (-\tfrac{1}{8} h(X) X^{\tilde{b}}, 0) \in \mathcal{T}^{k+1}$ with the fixed test polynomial $h(X) = \sum_{-N/2 \lt j \lt N/2} X^j = 1 + X + \cdots + X^{N/2-1} - X^{N/2+1} - \cdots - X^{N-1}$, over the concatenated RLWE secret $z = (1, z_1, \dots, z_k)$. Then recursively apply CMux for every party $i$ and bit $j$, using the uni-encryptions of the secret bits $s_{i,j}$, which blind-rotates the accumulator to satisfy

$$\langle c, z \rangle \approx -\tfrac{1}{8} h(X) \cdot X^{\,\tilde{b} + \sum_{i=1}^{k} \langle \tilde{a}_i, s_i \rangle} \pmod 1.$$

Since $\tilde{b} + \sum_i \langle \tilde{a}_i, s_i \rangle \approx 2N \cdot \langle \mathrm{ct}', (1,s) \rangle \approx N \cdot m \pmod{2N}$, the constant term of $\langle c, z \rangle$ becomes $\approx \tfrac{1}{4}m - \tfrac{1}{8}$; adding $\tfrac{1}{8}$ cancels the offset (Equation 9). This is exactly where the hybrid product is invoked, $k \cdot n$ times, and it dominates the whole pipeline's cost. Phase 3 (sample extraction and multi-key switching): extract an LWE ciphertext from the accumulator (the constant term of $c_i \cdot z_i$ equals $\langle a^*_i, z^*_i \rangle$ for the permuted coefficient vector $z^*_i$), giving $\mathrm{ct}^*$ with $\langle \mathrm{ct}^*, (1, z^*) \rangle \approx \tfrac{1}{4}m$; then run multi-key switching back to $s$, giving $\langle \mathrm{ct}'', (1, s) \rangle \approx \tfrac{1}{4}m$. The full procedure verbatim:

MKHE.NAND(ct_1, ct_2, {(PK_i, BK_i, KS_i)}_{i in [k]}):
  // PK_i = b_i,  BK_i = {(d_{i,j}, F_{i,j})}_{j in [n]},  KS_i = key-switch key

  // ---- Phase 1: linear combination (NAND) ----
  1-1. Extend ct_1, ct_2 to ct'_1, ct'_2 in T^{kn+1} under s=(s_1,...,s_k)
       by rearranging entries and putting zeros in the empty slots.
  1-2. ct' = (5/8, 0, ..., 0) - ct'_1 - ct'_2   (mod 1)
       // <ct',(1,s)> approx (1/2)m + e',  m = m_1 NAND m_2,  |e'| <= 1/4

  // ---- Phase 2: homomorphic accumulator (blind rotation) ----
  2-1. ct' = (b', a'_1, ..., a'_k);  bt = round(2N.b');  at_i = round(2N.a'_i)
       h(X) = sum_{-N/2 < j < N/2} X^j
       init  c = ( -(1/8) h(X) . X^{bt}, 0 )  in  T^{k+1}
  2-2. for i in [k], for j in [n]:           // at_i = (at_{i,j})_{j in [n]}
           c = RLWE.CMux( c, X^{at_{i,j}} . c, (d_{i,j}, F_{i,j}), {b_l}_{l in [k]} )
  2-3. c = (1/8, 0) + c   (mod 1)
       // <c,z> approx -(1/8) h(X) . X^{ bt + sum_i <at_i,s_i> },  rotates to N.m

  // ---- Phase 3: sample extraction + multi-key switching ----
  3-1. for c = (c_0,...,c_k):  b* = const term of c_0;  a*_i = coeff vector of c_i
       ct* = (b*, a*_1, ..., a*_k)  in  T^{kN+1}
  3-2. ct'' = LWE.MKSwitch(ct*, {KS_i}_{i in [k]})
  return ct''                          // <ct'',(1,s)> approx (1/4)m

Any binary gate (AND/OR/XOR and so on) follows by changing only the Phase-1 linear combination. The correctness conditions are: the quantized $(\tilde{b}, \tilde{a}_1, \dots, \tilde{a}_k)$ must satisfy $\tilde{b} + \sum_j \langle \tilde{a}_j, s_j \rangle = N m + \tilde{e}$ with $|\tilde{e}| \lt N/2$, where $\tilde{e} = 2N e' + e''$ collects the Phase-1 error and the rounding error $e''$; and the final output error (accumulation plus multi-key-switch error) must be small enough for decryption. The scheme inherits TFHE's circular-security assumption (each party publishes a bootstrapping key encrypting its own LWE secret bits, plus a key-switch from $z^*$ to $s$).

Variants: more gates, time-space trade-off, threshold decryption, packing

The basic scheme extends cleanly. Arbitrary bootstrapped gates (AND, OR, XOR, etc.) are obtained, as in TFHE, by modifying only the initial linear combination before bootstrapping. Time-space trade-off: Method 1's expanded bootstrapping key grows quadratically in $k$; following Brakerski-Perlman one can regenerate a temporary evaluation key in each step to get linearly-growing space, but this loses the reusability that was Method 1's only advantage over Method 2, so the authors see no motivation to adopt it.

Distributed (threshold) decryption follows because the scheme rests on standard LWE encryption, so noise smudging (noise flooding) applies directly. A simple two-step protocol: each party publishes a partial decryption, and a merge step recovers the bit. Verbatim:

MKHE.PartDec(ct, s_i):
  // ct = (b, a_1, ..., a_k) in T^{kn+1}
  sample e_i <- D_gamma          // smudging noise, medium size < 1
  return p_i = <a_i, s_i> + e_i   (mod 1)

MKHE.Merge(b, {p_j}_{j in [k]}):
  return the bit m minimizing | b + sum_{j=1..k} p_j - (1/4)m |

The smudging distribution is parametrized by $\gamma \gt 0$ and must be larger than the ciphertext error yet smaller than 1, to hide everything beyond the decrypted value. Faster LUT evaluation (vertical packing + circuit bootstrapping, or multi-value packing) carries over directly to the multi-key setting as well.

Complexity and comparison to prior MKHE

Net asymptotics: ciphertext length is $\tilde{O}(kn)$ (linear in parties), the evaluation key is $\tilde{O}(k^2 n^2)$, and a bootstrapped NAND costs $\tilde{O}(k^2 n^2)$, quadratic in the number of parties. This is the first MKHE that is simultaneously bootstrappable, compact (linear-size ciphertexts, like Brakerski-Perlman), and concretely fast. Figure 2 reproduces the paper's full comparison table.

The comparison (Figure 2): CZW17 has evaluation key and homomorphic-mult time $\tilde{O}(k^3 n)$ with $\tilde{O}(kn)$ ciphertexts (no bootstrapping); PS16 #2 has public key $\tilde{O}(kn^4)$, ciphertext $\tilde{O}(k^2 n^2)$, and homomorphic mult $\tilde{O}(k^{2.37} n^{2.37})$ (no bootstrapping); BP16 has public key $\tilde{O}(kn^3)$, linear $\tilde{O}(kn)$ ciphertexts and is bootstrappable, but only via an abstract branching program of length $L = \mathrm{poly}(k, n)$ whose concrete cost was never analyzed. This work (both methods) achieves $\tilde{O}(kn)$ ciphertexts and bootstrapped $\tilde{O}(k^2 n^2)$ homomorphic NAND, with Method 1 additionally needing an $\tilde{O}(k^2 n^2)$ evaluation key. The GSW-based schemes (CZW17, PS16, Clear-McGoldrick, Mukherjee-Wichs) all have at least quadratic-in-$k$ ciphertexts.

The improved (compact, symmetric) ciphertext-extension of Method 1 can be dropped into the GSW-based schemes (CZW17, BP16) to speed up their evaluation-key generation. Batched MKHE (e.g. Chen-Zhang-Wang's multi-key BGV with a larger plaintext space) is an orthogonal direction, but it is leveled, hides a large level-dependent constant in $\tilde{O}(\cdot)$, and its homomorphic-multiplication cost grows rapidly in $k$.

Only BP16 and this work are bootstrappable with linear-size ciphertexts; this work achieves $\tilde{O}(k^2 n^2)$ homomorphic-NAND time with $\tilde{O}(kn)$ ciphertexts.

Noise analysis (Appendix A)

Let $\epsilon^2 = 1/(12 B^{2d})$ be the variance of a uniform distribution over $(-\tfrac{1}{2}B^{-d}, \tfrac{1}{2}B^{-d}]$, and $V_B$ the mean square of a uniform distribution over $\mathbb{Z} \cap (-B/2, B/2]$, i.e. $V_B = \tfrac{1}{12}(B^2-1)$ if $B$ is odd and $V_B = \tfrac{1}{12}(B^2+2)$ if $B$ is even (so $V_B \approx B^2/12$). Analogous $\epsilon'^2$ and $V_{B'}$ are defined for the key-switching parameters $B', d'$. The analysis assumes errors behave Gaussianly and that the binary secrets give $\mathrm{Var}(r_i) = \mathrm{Var}(z_j) = 1/2$.

Method 1. Extension produces a multi-key RGSW whose per-row error has variance $V_{\exp} \le (N/2)\epsilon^2 + (1 + d V_B) N \beta^2$. The multi-key external product then yields output error variance $V_{ep} = \mu^2 \epsilon^2 (1 + kN/2) + (k+1) dN V_B \cdot \mathsf{VarErr}(D)$. Composing them, the dominant term is $V_1 \approx (k+1) d^2 N^2 V_B^2 \beta^2$.

Method 2. A direct expansion of $\langle c', z \rangle$ collects the decomposition errors $e'$ (variance $\epsilon^2(1 + kN/2)$), $e''$ (variance $\epsilon^2(k+1)$), and the cross terms, giving

$$V_2 = \mu_i^2 N\epsilon^2(1 + \tfrac{kN}{2}) + \tfrac{N^2}{2}(k+1)V_B\beta^2 + dN(1 + \tfrac{kN}{2})V_B\beta^2 + \tfrac{N}{2}\epsilon^2(k+1) + (k+1)N V_B\beta^2,$$

dominated by $V_2 \approx \tfrac{1}{2}(kd + k + 1) N^2 V_B \beta^2$, smaller than $V_1$ by about $(d V_B)^{-1}$.

Rounding error from $\tilde{b} = \lfloor 2N b' \rceil$ and $\tilde{a}_i = \lfloor 2N a'_i \rceil$ has variance $\tfrac{1}{12}(1 + kn/2)$. The CMux (Mux) gate has the same noise variance as the external product, $\mu^2 \epsilon^2(1 + kN/2) + (k+1)dN V_B \cdot \mathsf{VarErr}(C)$. The bootstrapping keys each have noise variance $\mathsf{VarErr}(C_{i,\ell}) = (N/2)\epsilon^2 + (1 + N + dN V_B)\beta^2$. Running the Mux gate $k n$ times, the accumulation error variance is

$$\tfrac{1}{2}kn\,\epsilon^2\!\left(1 + \tfrac{kN}{2}\right) + (k+1)kdnN \cdot V_B \cdot \left(\tfrac{N}{2}\epsilon^2 + (1 + N + dN V_B)\beta^2\right)$$

(Equation 10). The standard multi-key-switching error has variance $kN(\tfrac{1}{2}\epsilon'^2 + d' V_{B'}\alpha^2)$. A modified key-switching variant, which publishes LWE encryptions of $a \cdot B'^{\ell} \cdot t_{i,j}$ for $a \in \mathbb{Z}_{B'}$ so that switching becomes a pure summation of LWE vectors, attains the smaller variance

$$kN\!\left(\tfrac{1}{2}\epsilon_K^2 + d'\alpha^2\right)$$

(Equation 11), where $\epsilon_K^2$ is its decomposition-error variance. The total bootstrapping noise is simply the sum of the accumulation variance (Equation 10) and the (modified) key-switching variance (Equation 11). Because this standard deviation grows only linearly in $k$, parameter growth with the maximum party count is very slow and is controlled by adjusting the decomposition degree, which does not affect security.

Parameters, security, and experimental results

The proof-of-concept (a few days of coding) sits on top of the TFHE library and is open source at github.com/ilachill/MK-TFHE. It uses Method 2 for the hybrid product and binary LWE/RLWE secret distributions $\chi, \psi$. Three parameter sets share LWE dimension $n = 560$ ($\alpha = 3.05 \cdot 10^{-5}$, $B' = 2^2$, $d' = 8$) and RLWE dimension $N = 1024$ ($\beta = 3.72 \cdot 10^{-9}$); they differ only in the RGSW decomposition base/degree ($B = 2^9 / 2^8 / 2^6$, $d = 3 / 4 / 5$) so that larger sets tolerate more parties. All achieve at least 110-bit security per the LWE Estimator. The implementation also adopts the FHEW/TFHE space-time trade-off for key-switching (publishing all LWE encryptions of $a \cdot B'^i \cdot t_j$), so multi-key switching reduces to a summation of LWE vectors with no change in asymptotic complexity. The three sets are listed in Figure 3.

Sets I / II / III support up to $2 / 4 / 8$ parties respectively. Per-party local cost is independent of $k$: key generation 1.1 to 1.3 s, bootstrapping key 0.62 to 1.03 MB, key-switching key 70.1 MB, and a single-party ciphertext 2.19 KB. The global bootstrapped-NAND timings (single thread): $k=2$ in 0.27 s, $k=4$ in 1.45 s, $k=8$ in 7.16 s, the quadratic-in-$k$ growth visible. The merged multi-key ciphertext grows linearly: 4.38 KB ($k=2$), 8.77 KB ($k=4$), 17.32 KB ($k=8$). Figure 4 collects the end-to-end numbers. A smaller parameter set is faster, but a larger one is more flexible because more parties can join dynamically; the authors note the code has room for optimization.

Takeaways, limitations, and open directions

The work delivers the first concrete demonstration that multi-key FHE can be run at all: a bootstrapped binary gate across up to eight parties in seconds on a laptop, with ciphertexts that grow only linearly in the number of parties. Its core technical contribution, the hybrid product (especially Method 2 acting directly on the multi-key ciphertext), cleanly extends TFHE's external-product-based gate bootstrapping to the multi-key world while improving storage, time, and noise simultaneously.

Limitations. Cost still grows quadratically in the number of parties $k$, so the scheme is practical only for a modest number of participants (the experiments stop at 8). It is single-hop for keys in the sense that a bootstrapping key must be precomputed for the fixed set of parties in a computation. It inherits TFHE's circular-security assumption, and it depends on the CRS model: the shared public vector $a$ underlies the uni-encryption security proof. Gate-by-gate bootstrapping over the torus also caps the plaintext space at a single bit per ciphertext (absent the packing extensions).

Stated open directions: porting TFHE's advanced functionalities (circuit bootstrapping, packing) to the multi-key setting; building practical MKHE from amortized schemes such as BFV or CKKS; and the headline open problem of constructing MKHE without the CRS model.

Algorithms

Self-contained at-a-glance listings of every algorithm the body references. Notation: torus $\mathbb{T} = \mathbb{R} \bmod 1$, ring $R = \mathbb{Z}[X]/(X^N+1)$, gadget $g = (B^{-1}, \dots, B^{-d})$, key-switch gadget $g' = (B'^{-1}, \dots, B'^{-d'})$, $G_k = I_k \otimes g$, concatenated RLWE secret $z = (1, z_1, \dots, z_k)$, conventions $z_0 = 1$ and $b_0 = -a$.

RLWE.UniEnc(mu, z)               // single-key symmetric encryption (CRS = a)
  Input : plaintext mu in R, secret z (with z=(1,z)), CRS a in T^d
  Output: uni-encryption (d, F=[f_0|f_1]) in T^d x T^{d x 2}
  1. r <- psi ;  e_1 <- D_alpha^d
  2. d = r.a + mu.g + e_1                      (mod 1)
  3. f_1 <- U(T^d) ;  e_2 <- D_alpha^d
  4. f_0 = -z.f_1 + r.g + e_2                   (mod 1)
  5. return (d, [f_0|f_1])

RLWE.Extend((d_i, F_i=[f_0|f_1]), {b_j}_{j in [k]})   // Method 1, Step 1
  Output: multi-key RGSW Dbar_i in T^{d(k+1) x (k+1)} with Dbar_i.z ~ mu_i.G_{k+1}.z
  1. set z_0 = 1, b_0 = -a
  2. for j = 0 .. k:
  3.     M_j = matrix whose l-th row (l in [d]) is g^{-1}(b_j[l])
  4.     [x_j | y_j] = M_j . F_i                // in T^{d x 2}
  5. assemble sparse Dbar_i (block rows 0..k):
         row 0:  col0 = d_i + x_0,  col i = y_0
         row j (j != 0,i):  col0 = x_j,  col j = d_i,  col i = y_j
         row i:  col0 = x_i,  col i = d_i + y_i
  6. return Dbar_i

RLWE.Prod  (Method 1)            // extend then multi-key external product
  Input : c in T^{k+1}, uni-enc (d_i,F_i), public keys {b_j}
  1. Dbar_i = RLWE.Extend((d_i,F_i), {b_j})      // reusable, can precompute
  2. c' = G_{k+1}^{-1}(c) . Dbar_i               (mod 1)
  3. return c'                                   // <c',z> ~ mu_i.<c,z>

RLWE.Prod  (Method 2)            // act directly on c (used in implementation)
  Input : c = (c_0,...,c_k) in T^{k+1}, uni-enc (d_i,F_i=[f_0|f_1]), {b_j}
  1. set z_0 = 1, b_0 = -a
  2. for j = 0 .. k:
  3.     u_j     = < g^{-1}(c_j), d_i >
  4.     v_j     = < g^{-1}(c_j), b_j >
  5.     w_{j,0} = < g^{-1}(v_j), f_0 >
  6.     w_{j,1} = < g^{-1}(v_j), f_1 >
  7. c'_0 = u_0 + sum_{j=0..k} w_{j,0}           (mod 1)
  8. c'_i = u_i + sum_{j=0..k} w_{j,1}           (mod 1)
  9. c'_j = u_j  for j in [k]\{i}
 10. return c' = (c'_0,...,c'_k)                 // <c',z> ~ mu_i.<c,z>

RLWE.CMux(c_0, c_1, (d_i,F_i), {b_j})            // controlled selector
  return c_0 + RLWE.Prod(c_1 - c_0, (d_i,F_i), {b_j})
  // selects c_mu = (1-mu_i).c_0 + mu_i.c_1 under encrypted bit mu_i in {0,1}

LWE.MKSwitch(ct, {KS_i}_{i in [k]})              // multi-key key-switching
  Input : ct = (b, a_1,...,a_k) in T^{kN+1}, KS_i = {K_{i,j}}_{j in [N]}
  1. for i in [k]:
  2.     (b'_i, a'_i) = sum_{j=1..N} g'^{-1}(a_{i,j}) . K_{i,j}    (mod 1)
  3. b' = b + sum_{i=1..k} b'_i                  (mod 1)
  4. return ct' = (b', a'_1,...,a'_k) in T^{kn+1}

MKHE.KeyGen()  (party i)
  1. s_i <- LWE.KeyGen()                          // binary in {0,1}^n
  2. (z_i, b_i) <- RLWE.KeyGen() ; PK_i = b_i
  3. z*_i = (z_{i,0}, -z_{i,N-1}, ..., -z_{i,1})
  4. for j in [n]: (d_{i,j}, F_{i,j}) = RLWE.UniEnc(s_{i,j}, z_i)
     BK_i = {(d_{i,j}, F_{i,j})}_{j in [n]}
  5. KS_i = LWE.KSGen(z*_i, s_i)
  6. keep s_i ; publish (PK_i, BK_i, KS_i)

MKHE.NAND(ct_1, ct_2, {(PK_i, BK_i, KS_i)}_{i in [k]})
  // Phase 1: linear combination
  1. extend ct_1, ct_2 to ct'_1, ct'_2 in T^{kn+1} under s=(s_1,...,s_k)
  2. ct' = (5/8, 0,...,0) - ct'_1 - ct'_2        (mod 1)
  // Phase 2: homomorphic accumulator (blind rotation)
  3. (b', a'_1,...,a'_k) = ct' ;  bt = round(2N.b') ;  at_i = round(2N.a'_i)
  4. h(X) = sum_{-N/2 < j < N/2} X^j
  5. c = ( -(1/8) h(X) . X^{bt}, 0 ) in T^{k+1}
  6. for i in [k], for j in [n]:
  7.     c = RLWE.CMux( c, X^{at_{i,j}}.c, (d_{i,j},F_{i,j}), {b_l}_{l in [k]} )
  8. c = (1/8, 0) + c                            (mod 1)
  // Phase 3: sample extraction + multi-key switching
  9. b* = const term of c_0 ; a*_i = coeff vector of c_i ; ct* = (b*,a*_1,...,a*_k)
 10. ct'' = LWE.MKSwitch(ct*, {KS_i}_{i in [k]})
 11. return ct''                                 // <ct'',(1,s)> ~ (1/4)(m_1 NAND m_2)

MKHE.PartDec / MKHE.Merge        // threshold (distributed) decryption
  PartDec(ct=(b,a_1,...,a_k), s_i):
    e_i <- D_gamma                               // smudging noise
    return p_i = <a_i, s_i> + e_i                 (mod 1)
  Merge(b, {p_j}_{j in [k]}):
    return bit m minimizing | b + sum_j p_j - (1/4)m |

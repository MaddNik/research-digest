Hardware Patterns for FHE Acceleration

Research Tracker - background note

Why FHE needs special hardware, and what this note covers

Fully Homomorphic Encryption (FHE) lets a server compute on ciphertexts without decrypting them. The price is that every plaintext number becomes a high-degree polynomial with large coefficients, and the central recurring operation - bootstrapping - chains thousands of polynomial multiplications and key-dependent products together. A single CKKS ciphertext at a bootstrappable parameter set can reach tens of megabytes, and the evaluation (key-switching / bootstrapping) keys reach hundreds of megabytes to gigabytes. The arithmetic is almost entirely integer (or fixed-point) modular arithmetic over polynomial rings, so the floating-point throughput of CPUs and GPUs is largely wasted, and the working set is far too big to keep in registers or caches.

The result is that FHE accelerators are dominated by three concrete engineering patterns, and this note teaches each one from first principles and then ties it to the accelerators summarized elsewhere on this site - HEAX (CKKS on FPGA), FAB (bootstrappable CKKS on FPGA), FPT (TFHE PBS on FPGA), MATCHA (TFHE ASIC), and Strix (TFHE ASIC):

  • Systolic arrays for the polynomial and matrix arithmetic that makes up gadget decomposition, the external product, and key switching.
  • Pipelined NTT / FFT butterfly units and the on-chip handling of twiddle factors (the roots of unity those butterflies multiply by).
  • The HBM / memory-bandwidth bottleneck: large ciphertexts and especially the evaluation keys do not fit on chip, so the design point is almost always set by how fast keys can stream in from off-chip memory.

A useful framing, made explicit by FAB, is that the winning design is usually not more compute but a balanced design that is not memory-bound: a modest number of functional units, careful use of the limited on-chip SRAM, and scheduling that hides off-chip key traffic behind computation.

The algebraic objects the hardware operates on

Both major FHE families work in a polynomial ring. The power-of-two cyclotomic ring is

$$R_q = \mathbb{Z}_q[X]/(X^N+1),$$

where $N$ is a power of two (typically $N=2^{10}$ to $2^{16}$) and $q$ is the coefficient modulus. The defining relation $X^N \equiv -1$ makes the ring negacyclic: multiplying a polynomial by $X$ rotates its coefficients and negates the one that wraps around. This is Equation 1 and it is the reason FHE multiplication is not an ordinary convolution.

$$a(X)\cdot b(X) \bmod (X^N+1) \;=\; \sum_{k=0}^{N-1}\Big(\sum_{i+j=k} a_i b_j \;-\!\!\sum_{i+j=k+N} a_i b_j\Big) X^k.$$

A schoolbook product of two degree-$(N-1)$ polynomials costs $O(N^2)$ coefficient multiplications, which is the cost the transform-based engines (next section) are built to avoid. To keep the modular multiplications small and parallel, every accelerator splits the big modulus $q$ into machine-word-sized coprime primes using the Residue Number System (RNS / CRT). With $q=\prod_{i=1}^{L} q_i$, the Chinese Remainder Theorem gives a ring isomorphism (Equation 2):

$$R_q \;\cong\; R_{q_1}\times R_{q_2}\times\cdots\times R_{q_L}.$$

Each limb (residue) $x \bmod q_i$ is processed independently, which is what lets HEAX exploit "per-component" parallelism and FAB issue many narrow modular operations in parallel. The reverse direction, recombining limbs into a residue for a new modulus $p$, is base conversion (Equation 3), used inside key switching as ModUp / ModDown:

$$[x]_p \;=\; \sum_{i=1}^{L}\big[\, x_i \cdot \tilde{Q}_i \,\big]_{q_i}\cdot Q_i^{*} \pmod p, \qquad Q_i^{*}=\frac{Q}{q_i},\quad \tilde{Q}_i = (Q_i^{*})^{-1} \bmod q_i.$$

In TFHE the same ring appears but coefficients live on the discrete torus $\mathbb{T} = \mathbb{R}/\mathbb{Z}$ represented as fixed-point integers, so TFHE engines (FPT, MATCHA, Strix) use a complex FFT rather than an integer NTT, but the butterfly hardware is structurally the same.

Pattern A - Systolic arrays for polynomial and matrix arithmetic

A systolic array is a grid of small identical processing elements (PEs), each doing one multiply-accumulate (MAC) per cycle, where data is pumped rhythmically through the array: a value enters at an edge, is used by a PE, and is passed to its neighbor on the next cycle. No PE reads a global memory or register file during the steady state - every operand arrives from a neighbor. This converts an $O(N^2)$ or $O(N^3)$ computation with a huge memory footprint into a high-throughput stream with only local, nearest-neighbor wiring, which is exactly the property you want when the bottleneck is data movement, not arithmetic.

The canonical example is multiplying an $m\times n$ matrix by an $n\times p$ matrix on an $m\times p$ grid of MAC cells. Element $C_{ij}$ accumulates inside PE$(i,j)$ as rows of $A$ stream in from the left and columns of $B$ stream in from the top, perfectly skewed in time so that the right $a_{ik}$ and $b_{kj}$ meet at PE$(i,j)$ on cycle $i+j+k$. The recurrence each cell runs is Equation 4:

$$C_{ij} \;\leftarrow\; C_{ij} + a_{ik}\,b_{kj}, \qquad k = 0,1,\ldots,n-1.$$

Why this matters for FHE: the two heaviest non-transform operations are matrix-vector / matrix-matrix in shape. Key switching multiplies a decomposed ciphertext by a key-switching matrix; the TFHE external product multiplies a decomposed GLWE vector by a GGSW matrix. Both are inner products of many polynomial limbs accumulated together, and both map cleanly onto a systolic MAC fabric where decomposed limbs stream in one axis and key coefficients stream in the other.

The same dataflow discipline drives the transform-domain pointwise step. In the FFT/NTT domain a polynomial product is a coefficient-wise multiply-accumulate, so the "VMA" (vector multiply-accumulate) stage in Strix and the MAC stage in FPT are systolic-style streaming MAC arrays sitting between the forward and inverse transforms. The general lesson the FHE accelerators take from systolic design is the streaming, no-stall philosophy:

  • FPT is built as a fixed cascade of throughput-balanced stages (external product, forward FFT, inverse FFT, twisting, MAC), each consuming the same streaming width of coefficients every cycle, with static latency and minimal control logic - a streaming-DSP style rather than a register-file-plus-PEs style. The whole CMux is one long systolic-like pipeline.
  • Strix's Homomorphic Streaming Core is six fully-pipelined stages (Rotator, Decomposer, FFT, VMA, IFFT, Accumulator); one traversal equals one blind-rotation iteration and ciphertexts stream through with no bubbles - a textbook systolic steady state.
  • MATCHA's external-product (EP) cores and TGSW clusters run as a two-stage producer/consumer pipeline (the TGSW cluster builds the key bundle one step ahead of the EP core consuming it), again a systolic producer-consumer pattern with separate register banks to avoid memory conflicts.
  • HEAX's KeySwitch module is "fully end-to-end pipelined", processing many key-switching operations simultaneously without excessive FIFOs - the same idea applied to the RNS key-switch inner products.

The pseudocode below shows the output-stationary systolic MAC dataflow that all of these specialize.

SystolicMAC(A[m][n], B[n][p]):                # output-stationary array
  # grid of m*p PEs; PE(i,j) holds accumulator acc[i][j]
  for all i,j in parallel: acc[i][j] = 0
  for step = 0 to (m + n + p):               # rhythmic clock
    for each PE(i,j) in parallel:
      a_in = value arriving from west neighbor   # a[i][k] skewed by i
      b_in = value arriving from north neighbor  # b[k][j] skewed by j
      acc[i][j] = acc[i][j] + a_in * b_in        # one MAC per cycle
      pass a_in to east neighbor PE(i, j+1)
      pass b_in to south neighbor PE(i+1, j)
  return acc                                  # C = A * B

In FHE the multiply on each PE is a modular multiply (Pattern discussion below) for NTT-domain work, or a fixed-point complex multiply for FFT-domain TFHE work.

The modular multiplier each PE contains

Whatever the dataflow, the atomic operation is a modular multiply $c = a\cdot b \bmod q_i$. A hardware divide is far too expensive, so accelerators use a reduction that replaces division by precomputed multiplications and shifts. Barrett reduction precomputes $\mu = \lfloor 2^{2w}/q\rfloor$ for word width $w$ and computes Equation 5:

$$r \;=\; a\,b \;-\; \Big\lfloor \frac{(a\,b)\,\mu}{2^{2w}}\Big\rfloor\, q, \qquad \text{then } r \mathrel{-}= q \text{ if } r \ge q.$$

Montgomery reduction is the common alternative (it keeps operands in a residue form and avoids a final compare in the inner loop). HEAX uses a fused multiply-then-reduce "MulRed" core; FAB splits each $54$-bit operation into $27$-bit additions and $18$-bit multiplies to match the FPGA DSP block, loop-unrolling so a multiply finishes in $12$ cycles instead of $21$ while keeping every DSP pipeline register full. The point for hardware is that the reduction is itself deeply pipelined (HEAX reports an NTT core $50$ stages deep), so although a single modular multiply has high latency, the array completes one every cycle.

BarrettModMul(a, b, q, mu, w):       # mu = floor(2^(2w)/q) precomputed
  t   = a * b                        # 2w-bit product
  est = (t * mu) >> (2*w)            # estimated quotient
  r   = t - est * q
  if r >= q: r = r - q               # at most one correction
  return r

Pattern B - The NTT/FFT and the butterfly

To avoid the $O(N^2)$ schoolbook product (Equation 1), accelerators transform each polynomial, multiply pointwise, and transform back. Over $\mathbb{Z}_q$ this is the Number Theoretic Transform (NTT), the FFT done in a finite field using a primitive $N$-th root of unity $\omega$ with $\omega^N \equiv 1 \pmod q$. The forward transform is Equation 6:

$$\hat{a}_k \;=\; \sum_{j=0}^{N-1} a_j\,\omega^{jk} \bmod q, \qquad k = 0,\ldots,N-1.$$

and the convolution theorem turns the ring product into a coefficient-wise product (Equation 7):

$$a\cdot b \;=\; \mathrm{NTT}^{-1}\big(\,\mathrm{NTT}(a)\odot \mathrm{NTT}(b)\,\big),$$

which costs $O(N \log N)$ instead of $O(N^2)$. For the negacyclic ring $\mathbb{Z}_q[X]/(X^N+1)$ one folds in a $2N$-th root $\psi$ with $\psi^2=\omega$: pre-weighting $a_j$ by $\psi^{j}$ (a "twist") and post-weighting by $\psi^{-k}$ makes the cyclic NTT compute the negacyclic product directly, absorbing the $X^N=-1$ wraparound for free. TFHE engines instead run a complex FFT with the twisting Equation 8:

$$\hat{a} \;=\; \big(a_0,\; \psi\,a_1,\; \psi^{2} a_2,\; \ldots,\; \psi^{N-1} a_{N-1}\big)$$

then a real-to-complex FFT. The Cooley-Tukey decomposition factors the transform into $\log_2 N$ stages, each made of $N/2$ butterflies. A radix-2 decimation-in-time butterfly takes two coefficients and one twiddle factor $\omega^r$ and produces two outputs (Equation 9):

$$\begin{aligned} u' &= u + \omega^r\, v,\\ v' &= u - \omega^r\, v. \end{aligned}$$

This is the single most replicated piece of arithmetic in an FHE accelerator. Each butterfly is one modular (or fixed-point complex) multiply by the twiddle plus an add and a subtract. The iterative in-place NTT is below.

IterativeNTT(a[0..N-1], psi_powers, q):     # in-place, decimation-in-time
  a = bit_reverse_permutation(a)            # reorder inputs
  len = 2
  while len <= N:                           # log2(N) stages
    half = len / 2
    for start = 0 to N-1 step len:          # blocks within a stage
      for j = 0 to half-1:                  # butterflies in a block
        w = psi_powers[ twiddle_index(len, j) ]   # root of unity
        u = a[start + j]
        v = ModMul(a[start + j + half], w, q)
        a[start + j]        = ModAdd(u, v, q)      # u + w*v
        a[start + j + half] = ModSub(u, v, q)      # u - w*v
    len = len * 2
  return a                                  # NTT(a), bit-reversed order

How the accelerators realize this:

  • HEAX builds an NTT core that accepts two coefficients, a twiddle $w$, a precomputed companion $w_p$, and the prime $p$, and emits two transformed coefficients; all internal modular ops are pipelined. To feed many butterflies per cycle, each polynomial is stored across many parallel BRAMs that share read/write addresses, exploiting that the NTT's run of consecutive accesses is always a power of two.
  • FAB wires its $256$ modular add/sub/multiply units as radix-2 butterflies processing $512$ coefficients per pass, with a dedicated address-generation unit that maps data and twiddles per stage, and folds bit-reversal into the preceding automorphism / multiply so it costs nothing extra.
  • FPT and Strix use continuous-flow pipelined FFTs: $\log_2 N$ cascaded butterfly stages separated by stride-permutation / shuffle units (Strix's SHUs) that do all reordering on chip with no matrix transpose, which is what keeps the streaming dataflow stall-free. Strix uses CLP input lanes (e.g. $2$ butterfly units per stage for CLP $=4$) and shuffle units whose delay $L$ halves down the pipeline ($L=N/8$ down to $L=1$), with large delays in SRAM shift registers and small ones in flip-flops.
  • MATCHA uses a radix-4, multiplication-free FFT: each twiddle/lifting coefficient is quantized to a dyadic value $\alpha/2^{\beta}$, so multiplying by it becomes a few shifts and adds, which lets the whole design clock at $2\,\text{GHz}$.

On-chip twiddle-factor handling

Each NTT stage needs a specific set of roots of unity, the twiddle factors $\omega^r$ (or $\psi^r$ in the negacyclic case). There are $O(N)$ distinct twiddles, they are reused every transform, and they must arrive at the right butterfly on the right cycle. Recomputing them by repeated multiplication would add a serial dependency and burn multipliers, so every accelerator precomputes and stores them on chip in dedicated, banked memories, and the design effort goes into indexing and bandwidth, not arithmetic:

  • Precompute and bit-reverse: HEAX stores powers of $\psi$ in bit-reverse order in a twiddle-factor memory, together with a precomputed Barrett companion $\mathbf{Y}' = \lfloor \mathbf{Y}\cdot 2^w / p\rfloor$ so each butterfly's modular multiply is a single MulRed.
  • Per-stage address generation: FAB has a dedicated twiddle-factor-mapping sub-unit that reads exactly the twiddles a stage needs from a URAM "miscellaneous" bank, in lockstep with the data-mapping sub-unit, so the $512$-coefficient pass never waits on a twiddle.
  • Sharing to halve reads: MATCHA's radix-4 butterfly reads a single complex root per butterfly and lets two butterflies in the same block share a twiddle factor, halving twiddle-buffer reads, and storing dyadic-quantized coefficients so the "multiply" is shifts and adds.
  • Reduced-width twiddles: FHE tolerates noise, so twiddles can be stored in fewer bits. Strix keeps a $64$-bit FFT datapath but only $16$-bit twiddle factors; FPT sets twiddle width to the intermediate width minus four bits, shrinking the twiddle ROMs and the FFT area substantially.

Because the twiddle set is small and read-only, twiddle ROMs are rarely the capacity problem - they live comfortably in BRAM/URAM or distributed ROM. The capacity and bandwidth crisis is caused by the keys, which is Pattern C.

StreamingNTTStage(in_lanes, stage_s, twiddle_rom):  # one pipelined stage
  # processes 'width' coefficients per cycle, no stall
  for each cycle, for each butterfly lane l in parallel:
    (u, v)   = read two coefficients arriving this cycle
    r        = twiddle_index(stage_s, l, cycle)      # deterministic schedule
    w        = twiddle_rom[r]                         # precomputed, on-chip
    out_a    = ModAdd(u, ModMul(v, w))                # u + w*v
    out_b    = ModSub(u, ModMul(v, w))                # u - w*v
    emit (out_a, out_b) into shuffle/permute network  # reorder for next stage

Where these operations live in (T)FHE: gadget decomposition, external product, blind rotation

To see why butterflies and systolic MACs dominate, follow the TFHE bootstrap. The core homomorphic multiply is the external product of a GGSW ciphertext (an encrypted matrix) with a GLWE ciphertext (an encrypted vector). To keep noise bounded, the GLWE operand is first split into small digits by gadget decomposition against the gadget vector $g = (B^{0}, B^{1}, \ldots, B^{\ell-1})$ for base $B$ and level $\ell$ (Equation 10):

$$a \;\approx\; \sum_{i=0}^{\ell-1} a^{(i)}\,\frac{q}{B^{\,i+1}}, \qquad \lVert a^{(i)} \rVert_\infty \le \frac{B}{2}.$$

The external product is then the inner product of the decomposed digits with the GGSW rows (Equation 11):

$$\mathrm{GGSW} \boxdot \mathrm{GLWE} \;=\; \sum_{i} a^{(i)} \cdot \mathrm{row}_i(\mathrm{GGSW}),$$

a sum of polynomial products - hence $\ell(k{+}1)$ forward NTTs/FFTs, a systolic MAC across rows, and $(k{+}1)$ inverse transforms. The decomposer that produces the $a^{(i)}$ is just bit-slicing and rounding, so MATCHA, Strix, and FPT all implement it as a multiplier-free unit (shifts, adds, sign logic).

The external product is wrapped in a controlled multiplexer, the CMux, which selects between two ciphertexts under an encrypted bit $b$ (Equation 12):

$$\mathrm{CMux}(b,\, c_0,\, c_1) \;=\; c_0 + b\boxdot(c_1 - c_0).$$

Blind rotation iterates the CMux $n$ times (once per coefficient of the input LWE mask), each step rotating an accumulator polynomial by an encrypted exponent. This loop is the bootstrap, it is where essentially all the time goes, and at every iteration it must read a fresh bootstrapping-key element $\mathrm{BK}_i$ - which is what makes the bootstrap memory-bound. The two pseudocodes:

GadgetDecompose(a, base B, level L):       # multiplier-free
  out = []
  for i = 0 to L-1:
    digit = round( a / (q / B^(i+1)) ) mod B   # shift + round
    center digit into (-B/2, B/2]              # signed digit
    out.append(digit)
  return out            # a ~= sum_i out[i] * (q / B^(i+1))

BlindRotate(ACC, LWE = (a_1..a_n, b), BK[1..n]):  # the bootstrap loop
  ACC = X^{-b} * test_polynomial            # initialize rotation
  for i = 1 to n:                           # one CMux per mask coefficient
    diff = (X^{a_i} - 1) * ACC               # rotate by encrypted exponent
    ACC  = ACC + ExternalProduct(BK[i], diff)  # CMux via external product
  return SampleExtract(ACC)                 # output refreshed LWE

The ExternalProduct call expands to the forward-FFT / systolic-MAC / inverse-FFT cascade of Patterns A and B; the BK[i] read is the off-chip traffic of Pattern C.

Pattern C - The HBM / memory-bandwidth bottleneck

The defining constraint of FHE acceleration is not arithmetic throughput, it is feeding the arithmetic. A bootstrappable CKKS ciphertext can be tens of MB (FAB's parameter set reaches $28.3$ MB per ciphertext), the live key-switching data is over $100$ MB (FAB: $\sim 112$ MB = $84$ MB keys + $28$ MB ciphertext), and TFHE bootstrapping keys are GB-scale. On-chip SRAM is fast but tiny: an FPGA like the Alveo U280 has roughly $41$ to $43$ MB of BRAM+URAM at $\sim 30\,\text{TB/s}$ internal bandwidth, while off-chip HBM2 offers GB of capacity but only $\sim 300$ to $460\,\text{GB/s}$ per stack. The arithmetic can consume key coefficients faster than HBM can deliver them. FPT makes the gap explicit: at full CMux throughput the bootstrapping key demand can exceed $1.0\,\text{TB/s}$, more than HBM provides. This is the memory wall, and the roofline picture is Equation 13:

$$\text{Throughput} \;\le\; \min\!\Big(\underbrace{\text{peak compute}}_{\text{units}\,\times\,f},\;\; \underbrace{\text{BW}_{\text{HBM}} \times \text{operational intensity}}_{\text{bytes delivered}}\Big).$$

where operational intensity is useful arithmetic per byte fetched. FHE's intensity is low (every key byte is used once per bootstrap iteration), so most designs sit on the bandwidth-bound side of the roofline unless they actively raise intensity. The standard counter-measures, all visible in the site's accelerators:

  • Keep everything on chip when you can (HEAX, FAB). HEAX exploits the RNS isomorphism (Equation 2) to compute each limb independently, eliminating off-chip access for small parameters and minimizing it for large ones; only at its largest parameter set ($N=2^{14}$) must it stream key-switching keys from DRAM, at which point throughput becomes DRAM-bandwidth-limited. FAB deliberately picks $N=2^{16}$ as the largest power of two whose limbs still fit on chip, and packs limbs so $72$-bit URAM and $18$-bit BRAM words are fully used.
  • Reuse a fetched key across a batch (FPT, Strix, MATCHA). The most effective lever is batching: process many ciphertexts that are all at the same bootstrap iteration and under the same key, so the key element $\mathrm{BK}_i$ is fetched once and amortized over the whole batch. FPT fills its deep CMux pipeline with $b$ ciphertexts (e.g. $b=12$) so it reads $\mathrm{BK}_i$ from a small BRAM ping-pong buffer at $1.0\,\text{TB/s}$ internally while HBM refills $\mathrm{BK}_{i+1}$ at only $1.0/b\,\text{TB/s}$ - turning a bandwidth-bound design into a compute-bound one. Strix generalizes this to two-level (device + core) batching: each blind-rotation iteration fetches the $\mathrm{bsk}$ portion once and multicasts it to all cores, reusing it across the whole batch, with double buffering to prefetch the next portion. Strix shows the failure mode too: pushing core-level parallelism (CLP) too high shrinks the gap between iterations, forces the $\mathrm{bsk}$ to be refetched too often, demands $\sim 1\,\text{TB/s}$, and collapses throughput.
  • Shrink the keys (FPT, MATCHA, Strix). Because FHE is noisy, keys and twiddles can be stored in reduced-precision fixed-point, cutting the bytes that must move. FPT's per-variable fixed-point analysis trims the full $\mathrm{BK}$ to $\sim 15$ MB so it fits in a mix of BRAM and URAM; Strix uses $16$-bit twiddles on a $64$-bit FFT path.
  • Reduce key traffic algorithmically (FAB). FAB's modified key-switch dataflow reorders ModUp/decompose/inner-product so fewer NTTs run and no resultant limb is written back to HBM, hiding the $\sim 300$-cycle key-read latency behind computation in dual-port BRAM banks.
  • Provision enough channels (FAB, MATCHA, Strix). FAB feeds its kernel from all $32$ AXI ports across two $4$-GB HBM2 stacks (up to $460\,\text{GB/s}$); MATCHA uses $640\,\text{GB/s}$ of HBM2; Strix splits one $300\,\text{GB/s}$ HBM2e stack into channels dedicated to $\mathrm{bsk}$, $\mathrm{ksk}$, and ciphertexts.

The unifying takeaway is FAB's thesis restated: the scarce resource is HBM bandwidth, so the architecture is designed around data movement - balanced functional units, aggressive on-chip residency, key-reuse batching, and reduced-precision keys - rather than around piling on more multipliers. FPT and Strix represent the current end state of that philosophy for TFHE: by reusing each fetched key across a batch they become the first compute-bound bootstrapping designs, escaping the wall that bounds the CKKS accelerators.

BatchedBootstrapEpoch(batch[1..b], BK[1..n], BR_iters n):
  # all b ciphertexts at the same iteration, same key -> key reuse
  load test vectors into per-core scratchpads      # one-time
  for it = 1 to n:                                  # one BR iteration per pass
    fetch BK[it] from HBM once                      # single off-chip read
    prefetch BK[it+1] into alternate buffer         # ping-pong / double buffer
    for each ciphertext c in batch in parallel:     # amortize the fetch
      c.ACC = c.ACC + ExternalProduct(BK[it], rotate(c.ACC))
  return [ SampleExtract(c.ACC) for c in batch ]
  # effective off-chip key bandwidth = (key bytes per iter) / b

Putting the three patterns together

A complete FHE accelerator is the composition of the three patterns. Data arrives over PCIe, is staged in HBM, and is streamed on chip; the NTT/FFT butterfly engine (Pattern B) transforms polynomials, reading twiddles from on-chip ROMs; the transformed limbs flow into a systolic MAC fabric (Pattern A) that performs the external product or key-switch inner products against streamed key coefficients; an inverse transform returns to the coefficient domain; and the entire schedule is arranged so the off-chip key traffic (Pattern C) is reused across a batch and hidden behind computation. The single design knob that most often decides whether the machine is compute-bound or memory-bound is the batch / reuse factor that sets the effective key bandwidth in the last pseudocode of the previous section.

Mapping the site's accelerators onto the patterns:

Accelerator | Transform        | Dataflow / MAC          | Memory strategy
------------+------------------+-------------------------+-----------------------------

            | BRAM twiddles    |                          | (then DRAM-BW-bound)
FAB  (CKKS) | 256 radix-2 BFUs | reordered KeySwitch,     | N=2^16 limbs fit 43MB on-chip;
            | 512 coef/pass    | no limb write-back       | 460 GB/s HBM2, 32 AXI ports
FPT  (TFHE) | continuous-flow  | streaming CMux cascade,  | batch b=12 reuse; ping-pong BK

            | 16-bit twiddles  |                          | first compute-bound design

Strix(TFHE) | pipelined FFT    | 6-stage HSC, VMA array,  | two-level batching; bsk fetched

            | 16-bit twiddles  |                          | HBM2e split by data type

The recurring story across all five is that arithmetic is cheap and abundant on modern FPGAs and ASICs, while moving the large ciphertexts and especially the evaluation keys is the binding constraint - so the cleverness lives in the dataflow (keep it streaming and stall-free) and in the memory schedule (keep keys on chip, reuse them across a batch, and store them in as few bits as the noise budget allows).

Algorithms

Complete "at a glance" listings of every algorithm referenced above. These are standard formulations written for clarity, not tied to any single source PDF.

Algorithm 1 - Iterative in-place NTT (Cooley-Tukey, decimation-in-time).

Input : a[0..N-1] in Z_q, N a power of two; psi_powers (precomputed 2N-th
         roots, bit-reverse order); modulus q
Output: NTT(a) (negacyclic if pre/post twisting by psi^j / psi^-k applied)
 1: a <- bit_reverse_permutation(a)
 2: len <- 2
 3: while len <= N do                       # log2(N) stages
 4:   half <- len / 2
 5:   for start <- 0 to N-1 step len do      # blocks
 6:     for j <- 0 to half-1 do             # butterflies
 7:       w <- psi_powers[ twiddle_index(len, j) ]
 8:       u <- a[start+j]
 9:       v <- ModMul(a[start+j+half], w, q)
10:       a[start+j]      <- ModAdd(u, v, q)   # u + w*v
11:       a[start+j+half] <- ModSub(u, v, q)   # u - w*v
12:   len <- len * 2
13: return a

Algorithm 2 - Barrett modular multiply (the op inside each butterfly / PE).

Input : a, b in [0,q); q; w = word width; mu = floor(2^(2w)/q) (precomputed)
Output: a*b mod q
 1: t   <- a * b                 # full 2w-bit product
 2: est <- (t * mu) >> (2*w)     # quotient estimate
 3: r   <- t - est * q
 4: if r >= q then r <- r - q    # at most one correction
 5: return r

Algorithm 3 - Output-stationary systolic matrix multiply (external product / key-switch fabric).

Input : A[m][n], B[n][p] streamed skewed into an m x p PE grid
Output: C = A*B held in PE accumulators
 1: for all PE(i,j) in parallel: acc[i][j] <- 0
 2: for step <- 0 to (m+n+p) do                 # rhythmic clock
 3:   for each PE(i,j) in parallel do
 4:     a_in <- value from west neighbor          # a[i][k] skewed
 5:     b_in <- value from north neighbor         # b[k][j] skewed
 6:     acc[i][j] <- acc[i][j] + MAC(a_in, b_in)  # modular or fixed-point
 7:     forward a_in east  to PE(i, j+1)
 8:     forward b_in south to PE(i+1, j)
 9: return acc

Algorithm 4 - Streaming pipelined NTT/FFT stage with on-chip twiddles.

Input : coefficient stream (width per cycle); stage index s; twiddle_rom (on-chip)
Output: transformed stream for the next stage, reordered by shuffle network
 1: for each cycle do
 2:   for each butterfly lane l in parallel do
 3:     (u, v) <- two coefficients arriving this cycle
 4:     r <- twiddle_index(s, l, cycle)        # deterministic schedule
 5:     w <- twiddle_rom[r]                     # precomputed, reduced width
 6:     out_a <- ModAdd(u, ModMul(v, w))        # u + w*v
 7:     out_b <- ModSub(u, ModMul(v, w))        # u - w*v
 8:     push (out_a, out_b) into shuffle/permute unit (no transpose)

Algorithm 5 - Signed gadget decomposition (multiplier-free).

Input : a in Z_q; base B; level L (gadget g = (q/B^1, ..., q/B^L))
Output: digits a^(0..L-1) with |a^(i)| <= B/2 and a ~= sum_i a^(i)*(q/B^(i+1))
 1: out <- empty
 2: for i <- 0 to L-1 do
 3:   d <- round( a / (q / B^(i+1)) ) mod B    # shift + round, no multiply
 4:   if d > B/2 then d <- d - B                # center to signed range
 5:   out.append(d)
 6: return out

Algorithm 6 - Blind rotation / CMux loop (the TFHE bootstrap core).

Input : test polynomial tv; LWE = (a_1..a_n, b); bootstrap keys BK[1..n]
Output: refreshed LWE ciphertext
 1: ACC <- X^{-b} * tv                          # initialize accumulator
 2: for i <- 1 to n do                          # one CMux per mask coeff
 3:   diff <- (X^{a_i} - 1) * ACC                # rotate by encrypted exponent
 4:   prod <- ExternalProduct(BK[i], diff)       # forward FFT -> MAC -> iFFT
 5:   ACC  <- ACC + prod                         # CMux = c0 + b box (c1-c0)
 6: return SampleExtract(ACC)

ExternalProduct(GGSW, GLWE):
 7: for each component decompose GLWE via Algorithm 5 -> digits a^(i)
 8: forward-transform each digit (Algorithm 4)
 9: systolic-MAC digits against GGSW rows (Algorithm 3)  # Equation 11
10: inverse-transform the accumulated rows -> result GLWE

Algorithm 7 - Batched bootstrap epoch with key reuse (the memory-wall counter-measure).

Input : batch[1..b] (all at same iteration, same key); BK[1..n]; iterations n
Output: refreshed batch
 1: load test vectors into per-core scratchpads     # one-time, on-chip
 2: for it <- 1 to n do                             # one BR iteration per pass
 3:   fetch BK[it] from HBM once                     # single off-chip read
 4:   prefetch BK[it+1] into alternate buffer        # double buffering
 5:   for each ciphertext c in batch in parallel do  # amortize the fetch
 6:     c.ACC <- c.ACC + ExternalProduct(BK[it], rotate(c.ACC))
 7: return [ SampleExtract(c.ACC) for c in batch ]
 8: # effective off-chip key bandwidth = (key bytes per iteration) / b

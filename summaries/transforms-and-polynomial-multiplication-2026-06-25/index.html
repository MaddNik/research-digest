Transforms and Polynomial Multiplication for FHE

Research Tracker - background note

Why polynomial multiplication dominates (T)FHE

Modern lattice-based fully homomorphic encryption (FHE) schemes - BGV, BFV, CKKS and the FHEW/TFHE family - represent plaintexts and ciphertexts as elements of a polynomial quotient ring. The two ring choices that matter in practice are the cyclic ring $\mathbb{Z}_q[X]/(X^N-1)$ and, far more commonly, the negacyclic ring $\mathbb{Z}_q[X]/(X^N+1)$ with $N$ a power of two. A ciphertext component is a degree-$(N-1)$ polynomial $a(X)=\sum_{i=0}^{N-1} a_i X^i$ with coefficients in $\mathbb{Z}_q$. Encryption, homomorphic addition and especially homomorphic multiplication reduce to arithmetic in this ring.

The expensive primitive is multiplying two such polynomials and reducing modulo $X^N+1$. Done schoolbook style, computing every product $a_i b_j$ costs $O(N^2)$ coefficient multiplications. In RLWE schemes $N$ is typically $2^{10}$ to $2^{16}$, so $N^2$ ranges from roughly $10^6$ to $4\times 10^9$ multiplications per polynomial product, and a single homomorphic operation invokes many such products across a tower of RNS limbs.

In TFHE specifically, the central operation is programmable bootstrapping, whose inner loop is a blind rotation: a sequence of $n$ controlled multiplications (CMux gates) of an accumulator polynomial in $\mathbb{Z}_q[X]/(X^N+1)$ by monomials and by gadget-decomposed key material. Profiling on CPUs shows that the forward and inverse transforms used to perform these polynomial products consume on the order of $80\%$ of bootstrapping latency, and bootstrapping itself is around $99\%$ of a gate. That is why hardware accelerators (FPGA and ASIC such as MATCHA and FPT) are built almost entirely around a fast transform datapath. The remedy is to move multiplication into a domain where it becomes pointwise. Both the Number-Theoretic Transform (NTT) and the Fast Fourier Transform (FFT) achieve this, reducing the cost from $O(N^2)$ to $O(N \log N)$.

The convolution theorem is the engine. If $\mathcal{T}$ is a transform that diagonalizes convolution, then for the linear convolution underlying polynomial multiplication

$$ a(X)\, b(X) \;=\; \mathcal{T}^{-1}\!\big( \mathcal{T}(a) \odot \mathcal{T}(b) \big), $$

where $\odot$ is the elementwise (Hadamard) product. This is Equation 1. The two transforms cost $O(N \log N)$ each, the pointwise product costs $O(N)$, and the inverse transform is $O(N \log N)$, so the whole product is $O(N \log N)$. The subtlety, addressed below, is the reduction modulo $X^N+1$, which the negacyclic variant folds directly into the transform.

The Number-Theoretic Transform (NTT) and its inverse

The NTT is the Discrete Fourier Transform carried out over the finite field $\mathbb{Z}_q$ instead of the complex numbers. The complex root of unity $e^{-2\pi i/N}$ is replaced by a primitive $N$-th root of unity $\omega \in \mathbb{Z}_q$, an element satisfying $\omega^N \equiv 1 \pmod q$ and $\omega^k \not\equiv 1$ for $0 < k < N$. Such an $\omega$ exists iff $N \mid (q-1)$, which is why FHE parameter sets pick an NTT-friendly prime $q \equiv 1 \pmod{2N}$.

The forward NTT of $a=(a_0,\dots,a_{N-1})$ produces $\hat a=(\hat a_0,\dots,\hat a_{N-1})$ defined by

$$ \hat a_k \;=\; \sum_{j=0}^{N-1} a_j\, \omega^{jk} \pmod q, \qquad k = 0,\dots,N-1. $$

That is Equation 2. The inverse NTT (iNTT) recovers the coefficients using $\omega^{-1}$ and a scaling by $N^{-1} \bmod q$:

$$ a_j \;=\; N^{-1} \sum_{k=0}^{N-1} \hat a_k\, \omega^{-jk} \pmod q, \qquad j = 0,\dots,N-1. $$

This is Equation 3. Evaluated naively, Equation 2 is again $O(N^2)$. The win comes from the Cooley-Tukey factorization: because $\omega$ is an $N$-th root of unity with $N$ a power of two, the transform splits recursively into even and odd index sub-transforms, giving the familiar $\log_2 N$ stages of length-$N$ work, hence $O(N \log N)$ modular multiply-adds. The atomic operation is the butterfly. The decimation-in-time (DIT) Cooley-Tukey butterfly, with twiddle factor $\omega^s$, is

$$ \begin{aligned} U' &\;=\; U + \omega^{s}\, V \pmod q, \\ V' &\;=\; U - \omega^{s}\, V \pmod q. \end{aligned} $$

This is Equation 4. The dual decimation-in-frequency (DIF) Gentleman-Sande butterfly, used on the inverse pass to avoid a separate bit-reversal, is $U' = U + V$, $V' = (U - V)\,\omega^{s} \pmod q$. Pairing a DIT forward transform that consumes natural order and produces bit-reversed order with a DIF inverse transform that consumes bit-reversed order and produces natural order lets an implementation skip explicit bit-reversal permutations entirely.

The pseudocode below gives the in-place iterative radix-2 NTT. It operates on an array already in bit-reversed order and applies $\log_2 N$ stages of butterflies.

function NTT_iterative(a[0..N-1], omega, q):   # a is in bit-reversed order
    # precompute twiddles w[t] = omega^t mod q for t in [0, N/2)
    len = 2
    while len <= N:
        wlen = omega^(N / len) mod q            # principal len-th root
        for start in 0, len, 2*len, ..., N-len:
            w = 1
            for j in 0 .. (len/2 - 1):
                U = a[start + j]
                V = (w * a[start + j + len/2]) mod q
                a[start + j]          = (U + V) mod q
                a[start + j + len/2]  = (U - V) mod q
                w = (w * wlen) mod q
        len = len * 2
    return a                                    # a is now in natural order

In (T)FHE and in companion lattice signature/KEM schemes (Kyber, Dilithium) the NTT is the workhorse: ciphertext polynomials are often kept in the NTT (evaluation) domain so that homomorphic multiplications are just elementwise products, and transforms are applied only when a representation change is unavoidable. NTT is exact - it introduces no rounding error - which makes it attractive for the BGV/BFV/CKKS families that operate over large modulus chains.

Negacyclic (negative-wrapped) convolution in $\mathbb{Z}_q[X]/(X^N+1)$

Ordinary polynomial multiplication of two degree-$(N-1)$ operands yields a degree-$(2N-2)$ result. A length-$N$ NTT implements cyclic convolution, i.e. multiplication modulo $X^N-1$, because $\omega^N=1$ wraps index $N+r$ back to $r$. But FHE needs the negacyclic ring $\mathbb{Z}_q[X]/(X^N+1)$, where $X^N \equiv -1$, so wrapped-around terms must be negated:

$$ \big(a \circledast b\big)_k \;=\; \sum_{i+j=k} a_i b_j \;-\; \sum_{i+j=k+N} a_i b_j \pmod q. $$

This is Equation 5. The naive route - zero-pad both operands to length $2N$, do a length-$2N$ cyclic NTT, then fold and negate - doubles the transform length and wastes half the work. The negative-wrapped convolution (NWC) trick avoids this by twisting the coefficients with a primitive $2N$-th root of unity $\psi$ satisfying $\psi^2 = \omega$ and $\psi^N \equiv -1 \pmod q$ (this exists precisely when $q \equiv 1 \pmod{2N}$).

Define the pre-twist $\tilde a_j = \psi^{\,j} a_j$ and likewise $\tilde b_j = \psi^{\,j} b_j$. Then a standard length-$N$ NTT of the twisted sequences, elementwise multiply, length-$N$ iNTT, and a post-twist by $\psi^{-j}$ produces the negacyclic product directly:

$$ c \;=\; \Psi^{-1} \odot \operatorname{iNTT}\!\Big( \operatorname{NTT}(\Psi \odot a) \;\odot\; \operatorname{NTT}(\Psi \odot b) \Big), $$

where $\Psi = (\psi^0,\psi^1,\dots,\psi^{N-1})$ and $\Psi^{-1}=(\psi^0,\psi^{-1},\dots,\psi^{-(N-1)})$. This is Equation 6. The $\psi^j$ weighting exactly injects the sign flip of Equation 5 because $\psi^N=-1$, so the implicit wrap of the length-$N$ transform lands in the negacyclic ring rather than the cyclic one. No zero padding, no length doubling.

In high-performance code the pre- and post-twists are not applied as a separate pass; they are merged into the butterfly twiddles. The forward transform absorbs $\psi$ into its DIT twiddles, and the inverse absorbs $\psi^{-1}$ into its DIF twiddles, so the merged-twiddle NTT computes the negacyclic product with no extra multiplications beyond the transform itself. This merged form is what FHE libraries (SEAL, OpenFHE, HEXL) and FPGA NTT cores actually ship.

function negacyclic_mul(a, b, q, N, psi):
    # psi: primitive 2N-th root of unity, psi^2 = omega, psi^N = -1 mod q
    # 1. pre-twist (or fold psi into the forward NTT twiddles)
    for j in 0 .. N-1:
        ta[j] = (psi^j * a[j]) mod q
        tb[j] = (psi^j * b[j]) mod q
    # 2. forward transforms to evaluation domain
    A = NTT(ta, omega = psi^2, q)
    B = NTT(tb, omega = psi^2, q)
    # 3. pointwise (Hadamard) product
    for k in 0 .. N-1:
        C[k] = (A[k] * B[k]) mod q
    # 4. inverse transform
    c = iNTT(C, omega = psi^2, q)              # includes N^{-1} scaling
    # 5. post-twist (or fold psi^{-1} into the inverse NTT twiddles)
    for j in 0 .. N-1:
        c[j] = (psi^{-j} * c[j]) mod q
    return c                                    # c = a * b in Z_q[X]/(X^N+1)

The FFT view and the multiplication-less integer FFT/iFFT

The NTT is the exact, finite-field route. The complex FFT is the alternative route, and it is the one TFHE leans on hardest. TFHE coefficients live on the discrete torus and are scaled to small integers; the modulus is effectively a power of two ($2^{32}$ or $2^{64}$), for which no convenient NTT root of unity exists. So TFHE typically maps a polynomial in $\mathbb{Z}_q[X]/(X^N+1)$ into $\mathbb{C}^{N/2}$, multiplies with a complex FFT, and rounds back. The same convolution theorem (Equation 1) applies, and the negacyclic fold is obtained by evaluating at the odd roots of unity $e^{-\pi i (2k+1)/N}$, equivalently by a half-size complex FFT on the folded sequence $a_j + i\, a_{j+N/2}$ pre-twisted by $\psi^j = e^{-\pi i j / N}$.

The reason FHE tolerates FFT at all is its inherent noise budget: ciphertexts already carry an error term, so the rounding error of an approximate transform is acceptable as long as it stays well below the decryption-error threshold $q/4$ (more precisely below half the gap between encoded plaintext slots). This noise tolerance is what accelerators such as FPT and MATCHA exploit aggressively, replacing IEEE double precision with narrow fixed-point and approximate transforms.

The phrase multiplication-less FFT refers to two compounding ideas used in TFHE-style hardware. First, the FFT twiddle factors $W_N^s = e^{-2\pi i s / N}$ are constants known at design time; multiplying by a constant complex value is far cheaper than a general multiplier and can be realized with shift-and-add networks or distributed arithmetic (canonical signed digit encoding), eliminating runtime multipliers. Second, MATCHA-style designs use a depth-first iterative conjugate-pair split-radix FFT, which minimizes the number of nontrivial twiddle multiplications (the trivial twiddles $1, -1, i, -i$ need only sign swaps and real/imaginary swaps, never a real multiply). The conjugate-pair variant pairs a twiddle with its complex conjugate so the two sibling butterflies share one twiddle constant, halving the constant-multiply hardware. The net effect: an FFT/iFFT kernel built almost entirely from adders and constant shifters, which is precisely what makes it dense and energy-efficient on an FPGA or ASIC.

The split-radix decimation-in-frequency butterfly, the structure these accelerators pipeline, is sketched below. It mixes a radix-2 step for even outputs with a radix-4 step for the two odd output groups, and is the lowest-multiply-count power-of-two FFT.

function FFT_splitradix_DIF(x[0..M-1]):        # M = N/2 complex points, recursive view
    if M == 1: return x
    if M == 2:
        return [ x[0] + x[1], x[0] - x[1] ]
    # even-indexed half: a plain length M/2 transform
    u = FFT_splitradix_DIF( [ x[2t] for t in 0..M/2-1 ] )
    # two odd-indexed quarters with conjugate-pair twiddles
    p = [ x[4t+1] + x[4t+3] for t in 0..M/4-1 ]
    q = [ x[4t+1] - x[4t+3] for t in 0..M/4-1 ]
    s = FFT_splitradix_DIF( twiddle(p,  W) )    # W^k constant multiplies (shift-add)
    t = FFT_splitradix_DIF( twiddle(q, W3) )    # W^{3k} = conjugate-pair of W^k
    return interleave(u, s, t)                  # only constant twiddles, no runtime mult

On the inverse side the iFFT runs the conjugate twiddles and a final $1/M$ scaling (itself a constant), after which coefficients are rounded to the nearest integer mod $q$. Because every twiddle is a fixed constant, an FPGA realizes each as a hardwired shift-add tree, giving the multiplication-less datapath.

Where transforms sit inside TFHE: gadget decomposition, blind rotation, CMux

To see why the transform is the hot loop, follow a TFHE programmable bootstrap. The accumulator is a TRLWE (ring-LWE) ciphertext - a pair of polynomials in $\mathbb{Z}_q[X]/(X^N+1)$. Bootstrapping rotates this accumulator by the (encrypted) input phase via a chain of $n$ CMux gates, one per bit of the LWE secret key. Each CMux multiplies an external product of a TGSW (gadget-encrypted) bit with two TRLWE ciphertexts. Every such external product is a batch of negacyclic polynomial multiplications - hence a batch of forward FFTs, pointwise products, and inverse FFTs.

Gadget decomposition. Multiplying a ciphertext by a large value would blow up the noise. TFHE instead decomposes each polynomial coefficient into small base-$B$ digits against a gadget vector $\mathbf{g} = (B^{\ell-1}, \dots, B, 1)$ (or its scaled form), so products are taken only between small digits and small key polynomials. For a coefficient $a$ the (signed, balanced) decomposition yields digits $a^{(i)} \in [-B/2, B/2)$ with

$$ a \;\approx\; \sum_{i=1}^{\ell} a^{(i)}\, \frac{q}{B^{i}}, \qquad \big| a - \textstyle\sum_i a^{(i)} q/B^{i} \big| \le \frac{q}{2 B^{\ell}}. $$

This is Equation 7; the right-hand bound is the decomposition error, controlled by choosing $B$ and $\ell$. Each of the $\ell$ digit-polynomials is then transformed and multiplied against the corresponding key row, so the external product costs $\ell$ forward transforms per input polynomial - which is exactly why minimizing transform cost dominates the whole scheme.

function gadget_decompose(a_poly, B, ell, q):   # signed/balanced base-B digits
    h = B^ell                                    # decomposition modulus
    for each coefficient a in a_poly:
        r = round(a * h / q)                     # rescale into [0, h)
        for i in 1 .. ell:                       # extract base-B digits
            d = r mod B
            if d >= B/2: d = d - B                # balance to [-B/2, B/2)
            r = (r - d) / B
            digit[i][coeff_index] = d
    return digit[1..ell]                          # ell small-coefficient polynomials

CMux and blind rotation. A CMux selects between two TRLWE ciphertexts $d_0, d_1$ under an encrypted control bit $C$ via the external product $\boxdot$, and the blind rotation chains $n$ of them, multiplying the accumulator by $X^{\,\tilde a_i}$ monomials. The external product $C \boxdot (d_1 - d_0)$ is where the negacyclic transforms fire.

function CMux(C, d0, d1):                          # C is TGSW, d0/d1 are TRLWE
    return d0 + ExternalProduct(C, d1 - d0)        # = d1 if bit=1 else d0

function BlindRotate(ACC, a[1..n], b, BSK):        # BSK = bootstrapping key (TGSW)
    ACC = X^{-b} * testvector                       # initial monomial rotation
    for i in 1 .. n:
        rotated = X^{a[i]} * ACC                    # monomial mult = coeff permute+negate
        ACC = CMux(BSK[i], ACC, rotated)            # each CMux: ell negacyclic FFT mults
    return ACC                                      # phase now blindly rotated by sum a_i*s_i

function ExternalProduct(C, d):                     # C: TGSW (ell x 2 of TRLWE), d: TRLWE
    G = gadget_decompose(d, B, ell, q)              # ell small digit-polynomials
    acc = 0
    for i in 1 .. (2*ell):
        D = FFT(G[i])                               # forward transform (hot loop)
        acc = acc + pointwise_mul(D, FFT_C[i])      # FFT_C precomputed once
    return iFFT(acc)                                # inverse transform back to coeffs

Note the monomial multiplication $X^{a_i}\cdot \mathrm{ACC}$ is free of any transform: multiplying by $X^k$ in $\mathbb{Z}_q[X]/(X^N+1)$ is a cyclic shift of coefficients by $k$ with a sign flip on the wrapped tail (the same negacyclic rule of Equation 5). Only the CMux external products need transforms, and they are invoked $n$ times per bootstrap, each touching $2\ell$ polynomials - the source of the transform-dominated profile.

Montgomery and Barrett modular reduction

Every butterfly multiply in the NTT (Equation 4) is a product of two values in $\mathbb{Z}_q$ that must be reduced modulo $q$. A hardware division by $q$ is expensive, so implementations use multiplication-based reduction. Two standard methods are Barrett and Montgomery reduction; both replace division by a precomputed-constant multiply plus shifts.

Barrett reduction. Given the product $x < q^2$ and a precomputed $\mu = \lfloor 2^{2k} / q \rfloor$ where $2^{k-1} \le q < 2^{k}$, Barrett estimates the quotient with a multiply-and-shift and corrects with at most one or two subtractions:

$$ \hat t \;=\; \Big\lfloor \frac{x\,\mu}{2^{2k}} \Big\rfloor, \qquad r \;=\; x - \hat t\, q, \qquad \text{then while } r \ge q:\; r \mathrel{-}= q. $$

This is Equation 8. Barrett works for an arbitrary modulus and leaves operands in the ordinary residue domain, which is convenient when reductions are sparse.

function Barrett_reduce(x, q, k, mu):    # mu = floor(2^(2k)/q), x < q^2
    t = (x * mu) >> (2*k)                 # quotient estimate (multiply + shift)
    r = x - t * q
    if r >= q: r = r - q                  # at most 1-2 conditional subtractions
    return r

Montgomery reduction. Montgomery maps each operand into the Montgomery domain $\bar a = a R \bmod q$ for a radix $R = 2^{k} > q$ with $\gcd(R,q)=1$. Reduction of a product $T < qR$ uses the precomputed $q' = -q^{-1} \bmod R$ and replaces the division by $q$ with a division by $R$, which is a free bit-shift:

$$ m \;=\; \big( (T \bmod R)\, q' \big) \bmod R, \qquad t \;=\; \frac{T + m\,q}{R}, \qquad \text{then if } t \ge q:\; t \mathrel{-}= q. $$

This is Equation 9; the result is $t = T R^{-1} \bmod q$, so products stay in the Montgomery domain and only the final result is mapped back. Montgomery shines when many multiplications chain without leaving the domain - exactly the NTT butterfly cascade - because the per-multiply cost is two multiplies and a shift with no data-dependent division. A common hardware pattern keeps all twiddle constants pre-converted to Montgomery form so the entire transform runs in-domain and converts back only once at the end.

function Montgomery_reduce(T, q, R, qprime):   # R = 2^k, qprime = -q^{-1} mod R, T < q*R
    m = ((T mod R) * qprime) mod R              # both mod R are just bit masks
    t = (T + m * q) >> k                         # exact division by R = shift
    if t >= q: t = t - q                         # one conditional subtraction
    return t                                     # = T * R^{-1} mod q

For the power-of-two moduli used in TFHE's integer/FFT path, reduction modulo $2^{32}$ or $2^{64}$ is implicit machine wraparound, so neither Barrett nor Montgomery is needed there; they are the tool of the NTT-based schemes (BGV/BFV/CKKS and FHEW-NTT variants) and of the lattice KEM/signature world, where $q$ is an odd NTT-friendly prime.

Putting the cost together and the systolic dataflow

Collecting the pieces, a single negacyclic product needs two forward transforms, $N$ pointwise modular multiplies, and one inverse transform. Each transform is $\tfrac{N}{2}\log_2 N$ butterflies, each butterfly one twiddle multiply plus an add and a subtract, and each multiply one Barrett/Montgomery reduction. So the dominant term is

$$ \text{cost} \;=\; 3 \cdot \frac{N}{2}\log_2 N \cdot c_{\text{bfly}} \;+\; N\, c_{\text{mul}} \;=\; O(N \log N), $$

which is Equation 10, versus $O(N^2)$ for schoolbook. For $N=2^{12}$ this is the difference between about $1.7\times 10^7$ and $1.7\times 10^4$ butterfly-scale operations, a roughly thousand-fold reduction, and it compounds across the $\ell$ gadget digits and $n$ CMux steps of every bootstrap.

Hardware accelerators exploit the regular butterfly structure with a systolic or deeply pipelined array: a grid of butterfly processing elements streams coefficients through stage by stage, with twiddle constants stored locally and reductions pipelined. The conceptual dataflow for one streaming NTT stage is below.

systolic_NTT_stage(stream_in, twiddles, q):
    # P parallel butterfly PEs, one transform stage per pass; data streams left-to-right
    for each clock tick:
        for pe in 0 .. P-1:                      # PEs operate in lockstep (SIMD/systolic)
            (U, V)  = read_pair(stream_in, pe)   # two coefficients from the stage buffer
            w       = twiddles[pe]               # constant for this PE/stage (Montgomery)
            t       = Montgomery_reduce(V * w, q) # pipelined multiply + reduce
            write_pair(stream_out, pe,
                       (U + t) mod q,             # add path
                       (U - t) mod q)             # subtract path
    # chain log2(N) such stages back-to-back; ping-pong buffers between stages

FPGA designs (Medha, FPT and similar) pipeline the modular multiplier so a butterfly issues every cycle, fuse the $\psi$ twist into the twiddle ROM so negacyclic reduction is free, and keep operands in Montgomery form across all stages. TFHE-oriented designs (MATCHA) instead instantiate the constant-twiddle, shift-add FFT/iFFT so the same throughput is reached with adders rather than multipliers. In both cases the architecture is shaped by the same fact this note began with: polynomial multiplication is the cost, and a good transform turns it from quadratic into near-linear streaming work.

Algorithms

Complete at-a-glance pseudocode for every algorithm referenced above. Inputs, outputs and numbered steps are self-contained.

Algorithm 1 - Iterative in-place radix-2 NTT (Cooley-Tukey, DIT butterflies).

Input : a[0..N-1] in bit-reversed order; primitive N-th root omega in Z_q; modulus q (N power of two)
Output: a[0..N-1] = NTT(a) in natural order
 1. len <- 2
 2. while len <= N do
 3.     wlen <- omega^(N/len) mod q
 4.     for start <- 0 to N-1 step len do
 5.         w <- 1
 6.         for j <- 0 to len/2 - 1 do
 7.             U <- a[start + j]
 8.             V <- (w * a[start + j + len/2]) mod q
 9.             a[start + j]         <- (U + V) mod q       # Eq. 4 butterfly
10.             a[start + j + len/2] <- (U - V) mod q
11.             w <- (w * wlen) mod q
12.         end for
13.     end for
14.     len <- 2*len
15. end while
16. return a
# iNTT: replace omega by omega^{-1}, then multiply every output by N^{-1} mod q (Eq. 3).

Algorithm 2 - Negacyclic (negative-wrapped) polynomial multiplication in $\mathbb{Z}_q[X]/(X^N+1)$.

Input : a, b in Z_q[X]/(X^N+1); psi a primitive 2N-th root with psi^2 = omega, psi^N = -1
Output: c = a*b in Z_q[X]/(X^N+1)
 1. for j <- 0 to N-1 do ta[j] <- (psi^j * a[j]) mod q ; tb[j] <- (psi^j * b[j]) mod q   # pre-twist
 2. A <- NTT(ta, omega, q)                          # Algorithm 1
 3. B <- NTT(tb, omega, q)
 4. for k <- 0 to N-1 do C[k] <- (A[k] * B[k]) mod q   # Eq. 6 Hadamard product
 5. c <- iNTT(C, omega, q)                          # includes N^{-1} scaling
 6. for j <- 0 to N-1 do c[j] <- (psi^{-j} * c[j]) mod q   # post-twist
 7. return c
# Optimized form folds psi into forward twiddles and psi^{-1} into inverse twiddles (steps 1 and 6 vanish).

Algorithm 3 - Split-radix conjugate-pair FFT (multiplication-less constant twiddles).

Input : complex array x[0..M-1], M power of two
Output: X = FFT(x)
 1. if M == 1 return x
 2. if M == 2 return [ x[0]+x[1], x[0]-x[1] ]
 3. u <- FFT( x[0], x[2], x[4], ... )                  # even indices, length M/2
 4. p <- ( x[1]+x[3], x[5]+x[7], ... )                 # odd quarter, sum
 5. r <- ( x[1]-x[3], x[5]-x[7], ... )                 # odd quarter, diff
 6. s <- FFT( apply W^k  to p )                         # W^k  = constant shift-add multiply
 7. t <- FFT( apply W^{3k} to r )                       # W^{3k} = conjugate-pair twiddle of W^k
 8. X <- interleave(u, s, t)                            # only trivial 1,-1,i,-i runtime ops
 9. return X
# iFFT: conjugate all twiddles, run the same flow, scale result by 1/M, round to nearest integer mod q.

Algorithm 4 - Gadget (base-$B$ balanced) decomposition.

Input : polynomial a_poly over Z_q; base B; levels ell; modulus q
Output: ell small-coefficient polynomials digit[1..ell], coeffs in [-B/2, B/2)
 1. h <- B^ell
 2. for each coefficient a of a_poly at index n do
 3.     r <- round(a * h / q)                           # rescale into [0, h)   (Eq. 7)
 4.     for i <- 1 to ell do
 5.         d <- r mod B
 6.         if d >= B/2 then d <- d - B                  # balance digit
 7.         r <- (r - d) / B
 8.         digit[i][n] <- d
 9.     end for
10. end for
11. return digit[1..ell]

Algorithm 5 - External product, CMux, and blind rotation (TFHE bootstrapping core).

ExternalProduct(C, d):                                  # C: TGSW (2*ell TRLWE rows), d: TRLWE
 1. G <- gadget_decompose(d, B, ell, q)                 # Algorithm 4
 2. acc <- 0
 3. for i <- 1 to 2*ell do
 4.     D <- FFT(G[i])                                   # Algorithm 3 (forward) - hot loop
 5.     acc <- acc + pointwise_mul(D, FFT_C[i])          # FFT_C[i] precomputed once
 6. end for
 7. return iFFT(acc)                                     # inverse transform back to coefficients

CMux(C, d0, d1):
 1. return d0 + ExternalProduct(C, d1 - d0)              # = d1 if control bit = 1 else d0

BlindRotate(testvector, a[1..n], b, BSK):
 1. ACC <- X^{-b} * testvector                           # monomial rotation (coeff shift + sign)
 2. for i <- 1 to n do
 3.     rotated <- X^{a[i]} * ACC                         # free negacyclic monomial mult (Eq. 5)
 4.     ACC <- CMux(BSK[i], ACC, rotated)                 # each CMux: 2*ell negacyclic transforms
 5. end for
 6. return ACC

Algorithm 6 - Barrett modular reduction.

Input : x with 0 <= x < q^2; modulus q; k with 2^(k-1) <= q < 2^k; mu = floor(2^(2k)/q)
Output: x mod q
 1. t <- (x * mu) >> (2k)                                 # quotient estimate (Eq. 8)
 2. r <- x - t*q
 3. while r >= q do r <- r - q                            # at most 1-2 subtractions
 4. return r

Algorithm 7 - Montgomery modular reduction.

Input : T with 0 <= T < q*R; R = 2^k with gcd(R,q)=1; qprime = -q^{-1} mod R
Output: T * R^{-1} mod q
 1. m <- ((T mod R) * qprime) mod R                       # both mod R are bit masks (Eq. 9)
 2. t <- (T + m*q) >> k                                    # exact division by R = shift
 3. if t >= q then t <- t - q                              # one conditional subtraction
 4. return t
# Keep operands and twiddles in Montgomery domain across all NTT stages; convert back once at the end.

Algorithm 8 - Streaming/systolic NTT stage (one transform pass).

Input : stage buffer of N coefficients; per-PE twiddle constants (Montgomery form); q; P parallel PEs
Output: next-stage buffer of N coefficients
 1. for each clock tick do
 2.     for pe <- 0 to P-1 in lockstep do
 3.         (U, V) <- read_pair(stage_buffer, pe)
 4.         w      <- twiddles[pe]
 5.         t      <- Montgomery_reduce(V * w, q)          # Algorithm 7, pipelined
 6.         write_pair(next_buffer, pe, (U + t) mod q, (U - t) mod q)
 7.     end for
 8. end for
 9. chain log2(N) such stages back-to-back with ping-pong buffers between them

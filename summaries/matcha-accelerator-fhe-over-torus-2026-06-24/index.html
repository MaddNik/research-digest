MATCHA: A Fast and Energy-Efficient Accelerator for Fully Homomorphic Encryption over the Torus

DAC 2022 (59th ACM/IEEE Design Automation Conference) · February 2022 · full view ↗

Problem and Motivation

Fully Homomorphic Encryption (FHE) lets an untrusted cloud server compute directly on ciphertexts and return encrypted results without ever learning the plaintext, addressing both data-breach risk and privacy legislation. Formally a homomorphic operation $\diamond$ on ciphertexts mirrors a plaintext operation $\star$, meaning $\mathrm{Dec}[\mathrm{Enc}(x_1)\diamond\mathrm{Enc}(x_2)] = x_1 \star x_2$ for input plaintexts $x_1$ and $x_2$. Compared to Secure Multi-Party Computation, FHE needs neither frequent client-server communication nor heavy circuit-garbling on the client.

Among FHE schemes, FHE over the Torus (TFHE) is uniquely attractive: it supports arbitrary Boolean logic gates (NAND, XOR, XNOR, AND, OR) with unlimited computation depth, enabled by an exceptionally fast bootstrapping of roughly $13\,\text{ms}$ on a CPU. As the scheme comparison shows, BGV bootstrapping costs about $800\,\text{s}$, BFV more than $1000\,\text{s}$, CKKS about $500\,\text{s}$, and FHEW under $1\,\text{s}$; only TFHE reaches the $13\,\text{ms}$ range. BGV, BFV, and CKKS also support only homomorphic addition and multiplication on integers or fixed point, whereas FHEW and TFHE evaluate Boolean algebra on binary data. Because every FHE operation injects noise, schemes with slow bootstrapping must restrict computation depth with a large noise budget, while TFHE's cheap bootstrapping enables truly unlimited depth.

Despite fast bootstrapping, a TFHE circuit built from many gates is still extremely slow because every gate triggers a bootstrapping. A TFHE-based simple RISC-V CPU made of thousands of gates runs at only about $1.25\,\text{Hz}$, and on state-of-the-art GPUs and FPGAs each gate still takes more than $0.2\,\text{ms}$. TFHE is well implemented only on CPUs and GPUs; the sole prior FPGA TFHE accelerator is even slower than a GPU, and no ASIC accelerator existed. As Figure 1 shows, the bootstrapping accounts for about 99 percent of gate latency, and within bootstrapping the double-precision floating-point FFT and IFFT kernels consume about 80 percent of the time, hurting both speed and energy efficiency. MATCHA targets exactly this bottleneck, improving TFHE gate throughput by $2.3\times$ and throughput per Watt by $6.3\times$ over prior accelerators.

TFHE Background: Torus, TLWE, TGSW

TFHE operates over the torus $\mathbb{T} = \mathbb{R}/\mathbb{Z}$ (real numbers modulo 1). For any ring $\mathcal{R}$, polynomials in the variable $X$ with coefficients in $\mathcal{R}$ are written $\mathcal{R}[X]$. The scheme uses the quotient rings $\mathcal{R}_N[X] := \mathcal{R}[X]/(X^N+1)$, $\mathbb{Z}_N[X] := \mathbb{Z}[X]/(X^N+1)$, and $\mathbb{T}_N[X] := \mathcal{R}_N[X]/\mathbb{Z}_N[X]$, the ring of polynomials in $X$ modulo $X^N+1$ with real coefficients taken modulo 1. The set $\mathbb{B} := \{0,1\}$ holds bits; $\mathbf{s} \xleftarrow{\$} \mathcal{S}$ denotes uniform sampling of $\mathbf{s}$ from $\mathcal{S}$, and $e \leftarrow \mathcal{X}$ denotes sampling $e$ from distribution $\mathcal{X}$.

For a plaintext bit $m \in \mathbb{B}$, setup picks public parameters $n = n(\lambda)$ and $\sigma = \sigma(\lambda)$ for security parameter $\lambda$ and samples a binary secret key $\mathbf{s} \xleftarrow{\$} \mathbb{B}^n$. Encryption samples a uniformly random vector $\mathbf{a} \xleftarrow{\$} \mathbb{T}^n$ and a Gaussian noise $e \leftarrow \mathcal{D}_{\mathbb{T}_N[X],\sigma}$, then outputs the TLWE sample $(\mathbf{a}, b)$ with

$$b = \mathbf{a}\cdot\mathbf{s} + e + m/2.$$

Decryption rounds to recover the message,

$$\mathrm{Dec}[\mathbf{s},(\mathbf{a},b)] = \lceil 2(b - \mathbf{a}\cdot\mathbf{s}) \rfloor,$$

which is correct as long as $|e| \lt 1/4$, since $2(b - \mathbf{a}\cdot\mathbf{s}) = 2e + m$ with $|2e| \lt 1/2$, so $\lceil 2(b - \mathbf{a}\cdot\mathbf{s}) \rfloor = m$. Equation 1 is encryption and Equation 2 is decryption; the rounding in Equation 2 is what lets TFHE tolerate small approximation errors. A TFHE logic gate $\mathrm{Logic}[c_0,c_1]$ (XOR, NAND, AND, OR) performs an addition of the two input ciphertexts followed by a bootstrapping that resets the noise.

TLWE is the torus analogue of Learning With Error. With $k$ a positive integer and $N$ a power of 2, a TLWE secret key $\bar{\mathbf{s}}$ is a vector of $k$ binary-coefficient polynomials over $\mathbb{Z}_N[X]$, that is $\bar{\mathbf{s}} \in \mathcal{R}_N[X]^k$. Given a polynomial message $\mu \in \mathbb{T}_N[X]$, a TLWE ciphertext is a sample $(\bar{\mathbf{a}}, \bar{b}) \in \mathbb{T}_N[X]^k \times \mathbb{T}_N[X]$ with $\bar{\mathbf{a}} \leftarrow \mathbb{T}_N[X]^k$ and $\bar{b} = \bar{\mathbf{s}}\cdot\bar{\mathbf{a}} + \mu + e$, where $e \leftarrow \mathcal{X}$. TGSW is the matrix extension of TLWE, each row a TLWE sample. The external product $\boxdot : \mathit{TGSW} \times \mathit{TLWE} \rightarrow \mathit{TLWE}$ maps a TGSW ciphertext of $\mu_{\mathit{TGSW}} \in \mathbb{T}_N[X]$ and a TLWE ciphertext of $\mu_{\mathit{TLWE}} \in \mathbb{T}_N[X]$ to a TLWE ciphertext of the product $\mu_{\mathit{TGSW}}\cdot\mu_{\mathit{TLWE}} \in \mathbb{T}_N[X]$.

The Bootstrapping Bottleneck

Each TFHE logic operation injects noise, so a bootstrapping must be performed at the end of each gate to remove it, and that bootstrapping is the dominant cost. The bootstrapping consists of initialization, rounding, a BlindRotate loop of external products, sample extraction, and key switching (the full listing appears in the Algorithms appendix as Algorithm 1). With the TLWE dimension set to $k=1$, the sample reduces to a Ring-LWE pair $(\bar{a}, \bar{b}) \in \mathbb{T}_N[X] \times \mathbb{T}_N[X]$.

The most computationally intensive step is the homomorphic decryption inside BlindRotate, where the message of the accumulator $\mathit{ACC}$ becomes the polynomial $X^{\bar{b} - \bar{\mathbf{a}}\mathbf{s}} \cdot \mathit{testv}$. In particular, homomorphically computing

$$X^{-\bar{\mathbf{a}}\mathbf{s}} = X^{\sum_{i=1}^{n} -\bar{a}_i s_i} = \prod_{i=1}^{n} X^{-\bar{a}_i s_i}$$

requires a large number of degree-$N$ polynomial multiplications. Naive multiplication of two degree-$N$ polynomials costs $O(N^2)$; FFT and IFFT reduce each polynomial multiplication to $O(N\log N)$, where $N$ is the polynomial degree.

For the torus implementation, although the scale-invariant scheme is defined over $\mathbb{T}$ with all operations modulo 1, TFHE rescales torus elements by a factor $2^{32}$ and maps them to 32-bit integers, so the modular reduction is implicit in native $\bmod\,2^{32}$ arithmetic and never performed explicitly. Even so, TFHE uses 64-bit double-precision floating-point FFT/IFFT because 32-bit integer or single-precision kernels are not accurate enough to guarantee correct decryption. A TFHE accelerator differs from BGV, BFV, and CKKS accelerators in two ways: it must bootstrap at the end of every gate (those schemes are usually leveled, without per-operation bootstrapping), and it needs only FFT/IFFT without modular reduction rather than NTT/INTT. Crucially, in a TFHE gate the FFT-to-IFFT invocation ratio is $1{:}4$ (IFFT runs far more often, and as Figure 1 shows IFFT latency dominates FFT), unlike the $2{:}1$ NTT-to-INTT ratio of BGV/BFV/CKKS accelerators, which gives TFHE far fewer chances to amortize bit-reversal overhead.

Key Idea 1: Approximate Multiplication-less Integer FFT and IFFT

MATCHA's first technique replaces the expensive floating-point FFT/IFFT with approximate, multiplication-less integer FFT/IFFT that use only additions and binary shifts. The insight is that TFHE uniquely tolerates these approximation errors: because a bootstrapping runs at the end of every gate, the FFT/IFFT errors are rounded off together with the noise during the decryption step (Equation 2) and never accumulate. This trick does not transfer to BGV, BFV, or CKKS, whose NTT/INTT errors would accumulate without a per-operation bootstrapping and quickly cause decryption failures.

Two structural choices implement this. First, MATCHA adopts the depth-first iterative conjugate-pair FFT (CPFFT) instead of the breadth-first Cooley-Tukey or Gentleman-Sande flows. TFHE stores a polynomial modulo $X^N+1$ either as a list of $N$ coefficients or in the Lagrange half-complex representation, the complex evaluations of the polynomial over the roots of unity $\exp(i(2j+1)\pi/N)$ for $j \in [\![0, N/2[\![$; FFT and IFFT are needed only to convert between these two representations. As Figure 2 shows, the breadth-first flow processes the transform stage by stage, while CPFFT traverses sub-transforms depth-first (completing a sub-transform before moving to the next) to capture spatial locality. CPFFT needs only a single complex root of unity read per radix-4 butterfly, and lets two butterflies in the same block share a twiddle factor, halving twiddle-buffer reads. It avoids bit-reversal overhead, which TFHE has few chances to amortize given its $1{:}4$ FFT/IFFT ratio.

Second, MATCHA uses a lifting-structure butterfly (Figure 3), a lattice substrate built by cascading identity matrices with a single nonzero off-diagonal element. A basic lifting step is an integer-to-integer transform with rounding,

$$y_j(n) = x_j(n),\quad y_i(n) = x_i(n) + \lceil T\, x_j(n)\rfloor,$$

$$z_j(n) = y_j(n),\quad z_i(n) = y_i(n) - \lceil T\, y_j(n)\rfloor,$$

where $T$ is the lifting coefficient. The lifting matrix and its inverse are $\begin{bmatrix}1 & T\\ 0 & 1\end{bmatrix}$ and $\begin{bmatrix}1 & T\\ 0 & 1\end{bmatrix}^{-1} = \begin{bmatrix}1 & -T\\ 0 & 1\end{bmatrix}$, so perfect reconstruction is preserved. Each floating-point coefficient is quantized to a dyadic value $\alpha/2^{\beta}$ with $\alpha,\beta \in \mathbb{N}$ so that multiplication becomes a few shifts and adds. For example,

$$\frac{9}{128} = \frac{2^3 + 2^0}{2^7} = \frac{1}{2^4} + \frac{1}{2^7},$$

realized as 4-bit and 7-bit shifters plus an adder (Figure 3(b)). Equations 3 and 4 are the forward and reverse lifting steps and Equation 5 is the dyadic decomposition; perfect reconstruction holds for any dyadic-quantized coefficient.

Key Idea 2: Aggressive Bootstrapping Key Unrolling (BKU)

BlindRotate computes external products $X^{-\mathbf{a}\mathbf{s}} = X^{\sum_{i=1}^{n} -a_i s_i}$ sequentially, one per secret-key bit, which is the gate bottleneck. Bootstrapping Key Unrolling (BKU) instead processes two key bits per external product, computing $X^{\sum_{i=1}^{n/2} -a_{2i-1}s_{2i-1} - a_{2i}s_{2i}}$ and cutting the number of homomorphic additions from $n$ to $n/2$. Because $s_i \in \{0,1\}$, the two-bit exponent can be expanded with the truth table in Figure 4. Starting from

$$X^{-a_{2i-1}s_{2i-1} - a_{2i}s_{2i}} = X^{-a_{2i-1}-a_{2i}}\,s_{2i-1}s_{2i} - X^{-a_{2i-1}}\,s_{2i-1}(1-s_{2i}) - X^{-a_{2i}}\,(1-s_{2i-1})s_{2i}$$

and using the identity $s_{2i-1}s_{2i} + (1-s_{2i})s_{2i-1} + s_{2i}(1-s_{2i-1}) + (1-s_{2i-1})(1-s_{2i}) = 1$, the exponentiation simplifies to

$$X^{-a_{2i-1}s_{2i-1} - a_{2i}s_{2i}} = (X^{-a_{2i-1}-a_{2i}} - 1)\,s_{2i-1}s_{2i} + (X^{-a_{2i-1}} - 1)\,s_{2i-1}(1-s_{2i}) - (X^{-a_{2i}} - 1)\,(1-s_{2i-1})s_{2i} + 1.$$

As Figure 5 shows, BKU encrypts $s_{2i-1}s_{2i}$, $s_{2i-1}(1-s_{2i})$, and $(1-s_{2i-1})s_{2i}$ as TGSW ciphertexts $BK_{i,0}$, $BK_{i,1}$, $BK_{i,2}$ and builds a bootstrapping key bundle (BKB) per iteration that unrolls the original bootstrapping key by two times.

Equations 6 and 7 are the two-bit expansions. BKU generalizes to unroll $m$ bits per external product:

$$X^{\sum_{i=1}^{n/m} -a_{m\cdot i}s_{m\cdot i} - a_{m\cdot i+1}s_{m\cdot i+1} - \cdots - a_{m\cdot i+m-1}s_{m\cdot i+m-1}},$$

with $m \in [2, n]$ (Equation 8). On a CPU, however, increasing $m$ beyond 2 actually slows bootstrapping. Unrolling by two times ($m=2$) cuts bootstrapping latency by 49 percent, but larger $m$ suffers from three issues: (1) too few CPU cores for the many exponent terms (at $m=4$ there are 15 terms, each needing a TGSW scale-and-add, against only 8 physical cores, with heavy communication to sum them); (2) more cache conflicts, because the bootstrapping-key size grows exponentially with $m$ (BKU at $m=2$ already needs three bootstrapping keys instead of one), so each TGSW scale-and-add evicts and reloads its key in the shared last-level cache; and (3) no pipelining, since in the existing implementation the BKB construction and the external product run strictly sequentially within each iteration.

MATCHA Architecture and Pipelined Datapath

MATCHA resolves the CPU limitations of aggressive BKU with a hardware pipeline that splits each bootstrapping step into two stages: constructing the bootstrapping key bundle (handled by a TGSW cluster) and the external product (handled by an EP core). As Figure 6 shows, in stage one a TGSW cluster, made of multiple TGSW scale units and a TGSW adder tree, computes each BKB term, for example $(X^{-a_{2i-1}-a_{2i}} - 1)\cdot BK_{i,0}$ when $m=2$ (where $BK_{i,0}$ is the TGSW ciphertext of $s_{2i-1}s_{2i}$), then sums the terms into $BKB_i$. In stage two an EP core computes $ACC \leftarrow BKB_i \boxdot ACC$. The two stages run in a pipeline: in each time step the EP core consumes the $BKB$ produced by the TGSW cluster in the previous step, so over time steps $t_0, t_1, t_2, \ldots$ the cluster produces $BKB_0, BKB_1, BKB_2, \ldots$ while the EP core trails by one. The TGSW cluster and EP core use separate register banks to avoid on-chip memory conflicts, and the workload of the two stages can be approximately balanced by tuning $m$.

The full chip (Figure 7) contains a polynomial unit, eight TGSW clusters, and eight EP cores, all connected to 32 scratchpad memory (SPM) banks through crossbars, plus a memory controller driving HBM2 DRAM. The polynomial unit handles additions and subtractions for each logic operation, bootstrapping initialization, sample extraction, and key switching (additions, logic comparisons, and Boolean logic). One TGSW cluster plus one EP core form one bootstrapping pipeline. Each TGSW cluster holds 16 32-bit integer multipliers and 16 32-bit integer adders with only two register banks, since TGSW scale accesses have strong spatial locality, allowing concurrent read of one bank and write of the other. Each EP core has one FFT core and four IFFT cores (matching the $1{:}4$ ratio), four 32-bit integer multipliers and four 32-bit integer adders to manipulate TGSW ciphertexts, and eight register banks to serve the irregular FFT/IFFT access pattern. An FFT core is like an IFFT core except for data flow; each FFT core has an address generation unit, a twiddle-factor buffer, two input/output FIFOs, and 128 butterfly cores, each butterfly built from two 64-bit integer adders and two 64-bit binary shifters, with the address generation unit steering butterflies to the twiddle-factor buffer.

Implementation, Error, and Noise Analysis

MATCHA was implemented in RTL and synthesized in a $16\,\text{nm}$ PTM process with state-of-the-art tools, using CACTI to model the SPM and register-file banks. The simple structure lets the whole design run at $2\,\text{GHz}$. The on-chip network uses two $8\times 32$ and one $8\times 8$ 256-bit bit-sliced crossbars for the SPM-to-cores, cores-to-SPM, and cores-to-cores paths. In total MATCHA occupies $36.96\,\text{mm}^2$ and consumes $39.98\,\text{W}$, with $640\,\text{GB/s}$ HBM2 bandwidth. Per the area and power table, the eight EP cores plus TGSW clusters dominate at $30.8\,\text{W}$ and $18.06\,\text{mm}^2$ (each EP core $2.87\,\text{W}$ and $1.89\,\text{mm}^2$, each TGSW cluster $0.98\,\text{W}$ and $0.368\,\text{mm}^2$), the polynomial unit is $2.33\,\text{W}$ and $0.32\,\text{mm}^2$, the crossbars $2.11\,\text{W}$ and $0.44\,\text{mm}^2$, the 4 MB 32-bank SPM $3.52\,\text{W}$ and $3.25\,\text{mm}^2$, and the memory controller plus HBM2 PHY $1.225\,\text{W}$ and $14.9\,\text{mm}^2$.

On accuracy (Figure 8), all polynomial coefficients are 32-bit integers while the twiddle factors are quantized at various bit-widths. Increasing the twiddle-factor bit-width lowers the approximate-FFT/IFFT error toward that of double precision. With 64-bit dyadic-value-quantized twiddle factors (DVQTFs) the approximate FFT/IFFT error is about $-141\,\text{dB}$, still larger than the $-150\,\text{dB}$ of true double-precision FFT/IFFT because only adds and shifts are used.

At the gate level, the noise comparison between two-times BKU and MATCHA (which unrolls $m \ge 2$ times) shows three trends. The external-product noise scales as $\delta/m$ (versus $\delta/2$ for plain BKU, where $\delta$ is the per-EP noise), the rounding noise scales as $\mathrm{RO}/m$ (versus $\mathrm{RO}/2$, where $\mathrm{RO}$ is the per-rounding noise), so both decrease linearly with $m$. But the bootstrapping-key noise grows as $(2^m - 1)\,\mathrm{BK}$ (versus $3\,\mathrm{BK}$ for $m=2$, where $\mathrm{BK}$ is the per-key noise), increasing exponentially with $m$. As a result, smaller $m$ tolerates more FFT/IFFT error. Experiments over $10^8$ TFHE gates show 38-bit DVQTFs cause no decryption failure for small $m$, but large $m$ (e.g. $m=5$) requires the full 64-bit DVQTFs because the bootstrapping-key noise dominates. MATCHA therefore adopts 64-bit DVQTFs throughout.

Evaluation Setup and Results

Performance was modeled at cycle level with OpenCGRA, a CGRA modeling framework validated against multiple ASIC accelerators. OpenCGRA compiles each TFHE operation into a data-flow graph (DFG) of MATCHA-supported operations, resolves dependencies, removes structural hazards, and schedules the DFG onto an architecture description of MATCHA to compute per-operation latency and energy. Baselines were an 8-core $3.7\,\text{GHz}$ Xeon E-2288G running the TFHE library (CPU), a 5120-core Tesla V100 with 16 GB HBM2 running cuFHE (GPU), the TFHE Vector Engine (TVE, originally on a low-end ZedBoard Zynq-7000) replicated eight times on a Stratix-10 GX2800 (FPGA), and that same 8-copy design synthesized in $16\,\text{nm}$ PTM as the ASIC baseline. BKU was enabled on CPU, GPU, and MATCHA, but fixed at $m=1$ on FPGA and ASIC, which do not support BKU.

All TFHE logic gates (NOT, AND, OR, NAND, XOR, XNOR) were studied, but only NAND results are reported, since AND, OR, NAND, XOR, and XNOR have nearly identical bootstrapping-dominated latency while NOT has no bootstrapping. Standard 110-bit security parameters from the TFHE paper were used: polynomial degree $N = 1024$, TLWE dimension $k = 1$, and TGSW decomposition basis $B_g = 1024$ with length $\ell = 3$.

Latency (Figure 9): a NAND gate costs $13.1\,\text{ms}$ on CPU, dropping to $6.67\,\text{ms}$ at $m=2$ but no further, due to CPU core, cache, and pipelining limits. GPU takes $0.37\,\text{ms}$ at $m=1$, improving to $0.18\,\text{ms}$ at $m=4$. MATCHA reduces NAND latency by 13 percent over GPU at $m=3$; the GPU already uses all resources on a single gate at $m=1$ or $m=2$, and MATCHA cannot exploit $m=4$ efficiently because it has only 8 TGSW clusters. FPGA and ASIC each need more than $6.8\,\text{ms}$ at $m=1$ due to no pipelining or memory optimization for BKU.

Throughput (Figure 10): because FPGA and ASIC support only $m=1$, even CPU at $m=2$ beats them on gate throughput. GPU and MATCHA lead by a wide margin, and at $m=3$ MATCHA improves NAND throughput by $2.3\times$ over GPU thanks to its pipelined aggressive-BKU architecture. Throughput per Watt (Figure 11): the FPGA ($\sim 40\,\text{W}$) and ASIC ($\sim 26\,\text{W}$) baselines improve over CPU by $2.4\times$ and $8.3\times$ at $m=1$, but the GPU's best efficiency (at $m=4$) is only about 58 percent of the ASIC's because the GPU draws more than $200\,\text{W}$. MATCHA, at just $39.98\,\text{W}$, improves NAND throughput per Watt by $6.3\times$ over the ASIC baseline.

In summary, MATCHA accelerates TFHE gates without decryption errors by combining approximate multiplication-less integer FFT/IFFT with a pipelined datapath for aggressive bootstrapping key unrolling, delivering $2.3\times$ higher gate throughput and $6.3\times$ higher throughput per Watt than prior CPU-, GPU-, FPGA-, and ASIC-based solutions.

Limitations and Discussion

Several limitations and scope boundaries follow from the design. First, the approximate multiplication-less FFT/IFFT is correct only because TFHE bootstraps after every gate; the technique does not transfer to leveled BGV, BFV, or CKKS accelerators, where NTT/INTT errors would accumulate and break decryption. Second, MATCHA's $-141\,\text{dB}$ approximate-transform error is intrinsically worse than the $-150\,\text{dB}$ of double precision, and the noise budget forces a tradeoff: large unroll factors $m$ raise bootstrapping-key noise as $(2^m-1)\,\mathrm{BK}$, so MATCHA must commit to the full 64-bit DVQTFs even though 38-bit factors would suffice at small $m$. Third, the hardware caps the usable unrolling: with only 8 TGSW clusters, MATCHA cannot efficiently support $m=4$, and its latency advantage over the GPU appears only at $m=3$ (a 13 percent latency reduction, with the real win on throughput and energy). Fourth, results are reported only for the NAND gate and for a single fixed parameter set ($N=1024$, $k=1$, $B_g=1024$, $\ell=3$) at 110-bit security; behavior at other parameters or security levels is not characterized. Finally, the evaluation is based on RTL synthesis and OpenCGRA cycle-level modeling rather than a fabricated chip, and the ASIC baseline is a synthesized version of an FPGA design rather than an independently optimized ASIC.

Algorithms

Algorithm 1 is the TFHE bootstrapping invoked at the end of every gate; line 7 (BlindRotate via external products) is the bottleneck MATCHA accelerates. Algorithm 2 makes explicit the bootstrapping-key-unrolling inner loop of Figure 5 (the $m=2$ case), and Algorithm 3 sketches the MATCHA pipelined dataflow that overlaps BKB construction with the external product.

Algorithm 1: TFHE bootstrapping
Input : TLWE sample (a, b) with plaintext m_in;
        constant m_set;
        bootstrapping key BK_{s -> s'', alpha};
        key-switching key KS_{s' -> s, gamma'}, where s' = KeyExtract(s'')
Output: TLWE sample encrypting m_out = m_in * m_set

1  mu  = m_set / 2                                  // Initialization
   mu' = mu / 2
2  b_bar   = round(2 * N * b)                       // Rounding
   a_bar_i = round(2 * N * a_i)   for each i in [1, n]
3  testv = (1 + X + ... + X^{N+1}) * X^{N/2} * mu'
4  ACC = X^{b_bar} * (0, testv)                     // ACC = TLWE( X^{(b_bar - a_bar.s)} * testv )
5  for i = 1 to n do                                // BlindRotate
6      BK_i = h + (X^{-a_bar_i} - 1) * BK_i
7      ACC  = BK_i  [external-product]  ACC
8  u = (0, mu') + SampleExtract(ACC)                // Extract
9  return KeySwitch_{KS}(u)                          // KeySwitch
Algorithm 2: Bootstrapping Key Unrolling inner loop (m = 2)
Input : rounded coefficients a_bar_1 .. a_bar_n;
        per-iteration TGSW keys:
          BK_{i,0} = TGSW[ s_{2i-1} * s_{2i} ]
          BK_{i,1} = TGSW[ s_{2i-1} * (1 - s_{2i}) ]
          BK_{i,2} = TGSW[ (1 - s_{2i-1}) * s_{2i} ]
        accumulator ACC
Output: updated ACC after n/2 external products

1  for i = 1 to n/2 do
2      BKB_i =  (X^{-a_{2i-1} - a_{2i}} - 1) * BK_{i,0}
3            +  (X^{-a_{2i-1}}          - 1) * BK_{i,1}
4            +  (X^{-a_{2i}}            - 1) * BK_{i,2}
5            +  1                                    // identity term
6      ACC = BKB_i  [external-product]  ACC
7  return ACC
// general unroll factor m in [2, n] replaces n/2 by n/m and
// uses (2^m - 1) TGSW keys, giving 2^m - 1 scale terms per BKB
Algorithm 3: MATCHA pipelined datapath (one bootstrapping pipeline)
Components: one TGSW cluster (scale units + adder tree),
            one EP core (1 FFT core + 4 IFFT cores)
Input : accumulator ACC, unroll factor m, iterations L = n/m
Output: bootstrapped ACC

1  for t = 0 to L do                               // pipelined over time steps
2      // STAGE 1 (TGSW cluster), iteration t:
3      if t < L then
4          for each of the (2^m - 1) terms in parallel:
5              term = (X^{exponent} - 1) * BK_{t, term}
6          BKB_t = adder_tree( all terms ) + 1
7      // STAGE 2 (EP core), iteration t-1, runs concurrently:
8      if t >= 1 then
9          ACC = BKB_{t-1}  [external-product]  ACC   // via approx integer FFT/IFFT
10 return ACC
// Stage 1 of step t overlaps Stage 2 of step t-1;
// separate register banks per stage avoid on-chip memory conflicts;
// m is chosen to balance Stage 1 and Stage 2 workloads.

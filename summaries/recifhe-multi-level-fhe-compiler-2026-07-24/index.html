Ciphertext- and Polynomial-Level Optimization for Fully Homomorphic Encryption (Recifhe)

arXiv:2607.15750v2 [cs.CR] · July 2026 · full view ↗

Overview and Contribution

This work (by Seongho Kim, Heelim Choi, Jaemin Kim, Seonyoung Cheon, Dongkwan Kim, Jaeho Lee, Hoyun Youm, Dongyoon Lee, Hanjun Kim, and Yongwoo Lee) presents Recifhe, a multi-level compiler for fully homomorphic encryption (FHE) under the RNS-CKKS scheme. The central observation is that existing FHE compilers operate at the coarse-grained ciphertext level, where a single ciphertext operation expands into a sequence of underlying polynomial operations. Optimizing only at the ciphertext level misses redundant polynomial computations that span multiple ciphertext operations.

Recifhe adds a second, finer optimization tier: after applying global ciphertext-level transformations, it lowers each ciphertext operation into its polynomial-operation sequence and optimizes across ciphertext boundaries. The three polynomial-level optimizations are (i) common subexpression elimination, (ii) performance-aware ModDown hoisting, and (iii) operation fusion, backed by liveness-driven scheduling to contain memory growth.

Headline results: Recifhe achieves a 1.25x speedup over ciphertext-level-only optimization across 12 benchmarks, and a 1.36x speedup with 0.93x the memory footprint versus the manually optimized polynomial sequences in the Cheddar GPU FHE library.

Background: RNS-CKKS and its Attributes

RNS-CKKS is the residue-number-system variant of CKKS, well suited to machine learning because it supports fixed-point arithmetic on real numbers with SIMD vectorization. It encodes a vector of real numbers into a plaintext polynomial in $\mathbb{Z}[X]/(X^N+1)$, where $N$ is the polynomial modulus degree, providing $N/2$ fixed-point SIMD slots.

A plaintext is encrypted as a pair of ciphertext polynomials in $\mathbb{Z}_Q[X]/(X^N+1)$, where the coefficient modulus $Q$ bounds the coefficients. Each polynomial is represented by up to $L$ residue polynomials with per-limb moduli $q_i$, so that

$$Q = \prod_{i=1}^{L} q_i. \qquad \text{(Equation 1)}$$

The GPU library used in this work (Cheddar) adopts $N = 2^{16}$ and $L = 27$ with 32-bit moduli, so a plaintext occupies $4~\text{bytes} \times N \times L = 6.75~\text{MB}$, and a ciphertext twice as much.

Ciphertext-level compilers track two ciphertext attributes:

  • Level - the number of Rescale operations still applicable to the ciphertext, up to $L$.
  • Scale - the fixed-point scaling factor multiplying the encoded message.

A ciphertext multiplication enlarges the scale of the result, risking scale overflow. Rescale divides the scale by the last modulus and lowers the level by one, preventing overflow. Bootstrap restores a depleted level back to its maximum $L$. Compilers automatically place these two ciphertext management operations: scale-management compilers insert Rescale; bootstrapping-management compilers insert Bootstrap to support long multiplication chains while minimizing the Bootstrap count.

Polynomial Operations, Key Switching, and the RNS Basis

Each ciphertext operation decomposes into a sequence of polynomial operations, and each polynomial carries a third attribute beyond level and scale: the RNS basis, the set of moduli over which the polynomial is represented. A polynomial operation requires its operands to share the same RNS basis. Table I gives the ciphertext-to-polynomial mapping (omitting Rescale and Bootstrap, which appear identically at both levels):

Ciphertext Op.      Polynomial Op. sequence
-----------------   ---------------------------------------
AddCC, AddCP        pAdd
MulCP               pMulCP
MulCC               pMulCC -> ModUp -> KIP -> ModDown
Rotate              ModUp -> KIP -> ModDown -> Automorphism

Both MulCC (ciphertext-ciphertext multiply) and Rotate produce an intermediate result under a different key, so key switching restores it to the original key via ModUp, KIP (KeyInnerProduct), and ModDown. KIP computes the inner product of the ModUp result with the key-switching key, which amplifies key-switching error. To manage this, RNS-CKKS temporarily extends the working modulus from $Q$ to $PQ$ with an auxiliary modulus

$$P = \prod_j p_j. \qquad \text{(Equation 2)}$$

ModUp lifts a polynomial from the $Q$ basis to the $PQ$ basis by adding the $p_j$ moduli to its basis; after KIP, ModDown divides the result by $P$ to reduce key-switching noise and returns it to the $Q$ basis. Automorphism rotates the slots via an index permutation.

Operations split by their RNS-basis sensitivity (Table II):

Insensitive   pAdd, pMulCP, Automorphism
Sensitive     ModUp, KIP, ModDown, pMulCC, Rescale, Bootstrap

RNS-basis-insensitive operations apply identically on both the $Q$ and $PQ$ bases, processing each residue polynomial independently and preserving the factor $P$ introduced by key switching. The four sensitive operations that change the RNS basis - ModUp, ModDown, Rescale, and Bootstrap - internally use the number-theoretic transform (NTT) and its inverse (iNTT) for basis conversion. Fusing an adjacent pair of such operations can cancel their internal transforms.

Motivation: the Gap in Existing Compilers

Most existing FHE compilers optimize only at the ciphertext level (e.g. Hecate, ELASM, EVA, DaCapo, HALO, Orion), applying global optimizations such as Rescale hoisting but never removing redundant polynomial computations across ciphertext operations. A few compilers (HEaaN.MLIR, ANT-ACE) do decompose ciphertext operations into polynomial sequences, but they do not track polynomial attributes (the RNS basis) across ciphertext operations, so their optimization is confined to a single ciphertext operation.

The Cheddar FHE library instead hand-codes cross-operation optimizations (such as removing repeated ModDown operations), but its optimization is a fixed algorithm tuned to one dataflow: it does not adapt to new programs, and it ignores the trade-off between removed ModDown operations and the extra operations that must then run on the heavier $PQ$ basis. It also extends polynomial lifetimes, incurring high memory overhead. These limitations motivate a compiler that applies dataflow-adaptive optimization at the polynomial level - Recifhe's design goal.

Recifhe Compiler Architecture

Recifhe adopts a multi-level design: global optimizations run on a ciphertext-level intermediate representation (IR), and fine-grained cross-ciphertext optimizations run on a polynomial-level IR. Doing global work at the coarse ciphertext level keeps optimization complexity low compared with optimizing the whole program directly at the polynomial level (which contains far more operations).

The pipeline is:

  1. Ciphertext-level IR: transform an input Python DSL program into an FHE program - place ciphertext management operations (Bootstrap placement and scale management via prior techniques) and apply global Rescale hoisting. After this pass the level and scale of every value are known.
  2. Lowering: decompose each ciphertext operation into its polynomial-operation sequence (per Table I), which exposes cross-ciphertext optimization opportunities.
  3. Polynomial-level IR: run candidate analysis, performance-aware ModDown hoisting, operation fusion, and liveness-driven polynomial scheduling.

The paper's running example computes

$$y = p_0\cdot a + p_1\cdot \mathrm{rot}(b, r_1) + p_2\cdot \mathrm{rot}(c, r_2) + p_3\cdot \mathrm{rot}(d, r_3),$$

a rotate-multiply-accumulate pattern that recurs in FHE matrix-multiplication kernels. Each rotation lowers into ModUp, KIP, ModDown, and Automorphism. Because the results of the three ModDown operations flow only through RNS-basis-insensitive operations, hoisting them into a single ModDown reduces the ModDown count and enables fusing the surrounding operations. The input DSL for the example is:

t0  = Mul(a, p0);
t1  = Mul(Rot(b), p1);
t2  = Mul(Rot(c), p2);
t3  = Mul(Rot(d), p3);
t4  = Add(RS(t1), RS(t2));
t5  = Add(RS(t0), t4);
out = Add(t5, RS(t3));

Ciphertext-Level Optimization: Rescale Hoisting and Lowering

Recifhe first places Bootstrap operations and manages the program's scale using existing ciphertext-level techniques, so that level and scale are fixed before lowering. It inherits Rescale hoisting and keeps it as a ciphertext-level transformation - performing it here rather than at the polynomial level, where the program has many more operations, keeps the pass tractable. Recifhe hoists a Rescale past any operation that preserves both level and scale, merging Rescale operations to reduce their count.

Recifhe then lowers each ciphertext operation into its polynomial sequence. Because lowering translates each ciphertext operation independently, the same input polynomial may be recomputed across separate lowered sequences; Recifhe removes these with common subexpression elimination on the polynomial-level IR.

Polynomial-Level: Candidate Analysis

On the polynomial-level DAG of each function, Recifhe looks for regions where multiple ModDown operations can be hoisted and merged (Algorithm 1, lines 5-10). It visits each RNS-basis-sensitive operation $s$ other than ModDown, exploiting the fact that the earlier ciphertext-level passes already placed the ciphertext operations that lower into these sensitive operations. For each such $s$:

  • The boundary set $B$ (line 5) is the set of preceding sensitive operations (other than ModDown) that can reach $s$ only through RNS-basis-insensitive and ModDown operations.
  • The optimizable region $R$ (line 6) is every intermediate operation on any path from a boundary operation to $s$.
  • The hoisting target set $M$ (line 7) is the ModDown operations within $R$; their results can flow toward $s$ only through insensitive operations, which compute identically on the $Q$ and $PQ$ bases, so a hoisted ModDown can cross them without changing the encoded values.

A region becomes a candidate only when $M$ contains multiple ModDown operations ($|M| \ge 2$). Regions of different sensitive operations may be nested; Recifhe resolves this by iterative optimization, since hoisting in a nested region and in its enclosing region compose without conflict.

Polynomial-Level: Performance-Aware Hoisting

Hoisting removes the ModDown operations in $M$ and places one hoisted ModDown before each operation in the sensitive frontier $F$ - the sensitive operations that the results of $M$ first reach via insensitive operations only (Algorithm 1, lines 11-19). The set $L$ collects the insensitive operations on the paths from $M$ to $F$; after hoisting, these now process $PQ$-basis polynomials. Because $PQ$ holds more moduli than $Q$, operations in $L$ become more expensive, and whether this added cost is worthwhile depends on the region's dataflow.

Recifhe therefore profiles offline the latency $E$ of every polynomial operation at each level and RNS basis (the basis written as a subscript). It accepts a hoist only when the benefit outweighs the overhead:

$$\text{benefit} = \sum_{m \in M} E(m) - \sum_{f \in F} E(\text{ModDown}_f), \qquad \text{(Equation 3)}$$

$$\text{overhead} = \sum_{op \in L} \big( E_{PQ}(op) - E_{Q}(op) \big), \qquad \text{(Equation 4)}$$

i.e. the latency of the removed ModDown operations minus the latency of the hoisted ModDown operations at $F$, versus the latency increase from raising $L$ to the $PQ$ basis. When $\text{benefit} > \text{overhead}$, Recifhe applies the hoist by reordering the accumulations in the region (function reassociate) to gather the results of $M$, removing $M$, and inserting a hoisted $\text{ModDown}_f$ before each $f \in F$. Reordering is safe because modular addition is associative and commutative, so the encoded result is preserved.

Polynomial-Level: Operation Fusion and Scheduling

Operation fusion. Recifhe fuses adjacent RNS-basis-sensitive operations - e.g. a ModDown and a Rescale - to remove an iNTT/NTT pair when nothing lies between them. When a fusible pair is not adjacent (in the running example an addition separates a ModDown from a Rescale), Recifhe raises the addition's other operand, still on the $Q$ basis, to the $PQ$ basis by inserting an invModDown. This lets the ModDown sink past the addition and become adjacent to the Rescale. invModDown is a lightweight lift that multiplies each residue polynomial in the $Q$ basis by $P$, satisfying

$$\mathrm{ModDown}(\mathrm{invModDown}(a)) = a. \qquad \text{(Equation 5)}$$

Polynomial scheduling. Polynomial-level optimization inherently raises memory pressure: lowering places each ciphertext operation's polynomial operations consecutively (separating a value's producer from its consumers), and hoisting and fusion extend polynomial lifetimes across ciphertext boundaries. Recifhe applies liveness-driven scheduling - it moves each operation to immediately follow its most recently defined operand, shrinking that operand's live range when the operation is its last use, and then reuses the buffer of a dead value for a later same-size result. This reduces the number of polynomials live simultaneously.

Evaluation

Setup. Recifhe is built on the MLIR-based Hecate framework with a Python DSL frontend, targeting the Cheddar GPU library as backend; it is retargetable to other backends by re-profiling operation latencies. It is evaluated on 12 benchmarks (Polynomial Regression, Multivariate, SVM, Kmeans, MLP, LoLA, LeNet, AlexNet, VGG16, ResNet, SqueezeNet, MobileNet) against two baselines that share its frontend and backend:

  • Ciphertext-level-only baseline - prior ciphertext-level optimizations only.
  • Cheddar baseline - the above plus Cheddar's manually optimized polynomial sequences.

All three share the RNS-CKKS setting $N = 2^{16}$, default scale $2^{40}$, Cheddar's Bootstrap modulus configuration at 128-bit security, giving an effective multiplicative depth of 13 between Bootstrap operations. Benchmarks use pre-encoded plaintexts and follow prior suites for Bootstrap placement. Hardware: two Intel Xeon Gold 6326 CPUs, 512 GB RAM, and an NVIDIA RTX PRO 6000 Blackwell Max-Q GPU with 96 GB VRAM.

Results (latency, normalized to ciphertext-level-only). The Cheddar baseline speeds up 6 of the 9 benchmarks it completes but slows down 3, because it raises operations into the heavier $PQ$ basis regardless of whether the hoist is actually profitable - the fixed-algorithm limitation. Its average is $0.92\times$ (i.e. a net slowdown). Recifhe, applying hoisting only when Algorithm 1 finds it profitable, reaches $1.25\times$ speedup over ciphertext-level-only across all 12 benchmarks, and $1.36\times$ over Cheddar on the 9 it completes.

Results (memory). The Cheddar baseline's unconditional lifting, with no scheduling pass to reclaim extended live ranges, keeps $PQ$-basis polynomials live longer than needed: $1.23\times$ the ciphertext-level-only footprint on average, running out of memory on 3 of the 12 benchmarks. Recifhe's liveness-driven scheduling holds the footprint to $1.14\times$ the ciphertext-level-only baseline and $0.93\times$ the Cheddar baseline.

Fidelity and compile time. On ResNet, Recifhe reproduces Cheddar's artifact result within 0.9% (0.714 s vs 0.720 s), confirming it faithfully reimplements the library optimization. The extra polynomial-level passes take 39.0% of Recifhe's compile time on average (performance-aware hoisting 1.31%, fusion 14.3%). Compile times range from 0.0182 s (Polynomial Regression) to 22.4 s (MobileNet), averaging 5.14 s - $2.92\times$ the ciphertext-level-only baseline in geometric mean.

Conclusion

Recifhe is a multi-level FHE compiler that automatically generates RNS-CKKS FHE programs with global ciphertext-level optimizations and then removes redundant polynomial computations across ciphertext operations. Guided by measured per-operation latencies, it applies ModDown hoisting only when the estimated benefit exceeds the overhead, and it curbs the resulting memory growth with liveness-driven scheduling. Net result: a $1.25\times$ speedup over ciphertext-level-only optimization, and a $1.36\times$ speedup at $0.93\times$ the memory of the hand-optimized Cheddar library sequences.

Algorithms

The source contains one formal algorithm, the performance-aware ModDown hoisting pass, reproduced below verbatim in structure. Symbols: $\Pi$ is the polynomial-level function DAG; $E$ is the offline latency profile (subscript denotes RNS basis); $S$ is the sensitive operations other than ModDown; $I$ the insensitive-or-ModDown operations; $B$ the boundary set; $R$ the optimizable region; $M$ the hoisting targets; $F$ the sensitive frontier; $L$ the insensitive path operations raised to the $PQ$ basis.

Algorithm 1: Performance-Aware ModDown Hoisting
Input : Polynomial-level function DAG Pi, Latency profile E
Output: ModDown-hoisted DAG Pi

 1  function PerformanceAwareModDownHoisting(Pi, E):
 2      S = { op in Pi | isSensitive(op) and op != ModDown }
 3      I = { op in Pi | isInsensitive(op) or  op == ModDown }
 4      foreach s in S do
 5          B <- { b in S | exists pi in Path(b, s) : ops(pi) subset-of I }
 6          R <- union over b in B, pi in Path(b, s) of ops(pi)
 7          M <- { op in R | op == ModDown }
 8          if |M| < 2 then
 9              continue
10          end
11          F <- findNearestSensitiveSuccessors(Pi, M)
12          L <- { op in Path(m, f) | m in M, f in F }
13          benefit  <- sum_{m in M} E(m) - sum_{f in F} E(ModDown_f)
14          overhead <- sum_{op in L} ( E_PQ(op) - E_Q(op) )
15          if benefit > overhead then
16              Pi <- reassociate(Pi, R, M)
17              Pi <- removeModDown(Pi, M)
18              Pi <- insertModDownBefore(Pi, F)
19          end
20      end
21      return Pi
22  end

Supporting transformations invoked by the body but not given as separate numbered algorithms:

Common Subexpression Elimination (post-lowering):
  for each polynomial op produced by independent lowering:
    if an identical op on the same input polynomial already exists:
      reuse the existing result; drop the duplicate.

Operation Fusion:
  for each RNS-basis-sensitive pair (opA, opB) e.g. (ModDown, Rescale):
    if opA and opB are adjacent (no op between them):
      fuse -> cancel the internal iNTT/NTT pair.
    else if separated only by an insensitive op (e.g. an Add):
      insert invModDown on the other Q-basis operand   # lift to PQ basis
      sink ModDown past the op until adjacent to Rescale
      fuse.
  # invariant: ModDown(invModDown(a)) = a

Liveness-Driven Polynomial Scheduling:
  for each op in program order:
    move op to immediately follow its most-recently-defined operand
    if op is that operand's last use:
      mark the operand's buffer dead
  reuse a dead buffer for a later result of the same size

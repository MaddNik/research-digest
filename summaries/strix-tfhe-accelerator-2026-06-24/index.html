Strix: An End-to-End Streaming Architecture with Two-Level Ciphertext Batching for Fully Homomorphic Encryption

MICRO 2023; arXiv:2305.11423 · May 2023 · full view ↗

At a glance

Authors: Adiwena Putra, Prasetiyo, Yi Chen, John Kim, Joo-Young Kim (School of Electrical Engineering, KAIST, Daejeon, South Korea). Full title: "Strix: An End-to-End Streaming Architecture with Two-Level Ciphertext Batching for Fully Homomorphic Encryption with Programmable Bootstrapping" (MICRO 2023; arXiv:2305.11423).

One sentence: Strix is - to the authors' knowledge - the first hardware accelerator built specifically for high-throughput programmable bootstrapping (PBS) in the TFHE scheme; it attacks the GPU blind-rotation fragmentation bottleneck with a two-level (device plus core) ciphertext batching scheme realized in a fully-pipelined streaming datapath, and co-designs an FFT-friendly microarchitecture that TFHE permits but CKKS does not.

Headline numbers:

  • $1{,}067\times$ higher PBS throughput than a state-of-the-art CPU (Concrete on Intel Xeon Platinum) and $37\times$ higher than a GPU (NuFHE on Nvidia Titan RTX).
  • $7.4\times$ higher throughput and $1.25\times$ lower latency than the prior best TFHE ASIC, Matcha.
  • Up to $74{,}696$ PBS/s at $0.16$ ms latency on parameter set I; supports the widest range of TFHE parameter sets of any accelerator, including polynomial degree $N$ up to $16{,}384$ (set IV).
  • End-to-end $33$ to $38\times$ (vs CPU) and $8$ to $17\times$ (vs GPU) speedup on Zama Deep-NN inference (the conclusion quotes the peak $38\times$ / $17\times$).
  • $141.37\ \text{mm}^2$, $77.14$ W, 8 cores at $1.2$ GHz in TSMC 28 nm, with only $26$ MB on-chip memory and a single $300$ GB/s HBM2e stack - far smaller than CKKS accelerators (45 to 512 MB on-chip, roughly 1 TB/s of bandwidth from two HBM2e stacks).

The problem: TFHE has no packing, so PBS is throughput-starved

All FHE schemes hide data under noise (the Learning With Errors, or LWE, problem) and rely on bootstrapping to refresh that noise so an unbounded amount of computation can continue. Bootstrapping is the dominant cost: a baseline inference that takes about $100$ ms on a CPU can take up to 3 hours under FHE, a $>100{,}000\times$ slowdown. Schemes split into two families. BGV-like schemes (BGV, BFV, CKKS) encrypt at the vector level and pack many values per ciphertext, bootstrapping a whole vector at once - high throughput but MB-level ciphertexts, GB-level keys, and 256 to 512 MB on-chip on recent accelerators (one CKKS accelerator is estimated at roughly $418\ \text{mm}^2$ with 512 MB on-chip). GSW-like schemes (GSW, FHEW, TFHE) encrypt at the word or bit level. TFHE's key advantage is programmable bootstrapping (PBS): it evaluates an arbitrary univariate function (a homomorphic look-up table) while simultaneously refreshing noise, which makes it ideal for logical and relational operations, for neural-network activation functions, for tree-based ML, and even for emulating a CPU over encrypted programs. TFHE ciphertexts are KB-level and keys are tens to hundreds of MB. Table I in the paper contrasts the two: CKKS supports packing, high-throughput / high-latency bootstrapping, and real/complex messages; TFHE has no packing, low-latency / low-throughput bootstrapping, real/integer/boolean messages, and can use FFT or NTT for polynomial transforms.

The catch is that TFHE does not support ciphertext packing, so PBS must be run one message at a time, giving inherently low throughput. The authors profile a single TFHE gate operation on CPU (Concrete, 110-bit security) and find, as shown in Figure 1, that PBS is about $65\%$ of the time, keyswitching (KS) about $30\%$, and the remaining homomorphic add/subtract operations about $5\%$ (the split varies slightly with the security parameter). Within PBS, blind rotation (BR) is about $98\%$ of the time, the rest going to modulus switching and sample extraction. Within each BR iteration the dominant cost is the external product (FFT, then vector multiply, then accumulate plus IFFT). Because gadget decomposition produces $l_b$ polynomials per input, the FFT processes $l_b$ times more polynomials than the IFFT, giving an FFT-to-IFFT workload ratio of $l_b : 1$ that worsens as $l_b$ grows. Crucially, blind rotation is a sequence of non-parallelizable iterations, each multiplying a polynomial vector by the bootstrapping-key matrix, so latency cannot be hidden by simply adding more lanes inside one iteration.

TFHE data structures, PBS, and keyswitching

Four entities drive PBS and keyswitching. An LWE ciphertext is a vector of $(n+1)$ scalars $[a_1,\ldots,a_n,b]$, each a 32-bit or 64-bit integer, used to encrypt messages. A GLWE test vector stores the univariate function for PBS as a vector of $(k+1)$ polynomials of degree $N-1$, namely $[A_1(X),\ldots,A_k(X),B(X)]$, each polynomial being $N$ integer coefficients. The bootstrapping key ($\mathrm{bsk}$) is a vector of $n$ GGSW ciphertexts, where each GGSW is a $(k+1)\,l_b \times (k+1)$ matrix of degree-$(N-1)$ polynomials. The keyswitching key ($\mathrm{ksk}$) is a vector of $k\,N\,l_k$ LWE ciphertexts. Table II gives the typical TFHE parameter ranges: $n \in [256, 4096]$, $N \in [2^{10}, 2^{14}]$, $k \in [1,4]$, and $l_b \in [2,4]$.

Programmable bootstrapping (PBS). PBS has three building blocks: modulus switching (cheap, since each LWE element is switched from modulus $q$ to $2N$ and $N$ is a power of two), blind rotation (the core, about $98\%$ of PBS), and sample extraction (which forms a new LWE ciphertext by extracting one coefficient from the rotated GLWE). Each blind-rotation iteration performs polynomial rotation and subtraction, gadget decomposition, then the external product. The external product computes, for each output column $j$,

$$\mathrm{otv}[j] \;=\; \mathrm{otv}[j] \;+\; \sum_{r=0}^{(k+1)l_b-1} \mathrm{IFFT}\!\Big(\mathrm{FFT}\big(\mathrm{etv}[r]\big)\cdot \mathrm{FFT}\big(\mathrm{bsk}[i][r][j]\big)\Big)$$

as in Equation 1, exploiting the convolution theorem so that negacyclic polynomial multiplication becomes pointwise complex multiplication in the frequency domain. The full PBS procedure (Algorithm 1 of the paper) is reproduced below; the complete self-contained listing also appears in the Algorithms appendix.

Algorithm 1: Programmable Bootstrapping
Require: n, k: mask length in LWE / GLWE ciphertext
Require: lb: decomposition level of bootstrapping
Require: N: number of coefficients in polynomial
Require: c[n+1]: LWE ciphertext, each element is a scalar
Require: tv[k+1]: test vector, each element is a degree-N polynomial
Require: bsk[n][(k+1)lb][k+1]: bootstrapping key, each element a degree-N polynomial
Ensure:  o[kN+1]: LWE ciphertext, each element a scalar
 1: Local variables: etv[(k+1)lb], otv[k+1]   (initialized to zero)
 2: for i = 0 to n do
 3:     c[i] = ModSwitch(c[i])                      // Modulus switching (q -> 2N)
 4: tv = Rotate('left', tv, c[n])                   // initial rotation by the body term
 5: for i = 0 to n-1 do                            // Blind rotation
 6:     tv  = tv - Rotate('right', tv, c[i])        // Rotate and subtract
 7:     etv = Decompose(tv, lb)                      // Gadget decomposition -> lb polys
 8:     for j = 0 to k do                            // output columns
 9:         for r = 0 to (k+1)lb - 1 do              // input polynomials
10:             otv[j] = otv[j] + IFFT(FFT(etv[r]) * FFT(bsk[i][r][j]))  // External product (Eq. 1)
11:     tv = otv
12:     initZero(otv)
13: o = SampleExtract(tv)                            // Sample extraction

Keyswitching. After PBS the output LWE ciphertext is a vector of $k\,N + 1$ scalars encrypted under a different key. Keyswitching converts it back to an $(n+1)$-scalar ciphertext under the original key by decomposing the first $k\,N$ elements into a $k\,N\,l_k$-element vector and multiplying by a $k\,N\,l_k \times (n+1)$ matrix. Because this is purely scalar vector-matrix multiplication (no polynomials), it maps to a much simpler datapath than PBS, and Strix gives it a dedicated cluster. Algorithm 2 of the paper is reproduced below.

Algorithm 2: Keyswitching
Require: n, k: input mask length of LWE / GLWE ciphertext
Require: lk: decomposition level of keyswitching
Require: c[n+1]: LWE ciphertext, each element a scalar
Require: ksk[n*lk][m+1]: keyswitching key, each element a scalar
Ensure:  o[m+1]: LWE ciphertext, each element a scalar (initialized to zero)
1: Local variable: eic[n*lk]   (initialized to zero)
2: o[m] = c[n]                                       // copy body term
3: eic = Decompose(c[0..n-1], lk)                    // Decomposition
4: for i = 0 to m do
5:     for j = 0 to n*lk - 1 do
6:         o[i] = o[i] - eic[j] * ksk[j][i]           // Vector-matrix multiplication

The GPU bottleneck: blind-rotation fragmentation

The authors profile TFHE on an Nvidia Titan RTX (NuFHE, FFT path, optimal security parameter, Nsight metrics). A GPU naturally exploits device-level batching: each of the 72 SM cores runs a different ciphertext's PBS, but all cores share the same portion of the bootstrapping key in the same BR iteration, because the $\mathrm{bsk}$ (a matrix of polynomials) can be reused across nodes that are on the same iteration. This recasts a multi-ciphertext iteration as a polynomial matrix-matrix multiplication and gives spatial key reuse. As the left panel of Figure 2 shows, execution time stays flat for the first 72 ciphertexts. Beyond 72, cores must serialize, causing blind-rotation fragmentation: the BR kernel is replayed, stepping up the normalized execution time each time the ciphertext count crosses a multiple of the core count (the markers at 73, 145, 217 in the figure). The total GPU TFHE time is

$$\text{Total time} = \big(\#\,\text{BR fragmentations} + 1\big)\times \text{BR time per core}$$

as in Equation 2, where the number of fragmentations is

$$\#\,\text{BR fragmentations} = \left\lceil \frac{\#\,\text{ciphertexts}}{\text{batch size}} \right\rceil - 1$$

as in Equation 3. By Equation 2 and Equation 3, throughput improves either by reducing the BR time per core (more parallelism within an iteration, which targets latency) or by reducing the fragmentation count (batch more ciphertexts per BR fragment, which targets throughput). On NuFHE the batch size is fixed to the number of cores, and adding physical cores is costly and not scalable.

One could instead try core-level batching (give each core several ciphertexts per iteration, reusing the key temporally), but on a GPU this does not help: as the right panel of Figure 2 shows, the per-iteration time grows in proportion to the number of LWEs per core, so total time is unchanged. This is the central motivation of the paper. The authors also note the latency-versus-throughput tension: pushing latency down (fewer cycles per BR iteration) shrinks the time gap to fetch the next $\mathrm{bsk}$ and makes the workload memory-bound, whereas pushing throughput up (more temporal key reuse) widens that gap and makes it compute-bound. A specialized, fully-pipelined core can process a stream of ciphertexts per BR iteration at near-constant amortized cost - something a GPU cannot do. Keyswitching (about $30\%$ of total time) is likewise run as separate GPU kernels, adding latency that dedicated hardware can hide.

Core idea: two-level batching and four levels of PBS parallelism

Strix combines device-level batching (many cores, each a different ciphertext, sharing the key for spatial reuse) with core-level batching (each core streams multiple ciphertexts through one BR iteration for temporal reuse). Together they multiply the effective single-BR batch size to

$$\text{batch size} = \text{device-level batch}\times \text{core-level batch}$$

as in Equation 4, which is what kills fragmentation by driving the fragmentation count of Equation 3 toward zero for realistic ciphertext counts.

To implement core-level batching efficiently, the authors recast a multi-ciphertext BR iteration as a polynomial matrix-matrix multiplication (a test-vector matrix times the bootstrapping-key matrix) and identify four levels of parallelism, illustrated in Figure 3: TvLP (test-vector rows processed concurrently, equal to the number of cores), PLP (polynomials per row, maximum $(k+1)\,l_b$), CLP (coefficients processed in parallel per polynomial, that is, the FFT lanes), and CoLP (output columns, maximum $k+1$). Given the typical parameter ranges, the guaranteed minimum availability is $\text{PLP} = 4$ and $\text{CoLP} = 2$, while $\text{CLP}$ is the FFT width on a degree-$(N-1)$ polynomial with $N \in [1024, 16384]$, and $\text{TvLP}$ depends on the application (for a $28\times 28$ MNIST image, up to $784$ PBS can run in parallel, so $\text{TvLP}$ can reach $784$). Each level has a different hardware cost: TvLP equals the number of cores, CLP equals the number of lanes per computing unit, and PLP and CoLP correspond to replicating computing units.

Prioritizing throughput (the right goal for TFHE, since most applications need many PBS), the authors order the levels TvLP, CLP, PLP, CoLP. TvLP and CLP offer the largest opportunities; between them, increasing TvLP adds cores and processes more LWEs per epoch, whereas increasing CLP shortens the gap between BR iterations and therefore demands more external bandwidth to refetch the $\mathrm{bsk}$. Balancing area, HBM bandwidth, and interconnect, they pick $\text{TvLP}=8$, $\text{CLP}=4$, $\text{PLP}=2$, $\text{CoLP}=2$ for PBS. Keyswitching, being scalar rather than polynomial, uses $\text{TvLP}=8$, $\text{CLP}=8$, $\text{CoLP}=8$ with no PLP ($\text{PLP}=1$); TvLP matches the number of cores (one keyswitch cluster per core), CLP is set to match the PBS functional-unit throughput, and CoLP is set as large as possible to use all HBM channels.

Strix architecture: streaming cores, two clusters, two-level memory

Strix integrates up to 8 Homomorphic Streaming Cores (HSCs), a 21 MB global scratchpad, a NoC for key distribution, and an HBM2e controller, as shown in the top-level and HSC views of Figure 4. Each HSC has a local scratchpad ($0.625$ MB) and two compute clusters: a PBS cluster (six fully-pipelined stages: Rotator, then Decomposer, then FFT, then VMA, then IFFT, then Accumulator) and a separate Keyswitch cluster (a three-stage pipeline: Decomposer, then VMA, then Accumulator). A full traversal of the PBS pipeline equals one blind-rotation iteration; ciphertexts stream through with no stalls, which is what advances core-level-batching throughput.

Five specialized functional units cover both PBS and keyswitching: the Rotator (negacyclic rotate and subtract), the Decomposer (gadget decomposition into $l_b$ polynomials, used in both PBS and KS), the I/FFT unit (frequency-domain transform for polynomial multiplication via the convolution theorem of Equation 1, used only in BR), the VMA (vector multiply-accumulate via multipliers feeding an adder tree; complex arithmetic for PBS, integer for keyswitch), and the Accumulator (final time-domain sum). A workload-balancing trick splits the external-product accumulation between the frequency and time domains so the FFT-to-IFFT workload ratio becomes $1:1$ (otherwise FFT would process $l_b$ times more polynomials than IFFT). The pipeline is deliberately partitioned into six stages to balance the workload across these units.

Memory is two-level. The global scratchpad has a shared section that holds the $\mathrm{bsk}$ and $\mathrm{ksk}$, multicast to all cores over a fixed one-to-all unidirectional network, and a private section that holds per-core LWE ciphertexts and initial test vectors over a point-to-point network. The per-core local scratchpad holds intermediate test vectors (PBS cluster section) and the final-iteration output LWE that initiates keyswitching (keyswitch cluster section), connected to functional units by fixed point-to-point links. Strix schedules work in epochs of up to $\text{device-batch}\times\text{core-batch}$ LWEs, where the device-batch is the number of HSCs and the core-batch is how many test vectors fit in the local scratchpad. Each PBS cluster runs $n$ BR iterations, streaming the whole core-batch through the pipeline each iteration. When BR finishes, the output is forwarded to the keyswitch cluster while the next epoch's blind rotation begins, so keyswitching (faster, plain matrix-matrix multiplication tiled into vector-matrix steps) hides behind it. The $\mathrm{bsk}$ and $\mathrm{ksk}$ are double-buffered so HBM fetch is decoupled from compute, with the next key portions prefetched while the current iteration computes; finished outputs are collected in the global scratchpad and written back to the host.

Microarchitecture: pipelined FFT with folding, and a multiplier-free decomposer

I/FFT unit. Unlike CKKS NTT units that use a 2D NTT with costly matrix transposition between steps, Strix uses a pipelined FFT with $\log_2(N)$ butterfly (BFU) stages and shuffle units (SHU) that perform all data shuffling on-chip with no transpose, matching the streaming dataflow, as shown in Figure 5. It uses $\text{CLP}=4$ input lanes, hence 2 BFUs per stage; each lane carries fixed-point real and imaginary parts. Each SHU has two $L$-delay elements and two multiplexers whose selector toggles every $L$ cycles, with $L$ halving down the pipeline ($L=N/8$ at the first shuffle stage down to $L=1$). Large delays ($L \ge 32$) use SRAM-based shift registers, small ones ($L < 32$) use logic shift registers, with a Twiddle-ROM per butterfly stage. The initial latency to transform a degree-$(N-1)$ polynomial is $\tfrac{N}{\text{CLP}}$ cycles, after which one polynomial is produced every $\tfrac{N}{\text{CLP}}$ cycles.

Folding optimization. The usual trick packs two real polynomials into one complex FFT using FFT symmetry, but it needs an expensive separation buffer afterward that reorders the two signals and increases latency and area. Strix instead applies a folding scheme that splits a degree-$(N-1)$ polynomial into two degree-$\tfrac{N-1}{2}$ halves, placing one in the imaginary slot of the other and halving the signal length, with no separation module. This lets a $\text{CLP}=4$ unit virtually act as $\text{CLP}=8$ while using a smaller FFT (an 8,192-point transform instead of 16,384-point for $N=16{,}384$). Per Table VI, the folding scheme gives $1.99\times$ throughput, $1.68\times$ lower latency, $1.73\times$ smaller FFT area, and $1.48\times$ smaller total core (it roughly halves the FFT unit). Other functional units must run $\text{CLP}=8$ to benefit, but widening their lanes is far cheaper than widening the FFT.

Decomposer unit. TFHE uses gadget (digit) decomposition, not RNS. For each coefficient $a$ the decomposition into level $l_b$ digits in base $B$ satisfies the error bound

$$\left\| a - \sum_{i=1}^{l_b} \left\lfloor \frac{a}{Q/B^{\,i}} \right\rceil B^{\,i} \right\|_{\infty} \le \frac{Q}{B^{\,l_b}}$$

as in Equation 5, where $Q$ is the original ciphertext modulus, $B$ is the decomposition base, and $l_b$ is the decomposition level. A naive implementation needs a division and rounding per coefficient. As shown in Figure 6, Strix's fully-pipelined decomposer lane is built without any multipliers, using only masking, shifting, and adding, split into a rounding step (extract upper bits and carry with precomputed masks to round to the nearest representable value) and an extraction step (mask out each digit and add the carry from the previous digit). It runs $\tfrac{N}{\text{CLP}}\times l_b$ cycles per polynomial; implemented with $2\,\text{CLP}$ lanes it sustains a chip-wide throughput of $2\,\text{CLP}\times\text{CoLP}\times\text{TvLP}$ coefficients per cycle, matching the rotator and FFT.

Rotator, VMA, and Accumulator. The Rotator uses $2\,\text{CLP}=8$ lanes to match the folding scheme; it reads two coefficients per bank per cycle for both polynomials, cyclically rotates to align lanes, and adds/subtracts. With $\text{CoLP}=2$ it uses two instances for $2\,\text{CLP}\times\text{CoLP}\times\text{TvLP}$ coefficients per cycle at an initiation interval of one. The VMA multiplies test-vector and $\mathrm{bsk}$ polynomials (the $\mathrm{bsk}$ arriving over the multicast network) through an adder tree into a partial sum; with $\text{PLP}=2$, two instances yield $\text{CLP}\times\text{PLP}\times\text{TvLP}$ complex coefficients per cycle. The Accumulator takes the IFFT output, has $2\,\text{CLP}$ input lanes each with a buffer of $\tfrac{N}{2\,\text{CLP}}$ coefficients, and with $\text{CoLP}=2$ sustains $2\,\text{CLP}\times\text{CoLP}\times\text{TvLP}$ coefficients per cycle, writing final sums back to the local scratchpad for the next BR iteration.

Implementation, performance, and pipeline behavior

Strix is implemented in SystemVerilog and synthesized in the TSMC 28 nm PDK. Datapaths are 32-bit except the FFT unit (64-bit, with 16-bit twiddle factors for accuracy), clocked at $1.2$ GHz. The global scratchpad is a 21 MB dual-port (one read, one write) multi-bank 128-bit SRAM, double-buffered for $\mathrm{bsk}$, $\mathrm{ksk}$, and (R)LWE ciphertexts; the per-HSC local scratchpad is a $0.625$ MB true dual-port multi-bank 32-bit SRAM. Two broadcast buses carry the keys - 512-bit for $\mathrm{bsk}$ and 256-bit for $\mathrm{ksk}$. Of one $300$ GB/s HBM2e stack, 8 channels carry the $\mathrm{bsk}$, 4 the $\mathrm{ksk}$, and 4 the ciphertexts. Per Table III, the 8-HSC chip is $141.37\ \text{mm}^2$ and $77.14$ W; the I/FFT unit dominates a single core at $7.23\ \text{mm}^2$ and $5.49$ W (one core is $9.38\ \text{mm}^2$ / $6.21$ W; eight cores $75.03\ \text{mm}^2$ / $49.67$ W), and the 21 MB global scratchpad is $51.40\ \text{mm}^2$ / $26.24$ W. A custom object-oriented cycle-level simulator turns a workload into a computation graph whose nodes (PBS and/or KS) decompose into BR fragments, and evaluates four parameter sets (Table IV): set I ($n=500$, $k=1$, $N=1024$, $l_b=2$, 110-bit) is the common baseline used by all prior work; sets II ($n=630$, $N=1024$, $l_b=3$, 128-bit) and III ($n=592$, $N=2048$, $l_b=3$, 128-bit) match YKP and XHEC; set IV ($n=991$, $k=1$, $N=16{,}384$, $l_b=2$, 128-bit) is a new heavy/high-precision point.

PBS latency and throughput on set I (Table V): Concrete CPU $14$ ms / $70$ PBS/s; NuFHE GPU $37$ ms / $2{,}000$; YKP FPGA $1.88$ ms / $2{,}657$; XHEC FPGA $2{,}200$ PBS/s; Matcha ASIC $0.20$ ms / $10{,}000$; Strix $0.16$ ms / $74{,}696$ PBS/s. Strix is therefore $7.4\times$ Matcha's throughput at $1.25\times$ lower latency; the modest latency gain is expected at $\text{CLP}=4$, since Strix invests its parallelism in batched throughput rather than single-PBS latency. NuFHE supports only $N=1024$ and is tuned for $l_b=2$, so moving from set I to II collapses its performance (it drops the BR kernel and runs BR sequentially via the FFT kernel). On the heaviest set IV, Strix reaches $2{,}368$ PBS/s at $3.31$ ms, improving throughput by $2{,}368\times$ and latency by $292\times$ over Concrete, while most prior accelerators cannot run such large parameters at all.

Pipeline timing. Figure 8 traces all functional units across the first two blind-rotation iterations when each core processes three LWE ciphertexts (LWE-1, LWE-2, LWE-3) on set I. The three ciphertexts stream back-to-back through Rotator, Decomposer, FFT, VMA, IFFT, and Accumulator with no bubbles, while the HBM fetches the next $\mathrm{bsk}$ portion exactly once per BR iteration and reuses it across all three ciphertexts. The marked time gap is the window available to fetch the next keys, which is precisely what core-level batching widens so that key fetch hides behind compute. With this scheduling the decomposer, I/FFT, VMA, and accumulator units run close to $100\%$ utilization (the rotator about $50\%$); the local scratchpad is busy about $90\%$ of the time (rotator reads, accumulator writes) and HBM bandwidth is occupied about $60\%$ of the time, almost all of it $\mathrm{bsk}$ transfer. The design is compute-bound (bottlenecked by the I/FFT units) when LWEs per iteration are many or CLP is low, and memory-bound (bottlenecked by HBM) when LWEs per iteration are few or CLP is high, because the $\mathrm{bsk}$ must then be refetched too often.

Application benchmark and the TvLP vs CLP trade-off

On the Zama Deep-NN models (NN-20/50/100, taking a $28\times 28$ encrypted MNIST input with each pixel a separate ciphertext, a first $10\times 11$ convolution with ReLU producing a $[1,2,21,20]$ feature map, then dense layers of 92 neurons each with ReLU activations evaluated via PBS), Strix beats both CPU and GPU across all model depths and polynomial degrees ($N = 1024 / 2048 / 4096$), as shown in Figure 7. Speedups are $33$ to $38\times$ over CPU and $8$ to $17\times$ over GPU, and they grow as the workload gets heavier - exactly where TFHE accelerators matter most.

A TvLP versus CLP sweep at a fixed $300$ GB/s on set IV (Table VII) keeps the product $\text{TvLP}\times\text{CLP}$ constant and shows $\text{TvLP}=8$, $\text{CLP}=4$ is the sweet spot: $257$ GB/s, $3.8$ ms, $2{,}368$ PBS/s. At $\text{TvLP}=16$, $\text{CLP}=2$ the design is heavily compute-bound (only $200$ GB/s, $7.2$ ms). Pushing CLP higher makes it memory-bound (CLP $=8$ needs $371$ GB/s; CLP $=16$ needs $599$ GB/s and throughput halves to $1{,}240$ PBS/s); the extreme single-core case ($\text{TvLP}=1$, $\text{CLP}=32$) needs about $1\ \text{TB/s}$ ($1{,}053$ GB/s) and throughput collapses to $620$ PBS/s, because the gap between BR iterations shrinks and the $\mathrm{bsk}$ must be refetched too often. The takeaway is that adding cores (TvLP) scales throughput more cheaply than adding lanes (CLP), and $\text{TvLP}=8$, $\text{CLP}=4$ is the best fit for a single $300$ GB/s stack.

Where it sits in the TFHE-on-hardware line, and takeaways

Strix is positioned against the broader FHE-accelerator landscape. CKKS accelerators (F1, CraterLake, BTS, ARK, FAB, Poseidon) likely win on large packed ML workloads but are NTT-bound and memory-bound, needing 45 to 512 MB on-chip (FAB 45 MB, CraterLake 256 MB, ARK 512 MB) plus roughly $1\ \text{TB/s}$ from two HBM2e stacks. Strix instead exploits TFHE's lighter footprint: $26$ MB on-chip (TFHE's working set can be under 1 MB), one $300$ GB/s HBM2e stack, and is compute-bound rather than memory-bound. Among TFHE accelerators it advances past the GPU (NuFHE, whose FFT path beats its NTT path), the FPGA (YKP uses NTT for a $1.3\times$ gain over NuFHE; XHEC is a CPU-FPGA design), and the prior ASIC Matcha, which used bootstrapping-key unrolling to cut BR iterations at the cost of larger keys. Strix's distinct contribution is the streaming, two-level-batched approach plus FFT-specific algorithm-hardware co-design (the folding scheme) that TFHE permits because it can use FFT instead of NTT, and which supports a wider parameter range than any prior accelerator.

Limitations and caveats. Results come from a cycle-level simulator on synthesized RTL rather than fabricated silicon; the power and area are post-synthesis numbers (TSMC 28 nm). The architecture is specialized for TFHE PBS and keyswitching and does not accelerate CKKS-style packed SIMD workloads, where packing-based accelerators are expected to do better on large ML. The chosen design point fixes $\text{CLP}=4$, so the single-PBS latency improvement over Matcha is modest ($1.25\times$); the gains are predominantly in throughput. The design targets univariate PBS and the standard TFHE gate-bootstrapping flow, and HBM bandwidth becomes the binding constraint once CLP is pushed high or the per-iteration LWE batch is small.

Key takeaways:

  • The right lever for TFHE is throughput via batching, not single-PBS latency, and the blocker on commodity hardware is blind-rotation fragmentation, captured by Equation 2 and Equation 3.
  • Two-level (device plus core) batching, Equation 4, made practical by a fully-pipelined streaming core, is the architectural answer.
  • A transpose-free pipelined FFT with a folding scheme, plus a multiplier-free pipelined gadget decomposer obeying the bound of Equation 5, are the microarchitectural enablers; the external product of Equation 1 is the dominant inner kernel.
  • The result is the first accelerator purpose-built for high-throughput TFHE PBS: $1{,}067\times$/$37\times$ over CPU/GPU and $7.4\times$ over the prior ASIC, at a fraction of CKKS-accelerator area and bandwidth.

Algorithms

The two algorithms Strix accelerates, restated self-contained, plus the epoch scheduler the body describes but does not formally list. Algorithm 1 (Programmable Bootstrapping) refreshes noise and homomorphically applies an arbitrary univariate look-up table encoded in the test vector; its inner double loop is the external product of Equation 1 and is the dominant cost (about $98\%$ of PBS). Algorithm 2 (Keyswitching) converts the wide post-PBS LWE ciphertext back to the original key via scalar vector-matrix multiplication. Algorithm 3 summarizes the Strix epoch scheduler, showing how the two-level batch of Equation 4 is streamed through the pipeline so the $\mathrm{bsk}$ is fetched once per BR iteration and reused across the whole batch (the dataflow visualized in Figure 8).

Algorithm 1: Programmable Bootstrapping (PBS)
Inputs:
  n, k    : mask lengths of LWE / GLWE ciphertext
  lb      : decomposition level of bootstrapping
  N       : polynomial degree (power of two)
  c[n+1]  : input LWE ciphertext (scalars)
  tv[k+1] : test vector (degree-N polynomials encoding the LUT)
  bsk[n][(k+1)*lb][k+1] : bootstrapping key (degree-N polynomials)
Output:
  o[k*N+1] : refreshed LWE ciphertext (scalars)
Steps:
   1: allocate etv[(k+1)*lb], otv[k+1]; set all to zero
   2: for i = 0 to n:                          # modulus switching q -> 2N
   3:     c[i] = ModSwitch(c[i])
   4: tv = Rotate(left, tv, c[n])               # initial rotation by body term
   5: for i = 0 to n-1:                         # blind rotation (non-parallel, ~98% of PBS)
   6:     tv  = tv - Rotate(right, tv, c[i])    # rotate and subtract
   7:     etv = Decompose(tv, lb)               # gadget decomposition -> lb polys
   8:     for j = 0 to k:                       # output columns (CoLP)
   9:         for r = 0 to (k+1)*lb - 1:        # input polynomials (PLP)
  10:             otv[j] += IFFT( FFT(etv[r]) * FFT(bsk[i][r][j]) )   # external product (Eq. 1)
  11:     tv = otv
  12:     reset otv to zero
  13: o = SampleExtract(tv)                      # extract LWE from rotated GLWE
  14: return o
Algorithm 2: Keyswitching (KS)
Inputs:
  n, k   : input mask length (n = k*N for the post-PBS ciphertext)
  lk     : decomposition level of keyswitching
  c[n+1] : input LWE ciphertext (scalars)
  ksk[n*lk][m+1] : keyswitching key (scalars)
Output:
  o[m+1] : LWE ciphertext under the original key (scalars)
Steps:
   1: allocate eic[n*lk]; set o[] to zero
   2: o[m] = c[n]                                # copy body term
   3: eic = Decompose(c[0..n-1], lk)             # gadget decomposition
   4: for i = 0 to m:                            # output coordinates (CoLP)
   5:     for j = 0 to n*lk - 1:
   6:         o[i] = o[i] - eic[j] * ksk[j][i]    # vector-matrix multiply-accumulate
   7: return o
Algorithm 3: Strix Epoch Scheduler (two-level batching)
Inputs:
  H        : number of Homomorphic Streaming Cores (device-level batch = TvLP = 8)
  S        : core-level batch = number of test vectors fitting in local scratchpad
  LWEs[]   : queue of independent input ciphertexts for the application
  bsk, ksk : keys in shared global scratchpad (double-buffered)
Output:
  refreshed and key-switched LWEs[]
Steps:
   1: while LWEs queue not empty:
   2:     batch = take up to (H * S) ciphertexts          # batch size, Eq. 4
   3:     load batch test vectors into per-core local scratchpads (private net)
   4:     for it = 0 to n-1:                              # one BR iteration per pass
   5:         fetch bsk[it] portion once; multicast to all H cores (shared net)
   6:         prefetch bsk[it+1] into the alternate buffer  # double buffering
   7:         for each core h in parallel (device-level, spatial key reuse):
   8:             for each ciphertext s in core-batch (core-level, temporal reuse):
   9:                 stream s through PBS pipeline (initiation interval = 1):
  10:                     Rotator -> Decomposer -> FFT -> VMA -> IFFT -> Accumulator
  11:     for each finished ciphertext: run Algorithm 2 on the Keyswitch cluster
  12:                                   (hidden behind next epoch's blind rotation)
  13:     emit refreshed key-switched ciphertexts

Note: Strix does not introduce a new cryptographic algorithm; its contribution is the dataflow and microarchitecture (Algorithm 3) that batch and stream the standard TFHE PBS (Algorithm 1) and keyswitching (Algorithm 2).

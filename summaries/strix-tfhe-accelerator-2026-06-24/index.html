Strix: An End-to-End Streaming Architecture with Two-Level Ciphertext Batching for Fully Homomorphic Encryption with Programmable Bootstrapping

PDF (text-layer) — arXiv:2305.11423 · full view

At a glance

Authors: Adiwena Putra, Prasetiyo, Yi Chen, John Kim, Joo-Young Kim (School of Electrical Engineering, KAIST). Venue: MICRO 2023 (arXiv:2305.11423).

One sentence: Strix is the first hardware ASIC accelerator built specifically for high-throughput programmable bootstrapping (PBS) in the TFHE scheme, attacking the GPU "blind-rotation fragmentation" bottleneck with a two-level (device + core) ciphertext batching scheme realized in a fully-pipelined streaming datapath.

Headline numbers:

  • 1,067× higher PBS throughput than a state-of-the-art CPU (Concrete on Intel Xeon) and 37× higher than a GPU (NuFHE on Titan RTX).
  • 7.4× higher throughput and 1.25× lower latency than the prior best TFHE ASIC, MATCHA.
  • Up to 74,696 PBS/s at 0.16 ms latency (param set I); supports the widest range of TFHE parameter sets of any accelerator (incl. N up to 16,384).
  • End-to-end 33–38× (vs CPU) and 8–17× (vs GPU) speedup on Zama Deep-NN inference.
  • 141.4 mm², 77.1 W, 8 cores at 1.2 GHz in TSMC 28 nm, with only 26 MB on-chip memory and a single 300 GB/s HBM2e stack — far smaller than CKKS accelerators (256–512 MB, ~1 TB/s).

The problem: TFHE has no packing, so PBS is throughput-starved

Most FHE accelerators target CKKS, which packs many values per ciphertext and bootstraps a whole vector at once — high throughput but huge ciphertexts (MB), huge keys (GB) and 256–512 MB of on-chip memory. TFHE sits at the opposite end: it is a GSW-like, word/bit-level scheme whose key advantage is programmable bootstrapping (PBS), which evaluates an arbitrary univariate function (a homomorphic look-up table) while refreshing noise. This makes TFHE ideal for logical/relational operations and non-linear activations in ML, with KB-size ciphertexts and 10s–100s of MB keys.

The catch: TFHE does not support ciphertext packing, so PBS must be run one message at a time, giving inherently low throughput. The authors profile TFHE on CPU and find PBS is ~65% of a gate operation, keyswitching ~30%; within PBS, blind rotation (BR) is 96–98% of the time, and within each BR iteration the dominant cost is the external product (FFT × vector-multiply × IFFT). Blind rotation is a sequence of non-parallelizable iterations — each iteration multiplies a polynomial vector by the bootstrapping-key matrix.

The GPU bottleneck: blind-rotation fragmentation

The authors profile TFHE on an Nvidia Titan RTX (NuFHE). A GPU naturally exploits device-level batching: each of the 72 SM cores runs a different ciphertext's PBS, but all cores share the same bootstrapping key in the same BR iteration (spatial key reuse). Execution time stays flat for the first 72 ciphertexts. Beyond that, cores must serialize, causing blind-rotation fragmentation: the BR kernel is replayed, roughly doubling time each time the count exceeds the core count. Total time ≈ (#fragmentations + 1) × BR-time-per-core, where #fragmentations = ⌈#ciphertexts / batch_size⌉ − 1.

One could try core-level batching (give each core several ciphertexts to process per iteration, reusing the key temporally), but on a GPU this does not help: the per-iteration time grows in proportion, so total time is unchanged. This is the central motivation: a specialized, fully-pipelined core can process a stream of ciphertexts per BR iteration at near-constant amortized cost — something a GPU cannot do. Keyswitching (~30%) is likewise run as separate GPU kernels, adding latency that dedicated hardware can hide.

Core idea: two-level batching + four levels of PBS parallelism

Strix combines device-level batching (many cores, each a different ciphertext, sharing the key — spatial reuse) with core-level batching (each core streams multiple ciphertexts through one BR iteration — temporal reuse). Together they multiply the effective single-BR batch size to (core-level × device-level), which is what kills fragmentation.

To implement core-level batching efficiently, the authors recast a multi-ciphertext BR iteration as a polynomial matrix–matrix multiplication (test-vector matrix × bootstrapping-key matrix) and identify four levels of parallelism: TvLP (test-vector rows = number of cores), PLP (polynomials per row, max (k+1)·lb), CLP (coefficients per polynomial, the FFT lanes), and CoLP (output columns, max k+1). Prioritizing throughput, they pick TvLP=8, CLP=4, PLP=2, CoLP=2 for PBS (TvLP-CLP-PLP-CoLP priority), balancing area, HBM bandwidth and interconnect. Keyswitching uses TvLP=8, CLP=8, CoLP=8 (no PLP, since it is scalar not polynomial).

Strix architecture: streaming cores, two clusters, two-level memory

Strix integrates up to 8 Homomorphic Streaming Cores (HSCs), a 21 MB global scratchpad, a NoC for key distribution, and an HBM2e controller. Each HSC has a local scratchpad (0.625 MB) and two compute clusters: a PBS cluster (six-stage fully-pipelined: Rotator → Decomposer → FFT → VMA → IFFT → Accumulator) and a separate Keyswitch cluster (Decomposer → VMA → Accumulator). A full traversal of the PBS pipeline = one blind-rotation iteration; ciphertexts stream through with no stalls.

Five specialized functional units cover both PBS and keyswitching: Rotator (negacyclic rotate + subtract), Decomposer (gadget decomposition into lb polynomials), I/FFT unit (frequency-domain transform for polynomial multiplication), VMA (vector multiply-accumulate; complex multipliers for PBS, integer for keyswitch), and Accumulator (final time-domain sum). A workload-balancing trick splits the external-product accumulation between frequency and time domains so the FFT:IFFT workload ratio becomes 1:1 (otherwise FFT processes lb× more polynomials than IFFT). The dedicated keyswitch cluster lets keyswitching hide entirely behind the next epoch's blind rotation.

Memory is two-level: the global scratchpad's shared section holds the bsk/ksk (multicast to all cores, since the key is one-to-all), the private section holds per-core LWE ciphertexts and test vectors (point-to-point); the per-core local scratchpad holds intermediate test vectors. Strix schedules work in epochs of up to (device-batch × core-batch) LWEs; bsk/ksk are double-buffered so HBM fetch is decoupled from compute.

Microarchitecture: pipelined FFT with folding, and a multiplier-free decomposer

I/FFT unit. Unlike CKKS NTT units that use 2D NTT with costly matrix transposition between steps, Strix uses a pipelined-FFT with log₂(N) butterfly (BFU) stages and shuffle units (SHU) that do all data shuffling on-chip (no transpose), matching the streaming dataflow. It uses CLP=4 input lanes (2 BFUs/stage); large delay elements use SRAM shift registers, small ones use logic registers, with a Twiddle-ROM per stage. Initial latency to transform an (N−1)-degree polynomial is N/CLP cycles, then one polynomial every N/CLP cycles.

Folding optimization. Rather than the usual trick of packing two real polynomials into one complex FFT (which needs an expensive separation buffer afterward), Strix applies a folding scheme that splits an (N−1)-degree polynomial into two (N−1)/2 halves, placing one in the imaginary slot — halving signal length. This lets a CLP=4 unit virtually act as CLP=8 while using a smaller (8,192-point instead of 16,384-point) FFT. Result (Table VI): 1.99× throughput, 1.68× lower latency, 1.73× smaller FFT area, 1.48× smaller total core.

Decomposer unit. TFHE uses gadget (digit) decomposition, not RNS. A naive implementation needs divisions and rounding per coefficient. Strix's fully-pipelined decomposer is built without any multipliers — only masking, shifting and adding — split into a rounding step and an extraction step (precomputed masks). It runs N/CLP × lb cycles per polynomial; with 2×CLP lanes it sustains chip-wide throughput. The Rotator (8 lanes = 2×CLP for the folding scheme), VMA (PLP=2 instances, adder tree), and Accumulator (2×CLP lanes with N/(2·CLP)-coefficient buffers) are sized to keep the whole pipeline at one initiation interval.

Performance: throughput, latency, and parameter coverage

Strix is modeled in SystemVerilog/TSMC 28 nm (32-bit datapaths, 64-bit FFT, 16-bit twiddle, 1.2 GHz) and evaluated with a custom cycle-level simulator against Concrete (CPU), NuFHE (GPU), YKP & XHEC (FPGA), and MATCHA (ASIC). Four parameter sets are used; set I (n=500, k=1, N=1024, lb=2, 110-bit) is the common baseline, set IV pushes N=16,384.

PBS latency / throughput (Table V), param set I: Concrete CPU 14 ms / 70 PBS/s; NuFHE GPU 37 ms / 2,000; YKP FPGA 1.88 ms / 2,657; MATCHA ASIC 0.20 ms / 10,000; Strix 0.16 ms / 74,696 PBS/s. So Strix is 7.4× MATCHA's throughput at 1.25× lower latency. On the heaviest set IV, Strix hits 2,368 PBS/s at 3.31 ms — 2,368× throughput and 292× latency over Concrete, while most prior accelerators cannot run such large parameters at all.

The folding scheme contributes ~2× of the throughput edge (Table VI). Utilization analysis (Fig 8) shows the decomposer, I/FFT, VMA and accumulator units run near 100% (rotator ~50%); HBM is ~60% busy fetching the bsk, which is fetched once per BR iteration and reused across all batched LWEs.

Application benchmark: Zama Deep-NN

On the Zama Deep-NN models (NN-20/50/100 layers, 28×28 encrypted MNIST input, ReLU activations evaluated via PBS), Strix beats both CPU and GPU across all model depths and polynomial degrees (N=1024/2048/4096). Speedups are 33–38× over CPU and 8–17× over GPU, and grow as the workload gets heavier — exactly where TFHE accelerators matter most.

A TvLP-vs-CLP sweep (Table VII, fixed 300 GB/s, set IV) shows TvLP=8 / CLP=4 is the sweet spot: 257 GB/s bandwidth, 3.8 ms, 2,368 PBS/s. Pushing CLP higher (more FFT lanes) makes the design memory-bound (CLP=32, single core needs ~1 TB/s and throughput collapses to 620 PBS/s) because the gap between BR iterations shrinks and the bsk must be re-fetched too often. More cores (TvLP) scale throughput more cheaply than more lanes (CLP).

Where it sits in the TFHE-on-hardware line, and takeaways

Strix is positioned against the broader FHE-accelerator landscape. CKKS accelerators (F1, CraterLake, BTS, ARK, FAB, Poseidon) are NTT-bound, memory-bound, and need 45–512 MB on-chip plus ~1 TB/s (two HBM stacks). Strix instead exploits TFHE's lighter footprint: 26 MB on-chip, one 300 GB/s HBM2e stack, and is compute-bound rather than memory-bound. Among TFHE accelerators it advances past GPU (NuFHE), FPGA (YKP uses NTT; XHEC on CPU-FPGA) and the prior ASIC MATCHA (which used bootstrapping-key unrolling to cut BR iterations at the cost of bigger keys). Strix's distinct contribution is the streaming, two-level-batched approach plus FFT-specific algorithm-hardware co-design (folding) that TFHE permits because it can use FFT instead of NTT.

Key takeaways:

  • The right lever for TFHE is throughput via batching, not single-PBS latency — and the blocker on commodity hardware is blind-rotation fragmentation.
  • Two-level (device + core) batching, made practical by a fully-pipelined streaming core, is the architectural answer.
  • A transpose-free pipelined FFT with a folding scheme, plus a multiplier-free pipelined gadget decomposer, are the microarchitectural enablers.
  • Result: the first accelerator purpose-built for high-throughput TFHE PBS, 7.4× over the prior ASIC at a fraction of CKKS-accelerator area and bandwidth.

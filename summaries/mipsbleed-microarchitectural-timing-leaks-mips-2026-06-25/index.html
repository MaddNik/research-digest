MIPSBLEED: Uncovering Microarchitectural Timing Leaks in Pervasive Embedded Processors

Ahmed Najeeb & Billy Bob Brumley — arXiv:2606.16372 [cs.CR] (June 2026) · ARES 2026 · arXiv ↗

Introduction

Microarchitectural side-channel attacks have become one of the most consequential classes of hardware security vulnerability, exposing cryptographic secrets through physical phenomena such as cache access timing, execution latency, and resource contention. While the research community has extensively characterized these channels on x86 (Intel, AMD) and, more recently, ARM, the MIPS architecture has received comparatively little scrutiny — despite being embedded in an estimated 8.5 billion shipped chips as of 2024, powering routers, industrial controllers, IoT sensors, and network equipment worldwide.

This paper introduces MIPSBLEED, the first comprehensive framework for systematically uncovering microarchitectural timing leaks in SMT-enabled MIPS processors. The authors are Ahmed Najeeb and Billy Bob Brumley (ARES 2026). Their central finding is that Simultaneous Multithreading (SMT) — increasingly adopted to boost throughput in MIPS embedded platforms — creates powerful cross-thread timing channels through shared microarchitectural resources. By instrumenting three distinct channels (L1 data cache, L1 instruction cache, and execution engine), and by quantifying leakage rigorously with the NICV statistical metric, the authors demonstrate that an unprivileged attacker can recover cryptographic secrets from a co-located victim process with no kernel access, no hardware privilege escalation, and no reliance on speculative execution bugs.

End-to-end impact: The paper culminates in a single-trace key recovery attack against SECCURE 0.5 (an elliptic curve cryptography toolkit distributed in mainstream Linux distributions), recovering P-256 private scalar bits bit-by-bit by monitoring instruction cache contention during scalar multiplication.

Why MIPS is an Overlooked Target

Several properties distinguish MIPS from the better-studied x86/ARM threat landscape:

  • Ubiquitous deployment in long-lifecycle embedded systems. Network devices and industrial controllers may run for years or decades without security patches.
  • Weak isolation boundaries. Embedded Linux deployments often run multiple services on the same physical core with SMT enabled.
  • User-space cycle counter access. MIPS provides the rdhwr instruction for unprivileged access to the high-resolution cycle counter — a capability that ARM requires privilege for and x86 provides via rdtsc. This makes timing measurements trivially available to any user-space attacker.
  • No cache-flush instruction. Unlike x86 (which has clflush), MIPS lacks a user-accessible mechanism to flush specific cache lines. This prevents Flush+Reload attacks but does not eliminate cache-based timing channels; it redirects attackers toward Prime+Probe and contention-based approaches instead.

Background

The MIPS32 1004K Processor

The evaluation platform is the MIPS32 1004K, a multi-threaded embedded processor core used in routers and network SoCs. Its key microarchitectural parameters are:

With 256 sets and a 4-way LRU cache, the total cache capacity is $256 \times 4 \times 32 = 32{,}768$ bytes $= 32\ \text{KB}$. The distance between two cache lines mapping to the same set (i.e., the stride separating lines in different ways) is:

$$\text{stride} = \text{sets} \times \text{line size} = 256 \times 32\ \text{B} = 8192\ \text{B} = 8\ \text{KB}$$

This 8 KB stride is the key parameter that governs how attacker probes evict victim data from each cache set (see Methodology).

Simultaneous Multithreading (SMT)

SMT presents multiple logical processor cores to the operating system while sharing most physical resources. On the 1004K, the two logical cores share:

  • The entire L1 data cache (32 KB, 4-way)
  • The entire L1 instruction cache (32 KB, 4-way)
  • The execution engine: both the ALU and the Multiply/Divide Unit (MDU)

Each logical core maintains its own register file and program counter, but instruction fetch is interleaved between the two cores in alternating cycles. This sharing is the root cause of the cross-thread timing channel: when the victim thread on one logical core accesses a shared resource, the attacker thread on the other logical core observes a measurable latency increase.

Microarchitectural Side-Channel Taxonomy

The paper situates MIPSBLEED within the broader side-channel literature:

  • Flush+Reload (Yarom & Falkner, 2014): requires a shared memory mapping and a cache-flush primitive. Inapplicable to MIPS (no user-space flush).
  • Prime+Probe (Osvik et al., 2006): fills target cache sets with attacker data, then re-reads to detect evictions by the victim. Applicable to MIPS; MIPSBLEED's data cache channel is closely related.
  • Contention-based: attacker saturates a shared execution resource and measures slowdowns caused by victim usage. MIPSBLEED's execution engine channel uses this approach.

The LRU replacement policy in MIPS makes cache set eviction deterministic and predictable, which is essential for reliable Prime+Probe-style attacks. ARM's pseudo-random replacement substantially complicates such attacks.

The RDHWR Instruction and Cycle Counter

MIPS provides the RDHWR (Read Hardware Register) instruction, which allows user-space programs to read the High-Resolution Cycle Counter (CC) housed in Coprocessor 0 (CP0). By default on the 1004K, access to register 2 (the cycle counter) is enabled without privilege. This is a significant advantage for an attacker: timing measurements with cycle-level granularity can be taken from ordinary user-space code without any system call, kernel module, or hardware performance counter API. The instruction reads as:

rdhwr   $t0, $2     # read CC (cycle counter) into $t0

In contrast, x86 requires rdtsc (available user-space but with serialization caveats), while ARM requires either a privileged instruction or kernel-level performance counter configuration, substantially raising the attacker's required privilege level.

SECCURE Background

SECCURE (Suite for Elliptic Curve Cryptography Under Restriction of Effectiveness) version 0.5 (released August 2014) is a lightweight elliptic curve toolkit available in mainstream Linux package repositories. It implements elliptic curve Diffie-Hellman and digital signatures over a selection of NIST curves including P-256 (secp256r1). The core cryptographic operation is scalar multiplication $kP$, computed via the classical double-and-add algorithm operating on the secret scalar $k$ bit by bit.

Threat Model

The MIPSBLEED threat model is deliberately conservative — the attacker is granted minimal capabilities consistent with realistic embedded system deployments:

  1. Unprivileged user-space execution. The attacker runs arbitrary code on the victim system as an ordinary (non-root) user. No kernel exploits, no device drivers, no hardware debug access.
  2. SMT enabled. The system has SMT active (two logical cores sharing one physical core). This is the standard configuration on 1004K-based deployments for performance.
  3. Co-location on the same physical core. The attacker and victim threads must execute on the two logical cores of the same physical core, sharing L1 caches and execution units. The paper does not require cross-core or cross-socket attacks.
  4. Victim processes cryptographic material. The victim thread runs SECCURE (or equivalent), performing scalar multiplications with a long-lived private key. The attacker can trigger or observe repeated operations.
  5. Timing measurement available. The attacker can invoke rdhwr to read the cycle counter with nanosecond-level precision.
What the attacker does NOT need: Shared memory mappings with the victim (rules out Flush+Reload), root or kernel privileges, hardware performance counter access via kernel API, knowledge of the victim's virtual address space layout, or any speculative execution exploitation.

The threat model is consistent with the deployment context: embedded Linux on a router or industrial controller where multiple services (including attacker-controlled ones) run on the same CPU with SMT enabled and minimal process isolation.

Methodology

Framework Overview

MIPSBLEED decomposes the attack framework into three phases:

  1. Probe construction. Write a tight assembly loop that exercises one shared microarchitectural resource (cache set, execution unit) and measures the cycle-accurate time to do so using rdhwr.
  2. Leakage quantification. Use NICV (Normalized Inter-Class Variance) to determine whether timing measurements from the probe statistically correlate with victim behavior.
  3. Exploitation. Map the timing trace onto the victim's secret-dependent control flow to recover key bits.

NICV: Normalized Inter-Class Variance

NICV is a statistical metric originally introduced in the hardware side-channel community for evaluating leakage in power analysis. It is based on the one-way Analysis of Variance (ANOVA) F-statistic, which tests whether the means of multiple groups (classes) are significantly different relative to within-group variance.

Given a set of timing traces $T = \{t_1, t_2, \ldots, t_n\}$ labeled by a sensitive variable $Z$ (e.g., a secret bit) taking values in $\mathcal{Z}$, the NICV at a particular trace sample point is defined as:

$$\text{NICV} = \frac{\text{Var}_{z \in \mathcal{Z}}\!\left[\mathbb{E}[T \mid Z = z]\right]}{\text{Var}[T]}$$

Equivalently, NICV is the fraction of total timing variance that is explained by the class label $Z$:

$$\text{NICV} = \frac{\sum_{z \in \mathcal{Z}} \frac{n_z}{n}\left(\bar{t}_z - \bar{t}\right)^2}{\frac{1}{n}\sum_{i=1}^{n}(t_i - \bar{t})^2}$$

where $n_z$ is the number of traces in class $z$, $\bar{t}_z$ is the mean timing for class $z$, and $\bar{t}$ is the overall mean. NICV ranges in $[0, 1]$:

  • $\text{NICV} = 0$: the class label explains none of the variance; no leakage.
  • $\text{NICV} = 1$: all timing variance is explained by the class; perfect leakage.
  • High NICV (close to 1): strong, exploitable information leakage.
NICV is preferred over raw SNR (Signal-to-Noise Ratio) for this setting because it is normalized (always in $[0,1]$), works without specifying a leakage model, and captures nonlinear dependencies between timing and secret values.

The authors compute NICV over two classes per measurement point: traces where the victim is actively using the resource (causing contention) versus traces where the victim is idle. High NICV at a particular cache set index or time window identifies that cache set or moment as a leakage-carrying observation point.

Channel 1: L1 Data Cache Probe

The data cache is 32 KB, 4-way set-associative with 256 sets of 32-byte lines. The attacker probe operates as a cycle-accurate eviction-based timing probe, structured as follows:

The 8 KB stride between addresses is chosen because $\text{stride} = \text{sets} \times \text{line size} = 256 \times 32\ \text{B} = 8192\ \text{B}$, which is the exact distance between two addresses that map to the same cache set but different ways. Loading four addresses at this stride therefore touches all 4 ways of a single set, fully occupying it.

When the victim thread accesses a particular cache set $s_v$ concurrently, it displaces one of the attacker's probe lines via LRU eviction. The subsequent probe load for that set incurs a cache miss (or increased latency due to cache bank conflicts), creating a measurable spike in cycle count. The probe loops over all 256 sets, producing a 256-element timing vector per probe pass.

The dependency chain (multiplication instructions between loads) is critical: without it, out-of-order execution could reorder or overlap load latencies, washing out the per-set timing signal. Multiplication has a multi-cycle latency on the MDU, creating a reliable serialization barrier.

The I-cache probe works by analogy: if the victim's instruction stream causes fetches from cache set $s_v$, those fetches evict the attacker's probe instructions from that set. On the next probe pass, the attacker re-fetches from the same set and observes a miss. The timing signal mirrors that of the D-cache probe but reflects instruction fetch activity rather than load/store activity.

Channel 3: Execution Engine Probe

The execution engine probe targets the shared functional units rather than caches. Two variants are described:

ALU Variant

The attacker saturates the integer ALU by issuing 25 repetitions of dependent add/subtract instruction pairs:

// ALU contention probe (25 repetitions)
addu   $t0, $t1, $t2
subu   $t1, $t0, $t3
addu   $t0, $t1, $t2
subu   $t1, $t0, $t3
// ... (25x total)

Each pair creates a true data dependency ($\text{add output} \rightarrow \text{sub input}$), preventing the pipeline from parallelizing them. When the victim thread simultaneously issues ALU instructions, contention at the shared ALU increases the latency of the attacker's probe sequence. The cycle count of the 25-pair block is the timing measurement.

MDU Variant

The Multiply/Divide Unit (MDU) is a separate functional unit on the 1004K with higher latency and occupancy. The attacker floods it with repeated mul instructions:

// MDU contention probe
mul    $t0, $t1, $t2
mul    $t3, $t0, $t4
mul    $t1, $t3, $t2
// ... repeated

Because multiplication has multi-cycle latency and the MDU is not pipelined for dependent chains, a victim issuing mul or div instructions in the same cycle window creates highly visible latency spikes. The MDU variant yields the strongest leakage signal of the three channels.

Experimental Setup and Measurement Procedure

The full measurement procedure is:

  1. Spawn attacker thread on logical core 0; victim thread on logical core 1 (same physical 1004K core).
  2. Victim repeatedly executes the target cryptographic operation (SECCURE pointmul).
  3. Attacker continuously runs the probe loop, recording per-set or per-unit cycle counts.
  4. Collect a corpus of timing traces.
  5. Label each trace by the victim's secret-dependent operation (bit $= 0$: doubling only; bit $= 1$: doubling + addition).
  6. Compute NICV across all trace sample points.
  7. Identify high-NICV points as leakage hotspots.

Experimental Results

L1 Data Cache Leakage

Running the D-cache probe concurrently with a victim that repeatedly accesses a specific set of cache lines reveals a characteristic horizontal dark band at the victim's set index (set 50 in the paper's example figures). This band represents sustained, repeatable elevation in probe latency at that cache set index, and persists across many probe passes as long as the victim is active.

The NICV analysis over the D-cache probe trace confirms significant information leakage through the data cache channel. The signal is clean enough that the victim's accessed set is identifiable without averaging across many traces.

L1 Instruction Cache Leakage

The I-cache probe exhibits an analogous pattern. Victim instruction fetches displace attacker probe instructions from specific cache sets, producing the same horizontal-band signature. The NICV values for the I-cache probe are described as high, matching the D-cache result.

Critically for the cryptographic attack, the I-cache probe can distinguish between two different code paths executed by the victim: the point doubling only path (taken when the current scalar bit is 0) and the point doubling followed by point addition path (taken when the current scalar bit is 1). These two paths touch different instruction cache sets — the addition code occupies additional sets not used by the doubling-only path — creating a binary timing signal that directly encodes the key bit stream.

Execution Engine Leakage

The execution engine probe (both ALU and MDU variants) shows the highest NICV values of the three channels. The shared ALU creates measurable contention whenever the victim issues integer arithmetic; the MDU creates even stronger contention given its longer occupancy. The paper reports that the MDU probe yields "very high" NICV, indicating "strong leakage across the channel."

D-Cache Channel

High NICV. Victim cache set clearly identifiable. Distinguishes which cache lines victim accesses.

I-Cache Channel

High NICV. Distinguishes code paths (double-only vs. double+add). Directly encodes key bits.

Execution Engine

Very high NICV (strongest channel). MDU contention particularly pronounced. Leaks arithmetic operations directly.

End-to-End Attack: SECCURE Key Recovery

Target: SECCURE 0.5 and the pointmul Function

SECCURE 0.5 implements elliptic curve scalar multiplication for P-256 (secp256r1) using a classical left-to-right double-and-add algorithm. The vulnerable function is pointmul(), which iterates over the bits of the secret scalar $k$ from most-significant to least-significant:

// Pseudocode for SECCURE pointmul (double-and-add, vulnerable variant)
function pointmul(P, k):
    R = point_at_infinity
    for i = 255 downto 0:
        R = point_double(R)           // always executed (both branches)
        if bit_i(k) == 1:
            R = point_add(R, P)       // only executed when bit = 1
    return R

The conditional point_add call is the vulnerability. point_double and point_add are separate code paths that fetch different instruction cache lines. An attacker monitoring the instruction cache can observe:

  • When bit $k_i = 0$: only point_double instructions are fetched → fewer I-cache sets disturbed → lower probe latency.
  • When bit $k_i = 1$: both point_double and point_add instructions are fetched → more I-cache sets disturbed → higher probe latency.

This is a textbook Simple Power Analysis (SPA)-style vulnerability, here manifesting as a timing/contention channel rather than a power channel.

Key Generation and Demo Setup

To produce visually unambiguous evidence of the attack, the authors modified SECCURE's hash_to_exponent function to generate a deterministic scalar with a highly regular pattern: alternating 8-bit blocks of all ones and all zeros, i.e., a scalar whose bit pattern is $\underbrace{11111111}_{8}\underbrace{00000000}_{8}\underbrace{11111111}_{8}\cdots$

The authors explicitly note that this structured key is used strictly for visual demonstration purposes to make the leakage unambiguous in the published figures. The attack technique is equally applicable to uniformly random keys; the bit-by-bit distinguishability depends only on the I-cache timing separation between the two code paths, not on key regularity.

Attack Execution

  1. Setup. Attacker thread pinned to logical core 0; SECCURE key generation running on logical core 1 (same physical 1004K core).
  2. Trace collection. Attacker runs the I-cache probe loop continuously, recording per-set cycle counts. Each probe pass takes on the order of a few thousand cycles; scalar multiplication of P-256 takes many millions of cycles, providing ample time resolution.
  3. Single-trace analysis. The paper demonstrates that a single trace (one execution of pointmul) is sufficient to recover all key bits. No averaging across multiple executions is required.
  4. Bit recovery. The probe trace is segmented into 256 time windows corresponding to the 256 iterations of the scalar multiplication loop. In each window, the probe latency is classified as high (bit = 1: double+add) or low (bit = 0: double only).

Results

The partial trace (Figure 12 in the paper) clearly shows a sequence of eight consecutive high-latency windows followed by eight low-latency windows, matching the first 16 bits of the structured demo key ($\underbrace{11111111}_{8}\underbrace{00000000}_{8}$). The recovered bit string starts with 11111111, consistent with the demo key's design.

Significance

The SECCURE attack demonstrates several important properties:

  • Single trace sufficiency. Unlike DPA or correlation-based attacks that require thousands of traces, this attack recovers the full 256-bit key from a single scalar multiplication trace. This makes it effective even against ephemeral keys used only once.
  • No special access required. The entire attack runs from unprivileged user space using only rdhwr and standard memory operations.
  • Unmodified real-world software. SECCURE 0.5 is available in production Linux package repositories; no source modification is needed for the attack (only the demo key generation was altered).
  • P-256 private key recovery. The recovered scalar is the 256-bit ECDH private key, enabling full session decryption or signature forgery.

Mitigations and Defenses

Software-Level Countermeasures

Constant-Time Scalar Multiplication

The fundamental vulnerability in SECCURE is the secret-dependent branch in pointmul. The standard fix is to replace the conditional add with an unconditional always-add or Montgomery ladder formulation, which performs the same operations regardless of the scalar bit value:

// Montgomery ladder (constant-time, no secret-dependent branches)
function pointmul_montgomery(P, k):
    R0 = point_at_infinity
    R1 = P
    for i = 255 downto 0:
        if bit_i(k) == 0:
            R1 = point_add(R0, R1)    // these two branches perform
            R0 = point_double(R0)     // identical work; differ only
        else:                          // in which register is written
            R0 = point_add(R0, R1)
            R1 = point_double(R1)
    return R0

Or equivalently, using a conditional swap (cswap) primitive that is implemented without branching:

// cswap-based constant-time ladder
function pointmul_cswap(P, k):
    R0 = point_at_infinity;  R1 = P
    for i = 255 downto 0:
        b = bit_i(k)
        cswap(b, R0, R1)          // branchless conditional swap
        R1 = point_add(R0, R1)
        R0 = point_double(R0)
        cswap(b, R0, R1)
    return R0

A constant-time implementation causes both the I-cache and D-cache channels to observe identical patterns regardless of key bits, eliminating the leakage. This is the recommended fix for SECCURE and for any ECC implementation targeting adversarial environments.

Timing Noise Injection

Adding artificial random delays to cryptographic operations degrades the attacker's SNR. However, this countermeasure has practical limits: since the MIPSBLEED I-cache attack is single-trace (not an averaging attack), noise must be large enough to completely mask the signal in a single observation, which is difficult without unacceptable performance degradation.

Hardware-Level Countermeasures

Cache Partitioning

Assigning disjoint cache set ranges to different SMT threads via hardware or OS-level cache coloring would prevent cross-thread cache contention. However, MIPS embedded platforms typically lack the hardware support (e.g., Intel CAT) to enforce strict cache partitioning, and software-based cache coloring is complex in the embedded Linux context.

SMT Disabling

Disabling SMT entirely eliminates the shared-resource attack surface for all three channels. This is the most reliable countermeasure but imposes a performance penalty. For security-critical applications (e.g., a router handling VPN keys), disabling SMT may be the appropriate trade-off.

Privilege-Gating the Cycle Counter

Requiring kernel-level privilege to access the cycle counter (as ARM does by default) would eliminate the attacker's ability to make high-resolution timing measurements from user space. This is a targeted countermeasure that significantly raises the attacker's required privilege level without disabling SMT or degrading performance for non-attacker workloads. The paper highlights this asymmetry between MIPS (permissive default) and ARM (restrictive default) as a key policy distinction.

Lightweight Isolation in Embedded SMT

The authors call for future research into lightweight isolation mechanisms specific to SMT-enabled embedded systems. Unlike server-class x86 with many hardware security features (Intel MPX, CAT, SGX), embedded MIPS platforms have minimal built-in isolation. The challenge is designing mechanisms that provide meaningful isolation at the cost and silicon area constraints of embedded SoCs.

Limitations of the Countermeasures

  • Constant-time code is necessary but not sufficient if the compiler reintroduces branches during optimization; careful compiler flags and auditing are required.
  • Cache partitioning does not address the execution engine channel.
  • SMT disabling eliminates the attack vector entirely but reduces throughput, which may be unacceptable for performance-sensitive embedded applications.
  • Restricting rdhwr access prevents cycle-counter-based attacks but does not prevent attacks using coarser time sources (e.g., gettimeofday, performance counter APIs if available).

Limitations

  • Single hardware platform. All experiments are conducted on the MIPS32 1004K. It is unknown whether the exact NICV magnitudes and timing separations generalize to other MIPS cores (e.g., MIPS64 or other 1004K variants). The underlying vulnerability (shared L1 caches and execution units in SMT) is architectural, so the qualitative results likely generalize, but quantitative validation on other hardware is left as future work.
  • Co-location requirement. The attack requires the attacker and victim to be on the same physical core. In a multi-core system, the OS scheduler may move threads between cores; the attacker must ensure or manipulate co-location, which may require additional work.
  • Demo key regularity. The SECCURE key recovery is demonstrated on a structured (non-random) key for visual clarity. The authors assert the technique works on random keys, but a quantitative success rate for random keys under various noise conditions is not provided.
  • SECCURE only as end-to-end target. The cryptographic attack is demonstrated against one specific toolkit. Other ECC or asymmetric implementations (e.g., OpenSSL on MIPS) may have additional mitigations or different code structure.
  • No remote attack scenario. All measurements assume local process execution on the same system. Network-based remote timing attacks are not explored.
  • Mitigations not fully evaluated. The paper proposes lightweight isolation mechanisms as future work; no empirical evaluation of proposed hardware mitigations is presented.

Conclusion

MIPSBLEED establishes MIPS as a critical and underexplored target in the microarchitectural security landscape. By systematically characterizing three independent timing channels (L1 data cache, L1 instruction cache, and execution engine) on the MIPS32 1004K processor, the paper demonstrates that SMT-enabled MIPS platforms are vulnerable to cross-thread side-channel attacks that require no special privileges.

The NICV-based leakage quantification framework confirms high exploitable leakage on all three channels, with the execution engine (MDU) exhibiting the strongest signal. The end-to-end SECCURE key recovery attack demonstrates that this leakage is not merely theoretical: an attacker can recover a P-256 private key bit-by-bit from a single scalar multiplication trace using only user-space code and the rdhwr cycle counter instruction.

The broader implications are significant: the 8.5 billion deployed MIPS chips include long-lifecycle devices (routers, industrial controllers, IoT gateways) that are unlikely to receive microcode or hardware patches. For these devices, the most practical near-term mitigations are software-level: adopting constant-time cryptographic implementations and, where possible, disabling SMT for security-sensitive workloads. The paper calls for the research community to design lightweight, low-cost isolation primitives tailored to the embedded MIPS ecosystem.

Algorithms

The following algorithms are described or referenced in the paper:

  1. Algorithm 1: L1 Data Cache Probe
    An assembly loop that reads four cache lines per set at 8 KB intervals, covering all 4 ways of each of the 256 cache sets. Cycle counts measured via rdhwr before and after each per-set block. A multiplication dependency chain prevents out-of-order execution. Output: a 256-element timing vector per probe invocation.
  2. Algorithm 2: L1 Instruction Cache Probe
    A code traversal loop that jumps through four 8 KB-aligned code blocks per cache set, executing from addresses that map to all 4 ways of each set. Timing measured via rdhwr. Output: a 256-element timing vector per probe invocation.
  3. Algorithm 3: ALU Contention Probe
    Executes 25 iterations of dependent add/subtract instruction pairs to saturate the shared ALU. Timing measured via rdhwr. Output: a scalar cycle count per probe invocation.
  4. Algorithm 4: MDU Contention Probe
    Executes a sequence of dependent multiply instructions to saturate the shared Multiply/Divide Unit. Timing measured via rdhwr. Output: a scalar cycle count per probe invocation.
  5. Algorithm 5: NICV Computation (after Brier et al.)
    Given a set of $n$ timing traces labeled by secret variable $Z \in \mathcal{Z}$, compute $\text{NICV} = \text{Var}_{z}[\mathbb{E}[T \mid Z=z]] / \text{Var}[T]$ at each sample point. Identifies leakage hotspots in the trace.
  6. Algorithm 6: SECCURE Scalar Multiplication (Vulnerable, left-to-right double-and-add)
    The standard pointmul function in SECCURE 0.5. Iterates over 256 scalar bits; unconditionally calls point_double; conditionally calls point_add only when the current bit is 1. The conditional branch creates the I-cache timing leakage.
  7. Algorithm 7: Montgomery Ladder (Constant-Time Countermeasure)
    A secret-independent scalar multiplication that performs the same sequence of point operations regardless of key bits, using a conditional swap (cswap) primitive to avoid data-dependent branches.
  8. Algorithm 8: End-to-End SECCURE Key Recovery Attack
    (1) Pin attacker to logical core 0, victim to logical core 1. (2) Victim runs SECCURE P-256 key generation. (3) Attacker runs I-cache probe loop, collecting a continuous timing trace. (4) Segment trace into 256 windows aligned to scalar multiplication iterations. (5) Threshold each window's probe latency to recover the corresponding key bit. (6) Assemble the 256 recovered bits into the full scalar $k$.
